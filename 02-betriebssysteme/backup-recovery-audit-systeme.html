Kurzüberblick

Backup- und Recovery-Systeme stellen die Verfugbarkeit und Wiederherstellbarkeit von Daten im Katastrophenfall sicher. Audit-Subsysteme protokollieren sicherheitsrelevante Ereignisse auf Betriebssystemebene. Fur die digitale Forensik sind beide Bereiche von zentraler Bedeutung: Backups dienen als historische Datenquelle (was war wann auf dem System?), Audit-Logs als Beweismittel fur Benutzeraktionen (wer hat was wann gemacht?). Umgekehrt sind Backup- und Audit-Systeme selbst Angriffsziele — Ransomware verschlusselt oder löscht Backups, und Angreifer manipulieren oder deaktivieren Audit-Logs, um Spuren zu verwischen.

MHB-Mapping: BDF102 (Betriebssysteme) — Datensicherung, Wiederherstellung und Audit-Mechanismen moderner Betriebssysteme.

Backup-Typen: Voll, Differentiell, Inkrementell, Synthetisch

TypBeschreibungVorteileNachteile
VollbackupKomplette Kopie aller Daten. Jedes Backup ist unabhangig und vollstandig.Einfache Wiederherstellung: Nur das letzte Vollbackup wird benotigtLangsam, hoher Speicherverbrauch; Jedes Backup = 100% der Datenmenge
DifferentiellSichert alle Anderungen seit dem letzten Vollbackup. Kumulativ: Backup 3 enthalt Anderungen seit Backup 1 (Voll).Schnellere Wiederherstellung als Inkrementell (nur Voll + letztes Differentiell)Wachst mit jeder Generation; bei taglichem Backup wird Backup 7 so gross wie Backup 2
InkrementellSichert nur Anderungen seit dem letzten Backup (egal ob Voll oder Inkrem.).Kleinste Backup-Grosse, schnellste ErstellungWiederherstellung braucht Voll + ALLE Inkremente in Reihenfolge — ein fehlendes = Datenlucke
Synthetisches VollbackupErzeugt ein Vollbackup aus einem vorherigen Vollbackup + Inkrementen auf dem Backup-Server — ohne das Quellsystem zu belasten.Keine Belastung des Quellsystems; schnelle ErstellungRechenintensiv auf dem Backup-Server; abhangig von der Integritat der Inkremente
Forever-IncrementalEinmaliges Vollbackup, dann nur noch Inkremente. Synthetische Vollbackups ersetzen das initiale Vollbackup periodisch.Minimale Backup-Fenster; moderne Standardstrategie (Veeam, Commvault)Wiederherstellung komplex; Backup-Kette uber Jahre verwalten

Backup-Strategie: Grandfather-Father-Son (GFS)

Die GFS-Strategie organisiert Backups in drei Ebenen: Son (tagliche Inkremente, 7 Tage Rollierend), Father (wohentliche Vollbackups, 4–5 Wochen), Grandfather (monatliche Vollbackups, 12 Monate). Forensisch wertvoll: Die monatlichen Grandfather-Backups sind historische Zeitpunkte, zu denen der Systemzustand rekonstruiert werden kann — ideal zur Feststellung, wann eine Malware-Infektion erstmals auftrat.

RPO und RTO: Recovery-Kennzahlen

KennzahlDefinitionFragestellungForensischer Aspekt
RPO (Recovery Point Objective)Maximal tolerierbarer Datenverlust in Zeiteinheiten"Wie viele Daten durfen wir verlieren?"Je kleiner der RPO, desto mehr Backup-Punkte = mehr forensische Zeitpunkte zur Analyse
RTO (Recovery Time Objective)Maximal tolerierbare Ausfallzeit bis zur Wiederherstellung"Wie lange darf das System ausfallen?"Forensische Untersuchung verlangert RTO; Incident-Response-Teams mussen beide Bedurfnisse abwagen
# Beispiel: Webshop mit RPO=15min, RTO=2h
# RPO=15min: Alle 15 Minuten ein inkrementelles Backup
# → Forensisch: Datenandernungen in 15-Minuten-Auflosung nachvollziehbar
#
# RTO=2h: Nach einem Ransomware-Angriff muss der Shop in 2h wieder online sein
# → Forensisch: In 2h kann kein vollstandiges forensisches Imaging durchgefuhrt werden
# → Abwagung: Zuerst Backup-Recovery (Business), dann forensische Analyse

Snapshot-Technologien

Snapshots sind punktuelle Zustandsaufnahmen eines Speichersystems (Dateisystem, Volume, VM) ohne vollstandige Datenkopie. Sie sind keine Backups im klassischen Sinne — sie schutzen nicht vor physischem Storage-Ausfall — aber sie ermoglichen schnelle Wiederherstellung und sind forensisch hochgradig wertvoll.

Snapshot-Verfahren

VerfahrenPrinzipVorteilNachteilBeispiel
Copy-on-Write (CoW)Vor dem Uberschreiben eines Blocks wird der alte Block in einen Snapshot-Bereich kopiertPlatzsparend; Snapshot enthalt nur differenzielle DatenPerformance-Einbusse beim ersten Schreibzugriff (zusatzlicher Copy-Vorgang)LVM Snapshots (Linux), VSS (Windows)
Redirect-on-Write (RoW)Neue Schreibvorgange werden in neuen Bereich umgeleitet; Snapshot behalt OriginaldatenBessere Schreib-Performance (kein zusatzlicher Copy)Snapshot-Daten sind fragmentiert; Lesen verteilt sich uber Original + neue BlocksNetApp WAFL, ZFS (teilweise)
Split-MirrorVollstandige physische Kopie (Spiegel) wird zum Snapshot-Zeitpunkt abgetrenntVoll performant; Snapshot ist physisch unabhangigBenötigt mindestens doppelten Speicher; aufwandiges ErstellenEMC TimeFinder, klassische Storage-Array-Snapshots

Forensische Nutzung von Snapshots

# VSS (Volume Shadow Copy Service) auf Windows — forensische Goldgrube
vssadmin list shadows                       # Alle VSS-Snapshots auflisten
vssadmin list shadowstorage                 # Speicherverbrauch der Snapshots

# Mit sleuthkit auf VSS zugreifen:
mmls /dev/sda2                              # Partitionstabelle
mount -o ro,offset=... /dev/sda2 /mnt/vss   # VSS-Volume mounten

# VSS enthalt altere Versionen von:
# — Registry-Hives (SAM, SYSTEM, SOFTWARE)
# — Gelöschten Dateien (wenn im Snapshot-Zeitraum noch vorhanden)
# — Pre-Verschlusselungs-Zustanden (Ransomware: Dateien vor der Verschlusselung!)
Forensik-Tipp: VSS-Snapshots sind bei Ransomware-Untersuchungen oft die Rettung: Selbst wenn die Live-Dateien verschlusselt sind, enthalten die Snapshots die unverschlusselten Originale. Prufen Sie immer zuerst vssadmin list shadows, bevor Sie aufgeben. Tools wie ShadowExplorer oder Eric Zimmerman's VSCMount erleichtern den Zugriff.

Replikation: Synchron, Asynchron, Site Recovery

TypFunktionsweiseRPOEinsatz
SynchronJeder Schreibvorgang wird gleichzeitig auf Primar- und Sekundar-Storage geschrieben. Bestatigung erst, wenn beide bestatigen.0 (kein Datenverlust)Kritische Systeme (Finanzdatenbanken); erfordert geringe Latenz (< 10ms)

Site Recovery / Disaster Recovery Plan (DRP)

Ein DRP definiert, wie ein Unternehmen nach einem katastrophalen Ausfall (Brand, Hochwasser, Ransomware) den IT-Betrieb wiederherstellt. Fur Forensiker ist der DRP doppelt relevant: Erstens als zu untersuchendes Dokument (Wurde der DRP im Incident befolgt? Gab es Abweichungen?), zweitens als Verfahren, das die forensische Untersuchung beeinflusst (Muss ein Backup vor der forensischen Analyse zuruckgespielt werden, um den Geschaftsbetrieb wiederherzustellen?).

Audit-Subsysteme: auditd und Windows Event Log

auditd (Linux Audit Daemon)

auditd ist das native Audit-Framework des Linux-Kernels. Es protokolliert Systemaufrufe (syscalls), Dateizugriffe, Prozess-Starts und Authentifizierungsereignisse mit hohem Detailgrad.

# auditd-Architektur
# ┌─────────┐    ┌──────────┐    ┌─────────┐
# │ Kernel  │───#│ auditd   │───#│ audisp  │───# syslog / SIEM
# │ (Rules) │    │ (Daemon) │    │ (Plugin)│
# └─────────┘    └──────────┘    └─────────┘
# Netlink-Socket      Event Queue      Dispatcher

# Forensisch relevante Audit-Regeln setzen
# Alle ausgefuhren Kommandos protokollieren (execve-Syscall)
auditctl -a always,exit -F arch=b64 -S execve -k exec_commands

# Dateizugriffe auf /etc/shadow protokollieren
auditctl -w /etc/shadow -p rwxa -k shadow_access

# Regelabfrage und Log-Analyse
ausearch -k exec_commands --start today         # Alle execve-Events von heute
ausearch -f /etc/shadow -i                      # Alle Zugriffe auf /etc/shadow
aureport -x --start 07/01/2026 --end 07/04/2026 # Exec-Report fur einen Zeitraum

# Windows Event Forwarding (WEF)
# WEF leitet Events von vielen Windows-Clients zentral an einen Collector weiter.
# Forensisch extrem wertvoll: Events liegen zentral und sind vor Manipulation
# auf dem einzelnen Client geschutzt (wenn korrekt konfiguriert).
#
# Prufen der WEF-Konfiguration:
wecutil qc                    # Quick-Config: Collector einrichten
wecutil gs "ForensicEvents"   # Subscription-Config auslesen
Forensische Warnung: Windows Event Logs sind im laufenden Betrieb manipulierbar. Ein Administrator (oder Angreifer mit Admin-Rechten) kann Events löschen (wevtutil cl Security) oder den gesamten Event Log deaktivieren. Achten Sie bei der Analyse auf Lucken in den Event-Sequenzen: Fehlende Event-IDs zwischen zwei Eintragen konnen auf manuelle Löschung hindeuten. Korrelieren Sie mit VSS-Snapshots — altere Versionen der .evtx-Dateien konnen die gelöschten Events noch enthalten.

Forensische Relevanz

Backup-Timeline: Was war wann gesichert?

Backup-Kataloge (Bacula Catalog, Veeam Backup DB, Windows Server Backup History) sind eine unterschatzte forensische Quelle. Sie dokumentieren:

  • Welche Dateien zu welchem Zeitpunkt gesichert wurden
  • Welche Dateien nicht gesichert wurden (Ausschlusslisten — wurden bestimmte Verzeichnisse bewusst ausgeschlossen?)
  • Anderungen in der Backup-Grosse (ein plotzlicher Anstieg = Datenexfiltration? Ein Abfall = Ransomware?)

Ransomware-Erkennung durch veranderte Datei-Muster

Audit-Logs zeigen Massenanderungen an Dateien in kurzer Zeit — das klassische Ransomware-Muster. Ein Benutzerkonto, das innerhalb von 60 Sekunden 500 Dateien mit .encrypted- oder zufalliger Extension offnet, umbenennt und schreibt, ist ein hochsignifikanter Ransomware-Indikator.

# Beispiel: auditd-Regel fur Ransomware-Erkennung
# Protokolliere alle Datei-Umbenennungen (rename/renameat syscalls)
auditctl -a always,exit -F arch=b64 -S rename -S renameat -k rename_events

# Analyse: Gruppiere rename-Events nach Benutzer + Zeitfenster
ausearch -k rename_events --start 07/04/2026-14:00 --end 07/04/2026-14:05 \
  | aureport -f -i | awk '{print $6}' | sort | uniq -c | sort -rn
# Zeigt Top-Nutzer nach Anzahl der Rename-Operationen —
# ein Benutzer mit >100 Renames in 5 Min = Ransomware-Verdacht!

Disaster Recovery Plan und Forensik

Der DRP ist nicht nur ein Recovery-Dokument, sondern auch forensisches Beweismittel. Im Incident muss der Forensiker abwagen:

  • Recovery First: Systeme nach DRP wiederherstellen, um Geschaftsbetrieb zu gewahrleisten — aber dabei forensische Daten (RAM, Disk, Logs) vor dem Restore sichern
  • Forensics First: Erst das gesamte System forensisch kopieren (Imaging), dann Restore — aber dafur den Geschaftsbetrieb langer unterbrechen
  • Kompromiss: Triage-Sicherung (RAM + wichtige Logs + Registry) in 15 Minuten, dann Restore nach DRP, dann Post-Mortem-Analyse der Triage-Daten
