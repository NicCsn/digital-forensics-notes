Betriebssystem-Architekturen Grundlagen

Kurzueberblick

Jedes Betriebssystem hat eine Architektur, die bestimmt, wie Kernel und Userspace interagieren. Das Verstaendnis ist essenziell, um forensische Artefakte zu finden und Malware-Verstecke zu verstehen.

Kernel-Architekturen im Detail

TypBeispieleForensische Relevanz
MonolithischLinux, FreeBSDAlles laeuft im Kernel-Mode — ein kompromittiertes Kernel-Modul gefaehrdet das gesamte System. Rootkits koennen sys_call_table manipulieren. Module-Liste unter lsmod//proc/modules pruefen.
MikrokernelMINIX, QNX, L4, seL4Nur IPC, Scheduling, Speicherverwaltung im Kernel. Treiber und Dienste als Userspace-Prozesse — ein gecrasheter Treiber legt nicht das System lahm. Forensisch: Jeder Dienst hat eigene Prozessartefakte.
HybridWindows NT, macOS (XNU)Performance-Kritisches im Kernel (Scheduler, Speicher, I/O), andere Dienste im Userspace. Windows: Executive + Kernel + HAL. macOS: Mach-Mikrokernel + BSD-Schicht.
UnikernelMirageOS, IncludeOS, OSvAnwendung + OS als ein einziges Binary. Kein Userspace/Kernel-Split. Forensisch schwierig: kein traditionelles Dateisystem, keine Shell, nur Netzwerk-Debugging.
ExokernelMIT Exokernel, NemesisKernel stellt nur Hardware-Multiplexing bereit. Anwendungen verwalten Ressourcen selbst (eigene Page-Tables, Disk-Blocks). Forensisch: extreme Fragmentierung der Systemlogik — Spuren sind app-spezifisch.

Protection Rings und User-Mode vs. Kernel-Mode

Ring 3 — User-Mode
Alle Anwendungen, Browser, Malware (vor Escalation). Kein Hardware-Zugriff.
Ring 2 — (ungenutzt)
Ring 1 — (ungenutzt)
Ring 0 — Kernel-Mode
OS-Kernel, Geraetetreiber. Voller Hardware-Zugriff. Syscall = kontrollierter Uebergang.
Ring -1 — Hypervisor
Virtualisierungsschicht unter dem OS. Verwaltet Gast-VMs.
Ring -2 — Intel ME / AMD PSP
Management Engine / Platform Security Processor — eigener Mikrocontroller im Chipsatz. Unsichtbar fuer das Haupt-OS.
Ring -3 — Intel SMM / AMD SMU
System Management Mode — laeuft in isoliertem RAM (SMRAM). Code kann weder vom Kernel noch vom Hypervisor eingesehen werden.
Rootkit-Warnung: Kernel-Rootkits manipulieren die Syscall-Tabelle (SSDT unter Windows, sys_call_table unter Linux), um sich vor Userspace-Tools zu verstecken. SMM-Rootkits sind die gefaehrlichste Klasse — sie operieren in Ring -3 und sind selbst fuer den Kernel unsichtbar. Erkennung nur durch Analyse von SPI-Flash-Inhalten oder Zeitmessungen auf Instruktionsebene.
Forensiche Notiz zu Intel ME: Die Management Engine hat vollen Zugriff auf RAM und Netzwerk, laeuft aber auf einem eigenen Mikrokernel (ThreadX/minix-artig). Sie ist eine Blackbox und kann nicht vom Haupt-OS deaktiviert werden. In fortgeschrittenen APT-Faellen wurde ME-Code als Persistenzmechanismus missbraucht.

Systemaufrufe — der kontrollierte Uebergang von Ring 3 nach Ring 0

Wie funktioniert ein Syscall?

; x86 (32-bit): int 0x80 — langsam, durch IDT (Interrupt Descriptor Table)
mov eax, 1       ; Syscall-Nummer: sys_exit
int 0x80         ; Trigger Software-Interrupt → Kontextwechsel nach Ring 0

; x86 (32-bit): sysenter/sysexit — schnelle Variante (Intel)
; Ruft Kernel-Code direkt auf, ohne Interrupt-Overhead

; x86-64: syscall/sysret — 64-bit Fast-Syscall (AMD)
mov rax, 1       ; Syscall-Nummer
syscall          ; Springt direkt zum in LSTAR-MS-Register hinterlegten Handler

Ablauf eines Syscalls

  1. Anwendung ruft Library-Funktion auf (z.B. write(fd, buf, len) in libc)
  2. libc packt Syscall-Nummer in rax/eax und Argumente in rdi, rsi, rdx, r10, r8, r9
  3. syscall-Instruktion: CPU wechselt nach Ring 0, speichert rip in rcx, rflags in r11
  4. Kernel-Syscall-Handler (entry_SYSCALL_64 in Linux) dispatcht anhand der Nummer
  5. Kernel fuehrt Operation aus (z.B. Daten aus Kernel-Buffer in Userspace kopieren)
  6. sysret-Instruktion: CPU wechselt zurueck nach Ring 3

Context-Switch-Overhead

OperationGroessenordnung (x86-64)Forensische Spur
Syscall (Ring 3 → 0)~100-200 CPU-Zyklenstrace/perf protokollieren Aufrufe
User-Level-Context-Switch (Thread-Wechsel)~1-2 μssched_switch Trace-Events in perf.data
Kernel-Level-Context-Switch (Prozess-Wechsel)~3-5 μsProcess Accounting (/proc/PID/stat), Linux Auditd-Logs
Vollstaendiger VM-Exit (Gast → Hypervisor)~10-50 μsHypervisor-Logs, rekordierte VM-Exits per perf kvm stat
Forensischer Tipp: Der Syscall ptrace (Nr. 101) wird haeufig von Malware genutzt, um sich in andere Prozesse zu injizieren. memfd_create (Nr. 319) ermoeglicht ausfuehrbare, dateilose Payloads im RAM. Beide sollten in Forensik-Logs besonders beachtet werden.

Forensische Relevanz von VM-Snapshots

VM-Snapshots als Beweismittel: Ein Snapshot friert den gesamten Zustand einer VM ein — RAM, CPU-Register, Disk. Forensisch ist dies eine Momentaufnahme ersten Ranges. In Incident-Response-Faellen wird daher empfohlen, vor der Analyse einen Snapshot zu erstellen. Achtung: Snapshots enthalten auch fluechtige Daten (RAM) und koennen aktive Malware-C2-Verbindungen dokumentieren.
Snapshot-TypGespeichert inForensische Analyse
VMware Snapshot*-000001.vmdk (Delta-Disk) + *.vmsn (RAM)*.vmsn mit Volatility analysieren; Delta-Disk zeigt Aenderungen seit Snapshot
Hyper-V Checkpoint*.avhdx (Differencing Disk)AVHDX enthaelt nur geaenderte Bloecke — mit Powershell mergen oder direkt analysieren
VirtualBox SnapshotSnapshots/{UUID}.vdiSnapshot-VDI als separates Image mit TSK/Autopsy analysieren
Achtung: Snapshots sind kein Ersatz fuer ein forensisches Image nach Chain-of-Custody! Sie liegen auf dem Hypervisor-Dateisystem und unterliegen nicht den gleichen Integritaetsschutzmassnahmen wie ein Write-Blocker-Image. Fuer gerichtsfeste Beweise immer ein separates dd/EWF-Image erstellen.

Forensische Relevanz-Tabelle

Architektur-EntscheidungBeeinflusstes forensisches ArtefaktWarum relevant?
Monolithischer KernelKernel-Module (lsmod, /proc/modules)Rootkit-Erkennung: Jedes geladene Modul ist im Kernel-RAM und in /proc/modules gelistet. Manipulation der Liste = Rootkit-Indikator.
Windows HALHAL.dll Hash, Registry HKLM\SYSTEM\CurrentControlSet\EnumAenderungen an HAL.dll deuten auf Bootkit hin. HAL table Hooks sind ein APT-Indikator.

Type 2 HypervisorVMDK/VDI-Dateien auf Host-NTFS/ext4Host-Forensik: VMDK-Zugriffszeiten zeigen, wann die VM gestartet wurde. Host-Prefetch enthaelt vmware.exe.
Container (Namespace)/var/lib/docker/containers/<ID>/ — Logs, ConfigContainer-Logs persistieren nach docker rm! Enthalten stdout/stderr, Env-Vars (inkl. Secrets).
Syscall-MechanismusETW/Sysmon-Events, Linux AuditdSyscall-Tracing (Sysmon Event ID 1, Auditd -S execve) ist die zuverlaessigste Methode zur Malware-Erkennung — kann nicht von Userspace-Rootkits umgangen werden.
Goldene Regel der Architektur-Forensik: Je tiefer die Verteidigung, desto schwerer die Manipulation. Ring 3 kann Ring 0 beluegen. Ring 0 kann Ring -1 (Hypervisor) nicht beluegen. Ring -1 kann SMM (Ring -2) nicht beluegen. Die tiefste verfuegbare Analyseebene nutzen!

Praktische Anwendung

# Linux: Geladene Kernel-Module auf Rootkit-Hooks pruefen
lsmod | sort
cat /proc/modules

# Linux: Syscall-Tabelle auf Manipulation pruefen (braucht Kernel-Symbol-Map)
cat /proc/kallsyms | grep sys_call_table

# Windows: Boot-Konfiguration auf Hypervisor-Start pruefen
bcdedit /enum

# Windows: HAL-Version und Hash ermitteln
Get-FileHash C:\Windows\System32\hal.dll

# Linux: Namespace-Zugehoerigkeit eines Prozesses ermitteln
ls -la /proc/$(pgrep nginx)/ns/

# Volatility: Kernel-Module aus RAM-Dump extrahieren
volatility3 -f memory.dmp linux.lsmod
volatility3 -f memory.dmp linux.check_syscall
