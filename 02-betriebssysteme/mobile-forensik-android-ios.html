Kurzueberblick

Smartphones sind persoenliche Tagebuecher mit GPS, Kommunikationshistorie und App-Daten. Android und iOS haben fundamental unterschiedliche Sicherheitsarchitekturen — und entsprechend verschiedene forensische Extraktionsmethoden.

Android-Forensik

Sicherheitsarchitektur

FeatureBeschreibungForensische Implikation
Full-Disk Encryption (FDE)Android 5-9: gesamte Userdata-Partition verschluesselt mit dm-cryptOhne Passwort/PIN keine Entschluesselung. Brute-Force nur offline moeglich.
File-Based Encryption (FBE)Android 7+: Dateien separat verschluesselt. Direct Boot: eingeschraenkte Funktionen vor Erstentsperrung.Device-Encrypted (DE) Storage bei Boot verfuegbar. Credential-Encrypted (CE) erst nach PIN-Eingabe.
Keystore / KeymasterKryptographische Schluessel in Hardware-gestuetztem TEE (Trusted Execution Environment)Private Keys koennen nicht extrahiert werden. Keymaster protokolliert Schluesselverwendung.
Verified BootAndroid 7+: Prueft Bootchain-Integritaet (Bootloader → Kernel → System)Manipulationen am Boot-Image werden erkannt. Modifiziertes System = moeglicher Angriff.

Android-Applikationssandbox

# Jede Android-App hat:
# - eigene Linux-UID/GID (UID = App-ID + 10000)
# - eigenen Ordner unter /data/data/<paketname>/
# - SELinux-Label (u:r:untrusted_app:s0:cXX,cYY)
# - Kein Zugriff auf andere Apps (ausser ueber Content-Provider/Broadcast)

# Forensische Fundorte:
/data/data/<paket>/
├── databases/       ← SQLite-DBs (sms, contacts, chats)
├── shared_prefs/    ← XML-Key-Value-Paare (App-Einstellungen)
├── files/           ← App-eigene Dateien
├── cache/           ← App-Cache
└── lib/             ← App-eigene native Libraries

# Wichtige System-UDIDs:
/data/system/
├── batterystats.bin       ← Batterie-Protokoll
├── packages.xml           ← Installierte Apps + Berechtigungen
├── users/0/               ← User-Profil 0 (Owner)
│   ├── accounts.db         ← Synchronisierte Accounts
│   └── settings_secure.xml ← System-Einstellungen

Extraktionsmethoden

MethodeTypVoraussetzungErhaltene DatenForensische Eignung
ADB BackupLogicalADB-Debugging aktiv, Geraet entsperrtApps mit allowBackup=true; keine System-AppsBedingt — viele Apps (WhatsApp, Signal) erlauben kein Backup
ADB Pull (Root)FilesystemRoot-Zugriff oder engineering BuildKomplettes /data/data/ einsehbarGut — erfordert aber Root
Custom Recovery (TWRP)PhysicalEntsperrter Bootloader, Custom RecoveryVollstaendiges System-Image (dd/Nandroid)Hervorragend — vollstaendiges Disk-Image
JTAGPhysicalDebug-Pins identifiziert, geloetetRoh-Speicherzugriff (via Test Access Port)Hervorragend — aber invasiv und risikoreich
Chip-OffPhysicalFlash-Chip ausgeloetetRohdaten des Flash-SpeichersBestmoeglich — zerstoert aber das Geraet irreparabel
EDL-Mode (Qualcomm)PhysicalQualcomm-Chip, Emergency Download ModeRoh-Speicherabbild ueber Firehose-ProtokollHervorragend — verfuegbar auf vielen Xiaomi, OnePlus, etc.
Gefahrenstufen bei Android-Acquisition:
  • ADB Backup: Geringes Risiko, aber unvollstaendig. Wird von vielen Messengern explizit unterbunden.
  • Rooten: Modifiziert das System permanent. Xiaomi/Huawei loeschen bei Bootloader-Unlock alle Daten!
  • Chip-Off: Hoechstes Risiko — falsches Erhitzen zerstoert den Chip. Nur als letzte Option.

iOS-Forensik

Sicherheitsarchitektur

FeatureBeschreibungForensische Implikation
Secure EnclaveDedizierter ARM-Coprozessor (separiert vom A-Series-Chip) fuer SchluesselverwaltungSchluessel verlassen die Enclave nie. Kein Software-Extrakt moeglich. Brute-Force wird durch Enclave verzoegert.
Data ProtectionJede Datei hat eine Schutzklasse. Hierarchie: Geraeteschluessel → Klassenschluessel → DateischluesselDateien der Klasse NSFileProtectionComplete sind bei gesperrtem Geraet nicht lesbar.
KeychainVerschluesselter Key-Value-Store fuer Passwoerter, Tokens, ZertifikateKeychain-Eintraege haben eigene Schutzklassen. Einige sind immer verfuegbar (kSecAttrAccessibleAlways).
App SandboxJede App hat eigenen Container-Ordner; kein Zugriff auf andere AppsApp-Daten sind strikt isoliert. Nur ueber iOS-eigene Erweiterungen (Share Sheets) werden Daten geteilt.

iOS Data Protection Klassen

KlasseVerfuegbar beiForensischer Zugriff nach Erstentsperrung?
NSFileProtectionCompleteNur wenn Geraet entsperrtNicht bei gesperrtem/ausgeschaltetem Geraet
NSFileProtectionCompleteUnlessOpenEntsperrt + solange Datei offenJa (nach Erstentsperrung persistent lesbar)
NSFileProtectionCompleteUntilFirstUserAuthenticationNach erster Entsperrung nach BootJa (Default fuer die meisten Apps)
NSFileProtectionNoneImmer (nur per Geraeteschluessel)Ja (immer lesbar)
Forensischer Trick: Nach einem Neustart sind alle Complete-geschuetzten Dateien unlesbar. Vor der forensischen Sicherung muss das Geraet einmal entsperrt werden (After First Unlock, AFU). Im AFU-Zustand sind die meisten Dateien zugaenglich, was die Erfolgschancen bei logischer Extraktion massiv erhoeht.

SQLite-Forensik — das universelle mobile Datenbankformat

Architektur einer SQLite-DB

┌────────────────────────────────────────────────────────┐
│ SQLite Database File                                    │
│ ┌──────────────────────────────────────────────────────┐│
│ │ Header (100 Bytes)                                   ││
│ │ - Magic: "SQLite format 3\0"                         ││
│ │ - Page Size (meist 4096)                              ││
│ │ - Freelist Page Pointer: Zeiger auf erste freie Seite ││
│ │ - Number of Pages in DB                               ││
│ └──────────────────────────────────────────────────────┘│
│ ┌──────────┬──────────┬──────────┬──────────────────┐   │
│ │ Page 0   │ Page 1   │ ...      │ Page N           │   │
│ │ (Header) │ (Data)   │          │ (Data)           │   │
│ └──────────┴──────────┴──────────┴──────────────────┘   │
│                                                          │
│ Zugehoerige Dateien (im selben Verzeichnis):             │
│ ├── app.db-wal       ← Write-Ahead Log                  │
│ ├── app.db-shm       ← Shared-Memory (Index des WAL)    │
│ └── app.db-journal   ← Rollback-Journal (DELETE mode)   │
└──────────────────────────────────────────────────────────┘

Write-Ahead Log (WAL) — forensische Goldgrube

Die .db-wal-Datei enthaelt die juengsten Aenderungen, die noch nicht in die Haupt-DB zurueckgeschrieben wurden (vor dem naechsten Checkpoint):

  • Eingefuegte Zeilen, die durch spaetere DELETE oder UPDATE "verschwunden" sind, existieren noch im WAL-Frame.
  • SQLite/WAL-Analyse: Das Tool sqlite3 app.db ".recover" versucht, Daten aus einer defekten DB + WAL wiederherzustellen.
  • Freelist: SQLite ueberschreibt geloeschte Zeilen nicht sofort, sondern markiert sie in der Freelist zur Wiederverwendung. PRAGMA freelist_count; zeigt die Anzahl freier Seiten.
WAL-geloeschte Datensaetze finden: In der WAL-Datei sind auch Eintraege mit DELETE-Frames enthalten — der Frame selbst bleibt aber im WAL, bis er durch einen Checkpoint verdraengt wird. Tools wie undark oder kommerzielle Forensik-Software parsen die WAL-Frames direkt und extrahieren geloeschte Datensaetze.

SQLite-Forensik-Workflow

# 1. Datenbankstruktur analysieren
sqlite3 app.db ".schema"

# 2. Freelist und Integritaet pruefen
sqlite3 app.db "PRAGMA freelist_count;"
sqlite3 app.db "PRAGMA integrity_check;"
sqlite3 app.db "PRAGMA quick_check;"

# 3. WAL-Datei analysieren
sqlite3 app.db ".recover" > recovered.sql   # Daten aus DB + WAL rekonstruieren

# 4. Manual WAL-Parsing (Hex-Editor)
# WAL-Frame-Struktur:
#   - Frame Header (24 Bytes): Page Number, DB Size nach Commit
#   - Page Data (Page Size Bytes): Die eigentlichen Zellen-Daten

# 5. Nutze spezialisierte Forensik-Tools
# - undark (https://github.com/inflex/undark): WAL/Journal Carver
# - Sanderson Forensics SQLite Forensics Tool
# - epilog (SQLite forensic tool, C-based)

# 6. iOS: macOS-Zeitstempel konvertieren
# Apple CFAbsoluteTime = Sekunden seit 01.01.2001 00:00:00 UTC
sqlite3 sms.db "SELECT datetime(date/1000000000 + 978307200, 'unixepoch'), text FROM message;"

# 7. Android: Unix-Timestamp in Millisekunden
sqlite3 contacts2.db "SELECT datetime(last_time_contacted/1000, 'unixepoch'), display_name FROM contacts;"

Praktische Anwendung / Workflows

Android ADB-Workflow

# 1. Geraet verbinden und identifizieren
adb devices -l                              # Geraet auflisten + Details
adb shell getprop ro.build.fingerprint      # Build-Fingerprint
adb shell getprop ro.build.version.sdk      # API-Level

# 2. Logische Extraktion (ADB Backup)
adb backup -apk -shared -all -system -f android_backup.ab
# ACHTUNG: Viele Apps (WhatsApp, Signal) blocken Backup!
# Backupextraktion mit:
( printf "\x1f\x8b\x08\x00\x00\x00\x00\x00" ; tail -c +25 android_backup.ab ) | tar -xzf -

# 3. Root-Zugriff (falls vorhanden)
adb root
adb pull /data/data/com.android.providers.telephony/databases/mmssms.db ./sms.db
adb pull /data/data/com.android.providers.contacts/databases/contacts2.db ./contacts.db
adb pull /data/data/com.android.chrome/app_chrome/Default/History ./chrome_history.db

# 4. Live-Sicherung (fluechtige Daten)
adb shell dumpsys > dumpsys_full.txt        # Kompletter System-Zustand
adb shell dumpsys package > packages.txt    # Alle installierten Apps
adb logcat -d > logcat.txt                  # Aktuelle Logs
adb shell ps > processes.txt                # Laufende Prozesse

iOS-Backup-Analyse mit iLEAPP

# 1. Backup erstellen
#    - Auf macOS: iTunes/Finder → Backup jetzt erstellen
#    - Oder: idevicebackup2 backup /pfad/zum/backup (libimobiledevice)

# 2. Backup-Verzeichnis identifizieren
#    macOS: ~/Library/Application Support/MobileSync/Backup//

# 3. iLEAPP ausfuehren
git clone https://github.com/abrignoni/iLEAPP.git
cd iLEAPP
python3 ileapp.py -t filesystem -i /pfad/zum/backup/UDID/ -o /cases/ios/report

# 4. Analyse-Schwerpunkte im Report:
#    - sms.db → SMS/iMessage Extraktion
#    - KnowledgeC → App-Nutzung, Web-Verlauf
#    - locationd → GPS-Tracking
#    - Safari → Browser History + Cookies
#    - QuarantineEvents → Downloads aus dem Internet
#    - Keychain (nur verschluesseltes Backup) → Passwoerter

# 5. ALEAPP fuer Android (analog):
python3 aleapp.py -t filesystem -i /pfad/zu/adb-pull -o /cases/android/report

SQLite geloeschte Datensaetze extrahieren

# Freelist-Seiten als Hexdump ausgeben (jede freie Seite inspizieren)
sqlite3 app.db "PRAGMA freelist_count;"
sqlite3 app.db ".recover" | grep -i "geloescht"

# WAL-Datei separat sichern und analysieren
cp app.db-wal /evidence/wal_forensic/
cp app.db-shm /evidence/wal_forensic/

# undark zum Carven verwenden
undark app.db > app_carved.sql
# Ausgabe enthaelt: INSERT-Statements aus WAL + Freelist, inkl. geloeschter
# Datensaetze, die nicht mehr in der Haupt-DB sichtbar sind

# Manuelle Hex-Analyse des WAL (Frame-by-Frame)
# WAL-Frame startet bei Offset 0, 4096, 8192, ...
# Frame Header: [0-3] Page Number (Big End.), [4-7] DB Size, [8-23] Checksum
hexdump -C app.db-wal | less
