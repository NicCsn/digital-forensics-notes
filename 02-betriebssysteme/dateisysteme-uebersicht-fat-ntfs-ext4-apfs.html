Dateisysteme im Ueberblick Grundlagen

Kurzueberblick

Ein Dateisystem organisiert, wie Daten auf einem Speichermedium abgelegt, gefunden und verwaltet werden. Jedes hat eigene Metadaten, Zeitstempel und Strukturen — forensisch entscheidend. Diese Seite ist die wichtigste Grundlage fuer alle Datentraeger-Forensik.

FAT32 — das einfachste (und aelteste) Dateisystem

Aufbau

┌────────────┬───────────┬───────────┬──────────────────────────────────────┐
│ Bootsector │  FAT1     │  FAT2     │  Data Area                           │
│ (Sektor 0) │ (Kopie 1) │ (Kopie 2) │ Root-Dir + Dateien + Unterverz.      │
└────────────┴───────────┴───────────┴──────────────────────────────────────┘

FAT-Eintrags-Zustaende

Wert (32-Bit)Bedeutung
0x00000000Cluster ist frei
0x00000002 – 0x0FFFFFEFNaechster Cluster in der Kette (Zeiger)
0x0FFFFFF0 – 0x0FFFFFF6Reservierte Cluster
0x0FFFFFF7Bad Cluster (defekt)
0x0FFFFFF8 – 0x0FFFFFFFEnd-of-File (EOF) — letzter Cluster einer Datei

Geloeschte Dateien unter FAT32

# FAT32-Eintrag einer geloeschten Datei:
# 1. Erstes Zeichen des Dateinamens wird auf 0xE5 gesetzt
#    "REPORT .DOC" → "0xE5PORT .DOC"
# 2. FAT-Cluster-Eintraege werden auf 0x00 gesetzt (Kette zerstört!)
# 3. Daten-Cluster bleiben physisch erhalten — Carving möglich!
#
# Wiederherstellung:
# - Kurzer Dateiname: 0xE5 ersetzen
# - Fragmentierte Datei: Cluster-Kette nicht mehr rekonstruierbar
FAT32-Schwaeche: Bei Loeschung werden die FAT-Eintraege genullt — die Cluster-Kette ist unwiederbringlich verloren. Fragmentierte Dateien sind daher nach Loeschung nicht mehr per FAT rekonstruierbar. Nur File-Carving (Header/Footer-Suche) kann helfen.
LimitierungWert
Maximale Dateigroesse4 GB (minus 1 Byte)
Maximale Volumegroesse2 TB (32 KB Clustergroesse)
Maximale Pfadlaenge260 Zeichen (8.3-Format: 12 Zeichen)
JournalingKeines
BerechtigungenKeine
ZeitstempelNur Creation und Modified (2 Sekunden Aufloesung)

NTFS — das Windows-Arbeitspferd

Master File Table (MFT)

Jede Datei und jedes Verzeichnis hat einen MFT-Eintrag (1024 Bytes). Die ersten 16 Eintraege sind reservierte Metadaten:

EintragNameBeschreibungForensische Relevanz
0$MFTDie MFT selbstBasis fuer jede Dateisystem-Analyse
1$MFTMirrBackup der ersten 4 MFT-EintraegeVergleich mit $MFT bei Manipulation
2$LogFileTransaktions-Journal (4 KB-Seiten)Forensisches Gold: Zeichnet alle Transaktionen auf. Redo/Undo-Operationen rekonstruierbar.
3$VolumeVolume-Informationen (Label, Version)Erstellungsdatum des Volumes
4$AttrDefAttribut-DefinitionenHilft beim Parsen unbekannter Attribute
5.Root-VerzeichnisStartpunkt der Verzeichnisstruktur
6$BitmapBelegungs-Bitmap (1 Bit pro Cluster)Zeigt, welche Cluster frei/belegt sind — Loeschvorgaenge nachvollziehbar
7$BootBootsektor und BootcodeBootkit-Erkennung: mit sauberer Kopie vergleichen
8$BadClusBad-Cluster-ListeSpeicherorte von absichtlich markierten "defekten" Clustern (Datenversteck)
9$SecureSicherheits-Deskriptoren (ACLs)Datei-Berechtigungen, Ownership
10$UpCaseUnicode-Uppercase-TabelleKeine forensiche Relevanz
11$ExtendErweiterungen-VerzeichnisEnthaelt $UsnJrnl, $ObjId, $Quota, $Reparse

MFT-Eintrag-Struktur (1024 Bytes)

Offset  Inhalt
0x00    "FILE" (Magic Number)
0x14    Flag: 0x0000 = geloescht, 0x0001 = belegt, 0x0002 = Verzeichnis
0x18    File Reference to Parent Directory
0x20-0x38 $STANDARD_INFORMATION (Erstellt, Geaendert, MFT-geaendert, Zugriff)
0x38-0x50 $FILE_NAME (+ weitere four-CC-Attribute)
...
        $DATA (resident: Daten im MFT-Eintrag; non-resident: Cluster-Runs)

$STANDARD_INFORMATION vs $FILE_NAME — forensisches Gold

Timestomp-Erkennung: Malware manipuliert haeufig die Zeitstempel in $STANDARD_INFORMATION ("SI"), vergisst aber die vier Zeitstempel in $FILE_NAME ("FN"). Abweichung zwischen SI und FN = Manipulations-Indikator! Windows selbst aktualisiert FN nicht immer synchron — leichte Differenzen sind normal, extreme Abweichungen sind verdaechtig.
AttributEnthaltene ZeitstempelKann normaler Nutzer aendern?
$STANDARD_INFORMATIONCreated, Modified, MFT-Modified, AccessedJa (Timestomp-Tools, SetFileTime)
$FILE_NAMECreated, Modified, MFT-Modified, AccessedNicht direkt via API (nur NTFS-Treiber)

$LogFile — NTFS-Transaktionsjournal

Das $LogFile zeichnet alle NTFS-Metadaten-Aenderungen auf. Struktur:

  • 4 KB-Seiten, aufgeteilt in Log-Sequenzen (LSN)
  • Redo-Operationen: Wiederholbare Aenderungen (z.B. "schreibe Wert X an Offset Y")
  • Undo-Operationen: Rueckgaengigmachung bei Abbruch (z.B. "vorheriger Wert war Z")
  • Zirkulaerer Puffer — aeltere Eintraege werden ueberschrieben
Forensisch: $LogFile enthaelt eine zeitlich begrenzte Historie aller Dateioperationen. Bei geloeschten Dateien koennen die Redo-Operationen zeigen, wann sie existiert haben und was ihr letzter Pfad war. Tools: NTFS Log Tracker oder manuelle Analyse mit Hex-Editor.

$UsnJrnl — USN Change Journal

Datei: \$Extend\$UsnJrnl
Stream: \$Extend\$UsnJrnl:$J (Daten)
Stream: \$Extend\$UsnJrnl:$Max (Konfiguration)

# Jeder Eintrag enthaelt:
# - USN (Update Sequence Number — monoton steigend)
# - Timestamp
# - Change Reason (Flags: FILE_CREATE, FILE_DELETE, DATA_OVERWRITE, ...)
# - MFT Reference (File ID)
# - Parent MFT Reference
USN vs. $LogFile: $UsnJrnl persistiert laenger als $LogFile, enthaelt aber weniger Details. Beide zusammen geben ein umfassendes Bild der Dateisystem-Aktivitaet. Tools: usncarve.py, usnjrnl (Velociraptor).

ADS — Alternate Data Streams (NTFS-only)

# Jede NTFS-Datei kann mehrere Daten-Streams haben:
Dateiname.txt                  ← Haupt-Stream (benannt: ::$DATA)
Dateiname.txt:geheimer.exe     ← Alternativer Stream (unsichtbar im Explorer!)

# Stream anhaengen:
echo "versteckt" > datei.txt:geheim

# Stream auslesen:
more < datei.txt:geheim

# Forensich alle ADS finden:
dir /r                              # Windows: zeigt ADS-Tag
fls -r -o 2048 image.dd | grep :   # TSK: zeigt alle Streams
ADS als Malware-Versteck: Angreifer nutzen ADS, um Payloads vor dem Explorer zu verstecken. dir ohne /r zeigt nichts. Der IIS-Wurm "Sadmind/IIS" (2001) legte exploit.exe als index.asp::$DATA ab — der IIS fuehrte das Binary aus, der Explorer zeigte nur die harmlose ASP-Datei.

Struktur

┌─────────────────────────────────────────────────────────────────────────┐

│ sektor  │                                                               │

│         │ │ block    │ Descrip- │ Block  │ Bitmap   │ (je 256 Bytes) │  │

│         │ └──────────┴──────────┴────────┴──────────┴────────────────┘  │
└─────────────────────────────────────────────────────────────────────────┘

Inodes

FeldForensische Bedeutung
i_modeDateityp (regulaer, Verzeichnis, Symlink) + Berechtigungen
i_uid / i_gidBesitzer und Gruppe
i_sizeDateigroesse in Bytes
i_atime / i_mtime / i_ctimeAccess, Modified, Changed (inode-Aenderungen)
i_dtimeDeletion-Time — wann wurde die Datei geloescht?
i_links_count0 = geloescht (aber Inode noch nicht freigegeben?)

dtime — forensische Goldgrube: Das i_dtime-Feld bleibt auch nach Loeschung erhalten (wenn der Inode nicht ueberschrieben wurde). Es gibt die genaue Loeschzeit an — ein seltenes Detail, das z.B. NTFS nicht direkt speichert.

Journal-Modi

ModusWas wird protokolliert?Forensiche Relevanz
journalAlle Daten + Metadaten (langsam, sicher)Maximale Spuren: Dateiinhalte im Journal!
ordered (Standard)Nur Metadaten; Daten vor Metadaten auf DiskGute Spurenlage: Metadaten-Aenderungen protokolliert
writebackNur Metadaten; keine Reihenfolge-GarantieWenigste Spuren; moeglicherweise inkonsistent

Extent Tree

Ext4 verwendet Extents statt Block-Zeiger, um Fragmentierung zu reduzieren:

# Ein Extent beschreibt: "Logische Bloecke 0-99 = Physische Bloecke 5000-5099"
# Header (12 Bytes): magic=0xF30A, entries, depth, max_entries
# Extent Entry (12 Bytes): logischer Block, Laenge, physischer Start-Block

# Forensiche Relevanz:
# - Extent Tree ist kompakt und im Inode/i_block gespeichert
# - Kann mit fls und istat von TSK gelesen werden

Orphaned Inode List

Ext4 fuehrt eine Liste "verwaister" Inodes: Dateien, die geoeffnet, aber deren Verzeichnis-Eintrag geloescht wurde. Solange ein Prozess die Datei offen haelt, bleibt sie ueber die Orphan-Liste erreichbar und kann forensich wiederhergestellt werden.

APFS — Apple File System

Container-Modell

┌─────────────────────────────────────────────────────────────────────┐
│ APFS Container                                                      │
│ ┌───────────────────┬───────────────────┬─────────────────────────┐ │
│ │ Volume "Macintosh │ Volume "Data"     │ Volume "VM"             │ │
│ │ HD" (System, RO)  │ (Benutzerdaten)   │ (Swap, Sleep-Image)    │ │
│ └───────────────────┴───────────────────┴─────────────────────────┘ │
│                               ↑                                     │
│                   Space Sharing: Alle Volumes teilen sich            │
│                   denselben freien Speicherplatz                    │
└─────────────────────────────────────────────────────────────────────┘
FeatureBeschreibungForensische Relevanz
Copy-on-Write (CoW)Geaenderte Bloecke werden an neue Stelle geschrieben, alte bleiben erhaltenForensisches Gold: Alte Dateiversionen koennen auf Disk existieren, solange die Bloecke nicht ueberschrieben werden
SnapshotsEingefrorener Volume-Zustand zu einem ZeitpunktZeitreise! Geloschte Dateien sind im Snapshot noch vorhanden. macOS erstellt automatisch Snapshots vor Updates. tmutil listlocalsnapshots /
Space SharingAlle Volumes eines Containers teilen den SpeicherSchwierige Partitionierung — ein Snapshot kann mehr Platz belegen als zu erwarten
EncryptionPer-Volume-Schluessel; nativ und verpflichtend auf iOSOhne Schluessel = keine forensische Analyse moeglich
Fusion DriveSSD + HHD als logisches Volume; APFS managed den Tier-AbgleichDateien koennen auf zwei physischen Medien liegen — Imaging beider Laufwerke noetig!
APFS-Snapshot-Forensik in der Praxis: Auf einem Mac mit APFS gilt: sudo tmutil listlocalsnapshots / listet alle lokalen Snapshots. Mit sudo mount -t apfs -r -o snapname=com.apple.TimeMachine.2025-01-01-120000 /dev/disk1s1 /mnt/snapshot kann ein Snapshot als read-only eingebunden und mit Standard-Forensik-Tools analysiert werden. Geloeschte Dateien existieren oft noch in aelteren Snapshots!

APFS-Dateistruktur

APFS verwendet B-Trees (speziell B*-Trees) fuer alle Metadaten — kein statisches Inode-Konzept wie ext4. Das erhoeht die forensische Komplexitaet, da die Struktur selbst erst in RAM/Storage aufgebaut werden muss.

ZFS — das Enterprise-Dateisystem

Architektur

┌───────────────────────────────────────────────┐
│ ZFS Storage Pool (zpool)                       │
│ ┌─────────┬─────────┬─────────┬─────────────┐ │
│ │  vdev1  │  vdev2  │  vdev3  │  Cache/Log  │ │
│ │ (mirror)│ (RAID-Z)│ (RAID-Z2)│ (ZIL/L2ARC) │ │
│ └─────────┴─────────┴─────────┴─────────────┘ │
│                       ↑                        │
│  Alle vdevs bilden einen logischen Pool        │
│  ┌─────────────────────────────────────────┐   │
│  │ Datasets (Dateisysteme)                  │   │
│  │ tank/home   tank/data   tank/vms         │   │
│  └─────────────────────────────────────────┘   │
└───────────────────────────────────────────────┘

ZFS-Features mit forensischer Relevanz

FeatureBeschreibungForensische Implikation
Copy-on-WriteNie Daten am Original-Ort ueberschreibenAlte Versionen bleiben auf Disk bis TRIM/GC

ZFS-Forensik-Herausforderung: ZFS ist ein 128-Bit-Dateisystem — unvorstellbar grosse Adressraeume. Tools wie TSK haben nur begrenzten ZFS-Support. Die beste Forensik-Strategie: Snapshots nutzen und auf Dataset-Ebene mit nativen ZFS-Tools arbeiten, bevor ein dd-Image erstellt wird.

Praktische Anwendung: The Sleuth Kit (TSK) Befehle

# ---- Partition-Layout analysieren ----
mmls image.dd                    # Partitionstabelle (DOS, GPT, etc.) anzeigen
mmstat image.dd                  # Typ der Partitionstabelle bestimmen

# ---- Dateisystem-Informationen (alle FS-Typen) ----
fsstat -o 2048 image.dd          # Superblock/Metadaten des Dateisystems
fsstat -o 2048 image.ntfs        # NTFS: $MFT-Lage, Clustergroesse, Version
fsstat -o 0 image.apfs           # APFS: Container-Info

# ---- Datei-Listing (rekursiv, inkl. geloeschter/verwaister Dateien) ----
fls -r -o 2048 image.dd          # Rekursives Listing ab Root-Verzeichnis
fls -r -o 2048 -d image.dd       # Nur geloeschte Dateien
fls -r -o 2048 -m / image.dd     # Bodyfile-Format (fuer mactime-Timeline)
fls -r -o 2048 -z UTC image.dd   # Zeitzone setzen

# ---- NTFS-spezifisch ----
fls -r -o 2048 image.ntfs | grep :  # Alternate Data Streams finden
fls -o 2048 image.ntfs 0            # $MFT-Eintraege direkt listen
istat -o 2048 image.ntfs 35         # MFT-Eintrag 35 (File Reference) inspizieren

# ---- Inode-Analyse (ext4, UFS) ----
istat -o 2048 image.ext4 5           # Inode 5 inspizieren
ils -o 2048 image.ext4               # Alle Inodes auflisten (frei + belegt)
ifind -o 2048 image.ext4 -d 12345    # Welcher Inode ist an Block 12345?

# ---- Datei-Extraktion ----
icat -o 2048 image.dd 5            # Datei per Inode/MFT-Eintrag extrahieren
icat -o 2048 image.ntfs 35-128-1   # NTFS: $DATA-Attribut extrahieren

# ---- Timeline-Erstellung (alle Zeitstempel) ----
fls -r -o 2048 -m / image.dd > bodyfile.txt
mactime -b bodyfile.txt -z UTC > timeline.csv

# ---- NTFS $LogFile und $UsnJrnl (externe Tools) ----
# analyzeMFT -f \$MFT -o mft_output.csv
# MFTECmd.exe -f \$MFT --csv ./
# usncarve.py -f \$Extend\$UsnJrnl --csv output.csv

# ---- ext4: Geloschte Dateien wiederherstellen ----
# ext4magic image.dd -f /pfad/zur/datei -d /ausgabe/
# debugfs -R "ls -d /" image.ext4
