Kurzueberblick

Die forensische Analyse von macOS-Systemen unterscheidet sich grundlegend von Windows. macOS setzt auf das Apple File System (APFS) als Standard-Dateisystem (seit macOS 10.13 High Sierra), nutzt plist-Dateien (Property Lists) als zentrales Konfigurationsformat und verfuegt ueber ein Unified Logging-System, das alle System-, App- und Debug-Meldungen in einem strukturierten, durchsuchbaren Format speichert. Anders als Windows hat macOS keine Registry — stattdessen verteilen sich Konfigurationsdaten auf hunderte plist-Dateien und SQLite-Datenbanken.

macOS-Forensik ist eine Nische mit wachsender Bedeutung: In Unternehmensumgebungen steigt der macOS-Anteil kontinuierlich. Viele Incident-Responder haben jedoch nur Windows-Erfahrung. Die Artefakt-Landschaft auf macOS ist reichhaltig, aber grundsaetzlich anders organisiert. Wer sie beherrscht, hat einen massiven Vorteil.

APFS-Features forensisch relevant

APFS (Apple File System) wurde 2017 eingefuehrt und ersetzt HFS+. Es wurde speziell fuer Flash-Speicher (SSDs) entwickelt und bringt mehrere Features mit, die massive forensische Auswirkungen haben.

APFS-FeatureFunktionsweiseForensische Implikation
Copy-on-Write (CoW)Beim Aendern einer Datei wird NUR der geaenderte Block neu geschrieben, nicht die gesamte Datei. Das Original bleibt erhalten.Geloeschte oder ueberschriebene Dateifragmente koennen in freien APFS-Blocks ueberleben. Gleichzeitig: Eine Datei kann in hunderten nicht-sequenzieller Blocks verstreut sein — File-Carving wird extrem schwierig.
SnapshotsPoint-in-Time-Abbilder des gesamten Volumes ohne Daten-Duplizierung (nur Metadaten-Aenderungen)GOLDWERT: Snapshots koennen geloeschte Dateien im Originalzustand enthalten. macOS erstellt automatisch Snapshots vor System-Updates (tmutil listlocalsnapshots /). Diese sind eine unschaetzbare Quelle fuer geloeschte Daten.
Space SharingMehrere APFS-Volumes teilen sich denselben physischen Speicherplatz innerhalb eines ContainersEin physisches Image kann mehrere logische Volumes enthalten. Alle Volumes muessen als Einheit betrachtet werden — freier Speicher auf einem Volume kann Daten eines anderen enthalten.
EncryptionAPFS unterstuetzt native Volume-Verschluesselung (FileVault) ohne Performance-VerlustOhne Entschluesselungspasswort oder Recovery-Key ist ein APFS-Volume forensisch unbrauchbar. FileVault-Key kann im RAM (Memory-Dump) oder via Recovery-Key/Apple-ID wiederhergestellt werden.
Hard Links auf DirectoriesAPFS erlaubt Hard Links auf Verzeichnisse (keine Symlinks)Time Machine nutzt dies intensiv. Bei der Analyse von Backups muessen Hard-Link-Strukturen aufgeloest werden, um doppelte Dateien nicht als mehrere Kopien zu zaehlen.
APFS und geloeschte Dateien: Aufgrund von Copy-on-Write und TRIM (SSD) ist die Wiederherstellung geloeschter Dateien auf macOS deutlich schwieriger als auf NTFS mit HDD. Sobald ein Block per TRIM freigegeben wurde, ist er physisch unwiederbringlich. Snapshots sind oft die einzige Moeglichkeit, geloeschte Daten zu rekonstruieren.

FSEvents-Datenbank (/.fseventsd/)

FSEvents ist das macOS-Event-Benachrichtigungssystem, das Dateisystemaenderungen in Echtzeit verfolgt. Jeder APFS-Volume hat ein verstecktes Verzeichnis /.fseventsd/, das bis zu 64 KB grosse, GZIP-komprimierte Log-Dateien enthaelt. Diese Logs protokollieren JEDE Dateisystemaenderung — welche Datei wurde erstellt, geaendert, geloescht oder umbenannt, inklusive vollstaendigem Pfad und Event-ID.

# FSEvents-Logs auflisten
sudo ls -la /.fseventsd/

# FSEvents-Logs parsen mit Drittanbieter-Tools
fseventer /.fseventsd/0000000000001a2d
# Alternativ: FSEventsParser (macOS_Forensics_Artifacts)
python3 fsevents_parser.py -i /.fseventsd/ -o fsevents_output.csv
Event-TypBedeutungForensischer Wert
CreatedDatei/Verzeichnis wurde neu angelegtWann wurde Malware heruntergeladen? Wann wurde eine persistente Datei abgelegt?
RemovedDatei/Verzeichnis wurde geloeschtWelche Dateien hat der Angreifer nach der Exfiltration geloescht? Zeitpunkt der Spurenbeseitigung
ModifiedInhalt oder Metadaten wurden geaendertWann wurde eine Konfigurationsdatei manipuliert? Zeitpunkt der Schadcode-Aktivitaet
RenamedDatei/Verzeichnis wurde umbenanntWurde Malware getarnt (z.B. "malware.exe" zu "document.pdf")?
ExchangeAtomarer Austausch zweier DateienWurde eine Systemdatei durch eine manipulierte Version ersetzt?
FSEvents ist eine der wertvollsten Datenquellen auf macOS: Es protokolliert Aenderungen, die laengst aus anderen Logs verschwunden sind. Die Logs rotieren ueber Monate. Selbst nach Loeschung der eigentlichen Datei bleibt der FSEvents-Eintrag erhalten. FSEvents ist das macOS-Aequivalent zur USN-Journal / $LogFile auf NTFS.

Unified Logging — log show

Das macOS Unified Logging System (eingefuehrt in iOS 10 / macOS 10.12 Sierra) ersetzt die alten System-Logdateien (/var/log/system.log) durch eine zentrale, binaere Trace-Datenbank unter /var/db/diagnostics/. Es speichert Log-Eintraege aller Subsysteme in einer einheitlichen, strukturierten Form — durchsuchbar mit dem log-Befehl.

# Alle Logs der letzten 24 Stunden anzeigen
log show --last 24h

# Nur Logs eines bestimmten Subsystems (z.B. SSH-Logins)
log show --predicate 'subsystem == "com.apple.ssh"' --last 3d

# Prozess-spezifische Logs (z.B. Launchd-Aktivitaet)
log show --predicate 'process == "launchd"' --last 1h

# Nach bestimmten Event-Typen filtern
log show --predicate 'eventType == "logEvent" AND message CONTAINS "error"' --last 7d

# Logs als JSON exportieren (maschinenlesbar)
log show --style json --last 24h > unified_logs.json
Wichtige SubsystemeBeschreibungHunting/IR-Relevanz
com.apple.sshSSH-Dienst (Remote Login)Login-Versuche, erfolgreiche/failed Logins, verwendete Key-Typen
com.apple.trustdZertifikatsvalidierungWelche TLS-Zertifikate wurden geprueft? Welche abgelehnt? Man-in-the-Middle-Indikator?
com.apple.launchdDienst- und Job-ManagementWelche Dienste/Plist-Agents wurden gestartet? Neue Persistenz-Mechanismen?
com.apple.networkNetzwerkverbindungenVerbindungsaufbau, DNS-Aufloesungen, Proxy-Konfiguration
com.apple.tccTransparency, Consent, Control (Berechtigungen)Welche Apps haben auf Kamera, Mikrofon, TCC-Datenbank zugegriffen?
Unified Logs sind fluechtig: Anders als persistente Logdateien rotieren Unified Logs — aeltere Eintraege werden ueberschrieben. Auf einem aktiven System sind selten mehr als 7-14 Tage an Log-Daten verfuegbar. Bei Incident Response: SOFORT log collect ausfuehren, um die aktuellen Logs zu sichern.

Quarantine-Datenbank und Gatekeeper

Gatekeeper ist das zentrale Sicherheitssystem von macOS, das die Ausfuehrung nicht-signierter Software blockiert. Jede aus dem Internet heruntergeladene Datei erhaelt ein Quarantine-Attribut (com.apple.quarantine als Extended Attribute). Die Quarantine-Datenbank unter ~/Library/Preferences/com.apple.LaunchServices.QuarantineEventsV2 (SQLite) speichert jeden Download.

# Quarantine-Datenbank auslesen
sqlite3 ~/Library/Preferences/com.apple.LaunchServices.QuarantineEventsV2 \
  "SELECT datetime(LSQuarantineTimeStamp + 978307200, 'unixepoch', 'localtime') as DownloadDate,
          LSQuarantineAgentBundleIdentifier as App,
          LSQuarantineDataURLString as DownloadURL,
          LSQuarantineOriginURLString as OriginURL
   FROM LSQuarantineEvent
   ORDER BY LSQuarantineTimeStamp DESC;"
FeldBedeutungForensische Relevanz
LSQuarantineTimeStampZeitpunkt des Downloads (Mac-absolute-Zeit)WANN wurde die potenzielle Malware heruntergeladen?
LSQuarantineDataURLStringDirekter Download-URLVon welchem Server kam die Datei? IP/Domain als IOC
LSQuarantineOriginURLStringReferrer-URL (die Seite, die den Download-Link enthielt)Ueber welche Website wurde der Download ausgeloest? Phishing-Seite?
LSQuarantineAgentBundleIdentifierWelche App hat den Download ausgeloest?Browser (com.apple.Safari, org.mozilla.firefox) oder E-Mail-Client?
LSQuarantineAgentNameAnzeigename der downloadenden AppSafari, Firefox, Mail, Slack — welcher Kanal wurde fuer Malware-Delivery genutzt?
Gatekeeper umgehen: Angreifer umgehen Gatekeeper haeufig via signierte Payloads (Apple Developer ID), via App-Bundles ohne Quarantine-Flag, oder durch Ausfuehrung ueber legitime Prozesse (z.B. Python-Interpreter). Eine leere Quarantine-Datenbank bei gleichzeitiger Malware-Aktivitaet ist selbst ein Indikator (Loeschung der Quarantine-Eintraege durch Angreifer).

KnowledgeC — User Activity Database

KnowledgeC (/var/db/CoreDuet/Knowledge/knowledgeC.db) ist eine hochgradig forensisch wertvolle SQLite-Datenbank, die von Apples CoreDuet-Framework verwaltet wird. Sie zeichnet Nutzeraktivitaeten im Detail auf: Welche App wurde wann gestartet, wie lange genutzt, welche Webseite besucht, welches Dokument geoeffnet, wann wurde das Geraet entsperrt, wann in den Ruhezustand versetzt?

# KnowledgeC auslesen (Application Focus) — welche Apps waren wann im Vordergrund?
sqlite3 /var/db/CoreDuet/Knowledge/knowledgeC.db \
  "SELECT datetime(ZOBJECT.ZCREATIONDATE + 978307200, 'unixepoch', 'localtime') as Timestamp,
          ZOBJECT.ZVALUESTRING as AppName,
          ZOBJECT.ZSTARTDATE,
          ZOBJECT.ZENDDATE
   FROM ZOBJECT
   WHERE ZOBJECT.ZSTREAMNAME = '/app/usage'
   ORDER BY ZOBJECT.ZCREATIONDATE DESC
   LIMIT 100;"
KnowledgeC-StreamEnthaltene DatenForensischer Wert
/app/usageWelche App war wann im Vordergrund (Fokus)? Startzeit, Endzeit, Bundle-IDNutzungsprofil: Welche App hat der Angreifer wann genutzt? Terminal, Python, Sublime?
/display/isBacklitWann wurde das Display ein-/ausgeschaltet?War das Geraet zu bestimmten Zeiten aktiv? Nachtaktivitaet?
/device/isPluggedInWann wurde das Geraet ans Stromnetz angeschlossen?Korreliert mit laengeren Betriebszeiten, z.B. fuer Exfiltration
/safari/historySafari-Browser-History: URL, Zeitpunkt, Anzahl BesucheWelche Webseiten wurden besucht? Downloads getaetigt?
/bluetooth/isConnectedBluetooth-VerbindungsstatusWurde ein unbekanntes Bluetooth-Geraet verbunden?
KnowledgeC ist das macOS-Aequivalent zur Windows-Timeline: Es liefert eine chronologische Aufzeichnung der Nutzeraktivitaet, die weit ueber das hinausgeht, was Einzelartefakte zeigen. Zusammen mit FSEvents und Unified Logs entsteht ein extrem detailliertes Bild des Nutzer- und Angreiferverhaltens.

plist-Struktur und Parsing

Property Lists (.plist) sind das zentrale Datenformat fuer Konfigurations- und Metadaten auf macOS. Sie existieren in zwei Formaten: XML (menschenlesbar) und binaer (maschinenoptimiert). Praktisch alle macOS-Artefakte liegen als plist vor — LaunchAgents, LaunchDaemons, Preferences, App-Container, QuickLook, Spotlight und viele mehr.

# plist in XML konvertieren
plutil -convert xml1 com.apple.finder.plist -o finder_config.xml

# plist direkt auslesen mit PlistBuddy (/usr/libexec/PlistBuddy)
PlistBuddy -c "Print :DesktopViewSettings:IconViewSettings:arrangeBy" ~/Library/Preferences/com.apple.finder.plist

# plist in Python parsen
python3 -c "
import plistlib
with open('com.apple.loginwindow.plist', 'rb') as f:
    pl = plistlib.load(f)
    print(pl.get('lastUserName'))
    print(pl.get('lastLoginTimestamp'))
"
plist ist nicht gleich plist: macOS verwendet zunehmend JSON-basierte plist-Dateien (z.B. in LaunchAgents), insbesondere fuer Drittanbieter-Software. Tools muessen sowohl XML- als auch JSON-Plist unterstuetzen. Der file-Befehl zeigt "Apple binary property list" fuer binaere Plists.

macOS-spezifische Artefakte

Spotlight-Datenbank

Spotlight (/.Spotlight-V100/ auf jedem Volume) indiziert alle Dateiinhalte und Metadaten. Die Datenbank enthaelt eine vollstaendige Liste aller Dateien auf dem Volume, auch solcher die in anderen Artefakten nicht auftauchen. Loescht man eine Datei, bleibt der Spotlight-Eintrag oft noch erhalten — Spotlight kann als Referenz fuer geloeschte Dateien dienen.

# Spotlight-Datenbank parsen (read-only)
# Verzeichnis: /.Spotlight-V100/Store-V2/[UUID]/
# Dateien: store.db (SQLite), .store.db (Journal)
mdfind "kMDItemFSName == 'passwords.xlsx'"

QuickLook Thumbnails

QuickLook generiert Vorschaubilder fuer fast jeden Dateityp und speichert sie unter /var/folders/. Auch nach Loeschung der Originaldatei koennen Thumbnails erhalten bleiben — ein starkes Indiz dafuer, dass eine Datei existiert hat.

Recent Items / zuletzt benutzte Objekte

# Recent Items auslesen (gespeichert in verschiedenen plists)
plutil -p ~/Library/Preferences/com.apple.recentitems.plist
# Enthaelt: zuletzt geoeffnete Apps, Dokumente, Server — sortiert nach Zeit
Artefakt-PfadTypInhalt
~/Library/Preferences/com.apple.finder.plistplistFinder-Einstellungen, zuletzt geoeffnete Ordner (FXRecentFolders)
~/Library/Preferences/com.apple.LaunchServices.QuarantineEventsV2SQLiteQuarantine-Download-Datenbank

LaunchAgents sind die macOS-Entsprechung zu Registry-RunKeys: Ein Angreifer kann eine .plist-Datei in ~/Library/LaunchAgents/ ablegen, die bei der naechsten Anmeldung automatisch ausgefuehrt wird. Der Pfad ~/Library/LaunchAgents/com.apple.[random].plist mit Pfad zu /tmp/malware ist ein klassischer Persistenz-Indikator.

Triage-Skript fuer macOS

Ein grundlegendes Bash-Triage-Skript sammelt die wichtigsten fluechtigen und persistenten Artefakte eines macOS-Systems in einem Durchlauf:

#!/bin/bash
# macOS Live-Triage — schnelle Erstanalyse
OUTDIR="triage_$(hostname)_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$OUTDIR"
echo "[*] macOS Triage startet — Output: $OUTDIR"

# System-Info
sysctl -n kern.boottime > "$OUTDIR/boot_time.txt"
sysctl -n kern.osversion > "$OUTDIR/os_version.txt"
uptime > "$OUTDIR/uptime.txt"

# Netzwerkverbindungen und offene Ports
lsof -i -nP > "$OUTDIR/network_connections.txt"
netstat -an > "$OUTDIR/netstat.txt"
arp -a > "$OUTDIR/arp_cache.txt"

# Prozessliste (mit Parent-Child-Beziehung)
ps auxwww > "$OUTDIR/processes.txt"
ps -eo pid,ppid,user,command > "$OUTDIR/process_tree.txt"

# Geladene Kernel-Extensions (KEXTs) — Rootkit-Indikator
kextstat > "$OUTDIR/kextstat.txt"

# Unified Logs der letzten 7 Tage sichern
log collect --last 7d --output "$OUTDIR/unified_logs.logarchive"

# Quarantine-Datenbank kopieren
cp ~/Library/Preferences/com.apple.LaunchServices.QuarantineEventsV2 "$OUTDIR/"

# LaunchAgents und LaunchDaemons sichern
cp -r /Library/LaunchAgents "$OUTDIR/system_launchagents"
cp -r /Library/LaunchDaemons "$OUTDIR/system_launchdaemons"
cp -r ~/Library/LaunchAgents "$OUTDIR/user_launchagents"

# FSEvents-Datenbank falls vorhanden (root erforderlich)
sudo cp -r /.fseventsd "$OUTDIR/fsevents" 2>/dev/null

# Bash/Zsh-History
cp ~/.bash_history "$OUTDIR/bash_history" 2>/dev/null
cp ~/.zsh_history "$OUTDIR/zsh_history" 2>/dev/null

# KnowledgeC-DB
sudo cp /var/db/CoreDuet/Knowledge/knowledgeC.db "$OUTDIR/" 2>/dev/null

# Alle Outputs in TAR verpacken
tar -czf "$OUTDIR.tar.gz" "$OUTDIR"
echo "[+] Triage abgeschlossen: $OUTDIR.tar.gz"
Dieses Skript NICHT blind ausfuehren: Es liest nur (keine Aenderungen am System), aber root-Rechte sind fuer einige Quellen erforderlich. Auf einem kompromittierten System: Immer von einem externen, vertrauenswuerdigen Medium booten, wenn moeglich.

Beispiel: Malware-Analyse auf macOS

Szenario: Nutzer meldet, sein Mac sei seit 3 Tagen "langsam und komisch" — unerwartete Popups, Luefter laeuft auf Maximaldrehzahl.

  1. Persistenz pruefen: ls -la ~/Library/LaunchAgents/ und /Library/LaunchDaemons/ — alle .plist-Dateien auf ungewoehnliche Pfade und Programmnamen pruefen
  2. Quarantine-DB checken: Welche Downloads in den letzten 7 Tagen? Stimmt die Download-URL mit bekannten Malware-Domains ueberein?
  3. Prozesse anschauen: ps auxwww | grep -i -E "crypt|miner|coin|tor" — typische Indikatoren fuer Crypto-Miner
  4. Netzwerkverbindungen: lsof -i -nP | grep ESTABLISHED — welche ausgehenden Verbindungen sind offen? Ports > 1024 zu unbekannten IPs?
  5. Kernel-Extensions: kextstat | grep -v com.apple — nicht-Apple-KEXTs sind extrem verdaechtig auf macOS
  6. FSEvents analysieren: Welche Dateien wurden in den Troubleshooting-Tagen erstellt/modifiziert/gelöscht?
macOS-Triage ist zeitkritisch: Unified Logs rotieren, FSEvents-Daten werden ueberschrieben, KnowledgeC loescht alte Eintraege. Bei Verdacht SOFORT Triage-Skript ausfuehren und die fluechtigen Logs sichern — nicht erst eine Stunde lang recherchieren.
