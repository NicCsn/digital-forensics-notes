Kurzueberblick

Container (Docker, Podman) sind keine VMs — sie nutzen den Host-Kernel und sind per Namespaces isoliert. Forensisch bedeutet das: Container-Prozesse sind Host-Prozesse, Container-Dateisysteme sind Host-Dateien, und ein Container-Escape macht aus einem kompromittierten Container ein kompromittiertes Host-System.

Container vs. VM — Technische Unterschiede und forensische Implikationen

MerkmalContainer (Docker, Podman, LXC)Virtuelle Maschine (VMware, KVM, Hyper-V)
IsolationLinux Namespaces (PID, NET, MNT, UTS, IPC, User, Cgroup)Hypervisor (Ring -1) — vollstaendige virtuelle Hardware
KernelTeilt Host-Kernel — kein eigener KernelEigener Kernel (Gast-OS), eigener Treiber-Stack
Startzeit< 1 Sekunde (nur Prozess-Start)10–60 Sekunden (Boot-Prozess)
Disk-GroesseImage-Layer (meist MB bis wenige GB)Vollstaendiges Disk-Image (GB bis TB)
AngriffsvektorContainer-Escape → Host-Kernel-KompromittierungVM-Escape → Hypervisor-Kompromittierung
Forensik-AnsatzHost-Forensik: /var/lib/docker/, ps aux, Kernel-LogsImage-Forensik: VMDK/VHDX mit TSK/Autopsy analysieren
Fluechtige DatenRAM nur im Host-Kontext analysierbarVM-RAM als separates Dump-File analysierbar (VMEM, vmsn)
Forensische GoldgrubeContainer-Logs, Config-JSON, Layer-DiffsSnapshots, Memory-Dumps, Hypervisor-Logs
Forensische Faustregel: Container = Host-Forensik mit einer zusaetzlichen Abstraktionsschicht. VM = komplette Gast-Forensik plus Hypervisor-Forensik. VM ermoeglicht tiefere Isolation und damit mehr forensische Spuren auf Gast-Ebene, Container ermoeglicht direkte Einsicht vom Host aus.

Docker-Storage-Treiber: overlay2 im Detail

Funktionsweise

# overlay2 = Standard-Treiber (seit Docker 18.06)
# Arbeitet mit Verzeichnis-Overlay:

/var/lib/docker/overlay2/
├── <layer-id>/
│   ├── diff/          ← Dateien dieses Layers (Inhalt)
│   ├── link           ← Kurzer Link-Name (z.B. "ABC123")
│   ├── lower          ← Liste der unteren Layer-ID-Links
│   ├── merged/        ← Zusammengefuehrte Sicht (alle Layer uebereinander)
│   ├── work/          ← Arbeitsverzeichnis des OverlayFS
│   └── committed      ← Markierungsdatei: Layer ist committed

# Ein laufender Container hat als obersten Layer:
#   UpperDir = /var/lib/docker/overlay2/<container-layer>/diff/
#   LowerDir = Alle Image-Layer (read-only)
#   Merged   = Zusammengefuehrte Sicht = das, was der Container als "/" sieht
#   WorkDir  = OverlayFS-Arbeitsverzeichnis
Forensisch wichtig: UpperDir/diff/ enthaelt ALLE Aenderungen, die der Container am Dateisystem vorgenommen hat. Das ist der forensisch wertvollste Ordner — hier sind Malware-Downloads, modifizierte Konfigurationsdateien und geloeschte Dateien (als Whiteout-Character-Devices) zu finden.

OverlayFS-Whiteouts

# Wenn eine Datei im Container geloescht wird, die aus einem Lower-Layer stammt:
# - OverlayFS erstellt eine Whiteout-Datei im UpperDir:
#   char device 0:0 mit Namen ".wh.<originaler_dateiname>"
# - Das Merged-Ergebnis zeigt die Datei nicht mehr an
# - Forensisch: Whiteout-Dateien im UpperDir/diff/ zeigen, WAS geloescht wurde!

# Finde alle geloeschten Dateien:
find /var/lib/docker/overlay2/*/diff/ -name ".wh.*" -type c

Vergleich der Storage-Treiber

TreiberForensische EignungAnmerkung
overlay2HervorragendAlle Layer als Verzeichnisse direkt lesbar. Whiteouts zeigen Loeschvorgaenge. Standard-Treiber — den wirst du in 99% der Faelle antreffen.
aufsGutAehnlich overlay2. Vorgaenger. Verzeichnisse unter /var/lib/docker/aufs/.
devicemapperMittelBlock-Device-basiert. Thin-pool Snapshots. Braucht thin_check/thin_dump zum Auslesen. Aeltere Docker-Versionen.
btrfsMittelSubvolumes + Snapshots. Btrfs-Tools noetig. Selten in Produktion (meist nur SuSE).

Forensische Artefakte — Fundorte und Analyse

/var/lib/docker/ — die zentrale Fundgrube

PfadInhaltForensische Bedeutung
/var/lib/docker/containers/<ID>/<ID>-json.logstdout/stderr des ContainersApplikations-Logs — persistiert auch nach docker rm!
/var/lib/docker/containers/<ID>/config.v2.jsonContainer-KonfigurationGoldgrube: Env-Vars (inkl. Secrets!), CMD, Entrypoint, Mounts, Ports, Networks
/var/lib/docker/containers/<ID>/hostconfig.jsonHost-KonfigurationPrivileged-Flag, Capabilities, Memory-Limits, Restart-Policy
/var/lib/docker/containers/<ID>/mounts/Volume-Mounts (alte Docker-Versionen)Gemountete Volumes — Schattenkopie?
/var/lib/docker/overlay2/<layer-id>/diff/Dateisystem-LayerContainer-Dateisystem — Malware-Downloads, modifizierte Configs
/var/lib/docker/image/overlay2/layerdb/Image-Layer-MetadatenLayer-ID-Mapping, Cache-IDs, Parent-Beziehungen
/var/lib/docker/volumes/Docker-VolumesPersistente Daten — ueberleben docker rm
Achtung: Container-Logs (*-json.log) werden NIE automatisch geloescht — auch docker rm entfernt nur den Container, nicht das Log-Verzeichnis! Erst docker system prune oder manuelles Loeschen der Container-ID-Verzeichnisse entfernt sie. Das bedeutet: Logs von vor Wochen geloeschten Containern koennen noch existieren.

Live-Forensik — Container-Prozesse auf dem Host

# Container-Prozesse sind normale Host-Prozesse!
ps aux | grep nginx       # Container-Nginx erscheint als Host-Prozess
ls /proc/<PID>/ns/        # Namespace-Zuordnung eines Container-Prozesses

# PID-Mapping: Container-PID → Host-PID
docker inspect --format '{{.State.Pid}}' <container>
ls /proc/<Host-PID>/root/  # Container-Dateisystem vom Host aus erreichen!

# Netzwerk: iptables-Regeln fuer Port-Mappings
iptables -t nat -L -n -v    # DNAT-Regeln zeigen Host-Port → Container-Port
ip netns list                # Docker-Netzwerk-Namespaces

Image Layer Analyse

Docker-Image-Historie und Inspektion

# Layer-Historie eines Images anzeigen:
docker history nginx:latest
docker history --no-trunc nginx:latest     # Voller Befehl pro Layer
docker image inspect nginx:latest | jq '.[0].RootFS.Layers'

# Container-Aenderungen anzeigen (vor stop/rm):
docker diff <container>                    # C=Created, A=Added, D=Deleted

# Image als Tar exportieren (forensisches Image):
docker save nginx:latest -o nginx_forensic.tar
tar -xf nginx_forensic.tar
# Enthaelt: layer.tar pro Layer + manifest.json + Config

# Layer einzeln analysieren:
mkdir layer_analysis
for f in */layer.tar; do
  id=$(dirname "$f")
  mkdir "layer_analysis/$id"
  tar -xf "$f" -C "layer_analysis/$id"
done

# Container als Image sichern (vor shutdown):
docker commit <container> forensic-snapshot:$(date +%Y%m%d-%H%M%S)
docker save forensic-snapshot:$(date +%Y%m%d-%H%M%S) -o forensic.tar

Container-Escape — Erkennung und Analyse

Escape-MethodeTechnikErkennung in Artefakten
Privileged Containerdocker run --privileged — alle Devices und Capabilitieshostconfig.json: "Privileged": true
Container hat /dev/sda, /dev/mem, alle Capabilities
Docker Socket Mount-v /var/run/docker.sock:/var/run/docker.sockconfig.v2.json: Mount-Eintrag auf /var/run/docker.sock
Container-Logs: docker run-Befehle vom Container aus
CAP_SYS_ADMINErlaubt Mount-Operationen im Containerhostconfig.json: "CapAdd": ["SYS_ADMIN"]
Kernel-Log: ungewoehnliche Mounts im NS-Kontext
CVE-2019-5736 (runc)Ueberschreibt runc-Binary vom Container ausHash von /usr/sbin/runc pruefen; Container mit /proc/self/exe-Zugriff
CVE-2022-0847 (Dirty Pipe)Schreibt in beliebige Page Cache Seiten (Kernel >= 5.8)Kernel-Version pruefen; ungewoehnliche splice()-Syscalls
CVE-2024-21626 (runc)File-Descriptor-Leak via WORKDIR /proc/self/fd/Dockerfile-Inspektion; Host-Dateisystem-Spuren; runc-Version pruefen
Escape-Indikatoren (Host-basierte Erkennung):
  • Ungewoehnliche Prozesse ausserhalb des erwarteten Namespace
  • /proc/<PID>/root-Symlinks, die auf Host-Root zeigen
  • Neue Netzwerk-Sockets mit Quell-IP des Containers aber Ziel-IP ausserhalb der Docker-Bridge
  • Unerwartete Kernel-Module geladen
  • Mount-Operationen im Container, die Host-Geraete einbinden (/dev/sda)

Kubernetes-Forensik

Control Plane — Artefakte

ArtefaktFundortAussage
etcd/var/lib/etcd/ auf Control-Plane-NodesKompletter Cluster-Zustand — alle Objects (Pods, Secrets, ConfigMaps) als Key-Value-Store. etcdctl get / --prefix --keys-only
Audit Logs/var/log/kubernetes/audit.log (falls konfiguriert)Wer hat wann welchen API-Aufruf getaetigt? kubectl exec, kubectl apply, kubectl delete
kube-apiserver Logs/var/log/pods/kube-system_kube-apiserver-*/API-Aufrufe — Request-Bodies enthalten Secrets!
kubelet Logsjournalctl -u kubeletPod-Starts/-Stopps, Container-Neustarts, Image-Pulls

Workload — Artefakte

ArtefaktZugriffAussage
Pod-Definitionkubectl describe pod <pod> -o yamlImage, Env-Vars, Mounts, SecurityContext, Capabilities
Pod-Logskubectl logs <pod> --previousLogs des VORHERIGEN Containers (nach Crash/Restart)!
Eventskubectl get events --all-namespacesPod-Starts, Image-Pulls, OOM-Kills, Schedule-Fehler
Secretskubectl get secrets -o yamlBase64-codierte Secrets (einfach dekodierbar: echo <base64> | base64 -d)
ConfigMapskubectl get configmaps -o yamlUmgebungsvariablen, Config-Dateien
Kubernetes-Forensik-Best-Practices:
  1. etcd-Dump als erste Massnahme — enthaelt den gesamten Cluster-Zustand.
  2. Audit-Logs aktivieren (muss vorher konfiguriert sein, Feature: Audit-Policy).
  3. --previous nicht vergessen! Restartende Container haben oft die interessanteren Logs.
  4. Node-Ebene nicht vergessen: Container-Dateisysteme liegen auf den Worker-Nodes wie normale Docker-Container unter /var/lib/containerd/ oder /var/lib/docker/.

Praktische Anwendung — Docker/K8s-Forensik-Toolkit

# ---- Docker-Forensik-Kommandos ----

# Alle Container (auch gestoppte) anzeigen
docker ps -a

# Container-Konfiguration komplett ausgeben
docker inspect <container> | jq .

# Container-Logs sichern (JSON-File-Treiber)
cp /var/lib/docker/containers/<ID>/<ID>-json.log /evidence/

# Container-Dateisystem vom Host sichern
docker export <container> -o container_filesystem.tar
# ODER: direkt vom Host-Dateisystem:
tar -czf /evidence/layer_diff.tar.gz /var/lib/docker/overlay2/<layer-id>/diff/

# Image-Layer inspizieren (dive Tool)
dive <image:tag>

# Docker-Diff vor Container-Stopp:
docker diff <container> > /evidence/diff.txt

# Alle Docker-Logs sammeln (Skript):
for dir in /var/lib/docker/containers/*/; do
  id=$(basename "$dir")
  cp "$dir/$id-json.log" "/evidence/containers/$id-json.log"
  cp "$dir/config.v2.json" "/evidence/containers/$id-config.json"
  cp "$dir/hostconfig.json" "/evidence/containers/$id-hostconfig.json"
done

# ---- Kubernetes-Forensik-Kommandos ----

# Kompletten Cluster-Zustand sichern
kubectl get all --all-namespaces -o yaml > /evidence/cluster-state.yaml

# Alle Secrets und ConfigMaps sichern
kubectl get secrets --all-namespaces -o yaml > /evidence/secrets.yaml
kubectl get configmaps --all-namespaces -o yaml > /evidence/configmaps.yaml

# Pod-Logs (inkl. vorheriger Container) sammeln
for ns in $(kubectl get ns -o name | cut -d/ -f2); do
  for pod in $(kubectl get pods -n $ns -o name | cut -d/ -f2); do
    kubectl logs -n $ns $pod --all-containers > "/evidence/logs/${ns}_${pod}.log"
    kubectl logs -n $ns $pod --all-containers --previous > "/evidence/logs/${ns}_${pod}_previous.log"
  done
done

# etcd-Dump (braucht Zugriff auf Control Plane Node)
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get / --prefix --keys-only

# Audit-Log auf verdaechtige Events durchsuchen
grep -E "(exec|create|delete)" /var/log/kubernetes/audit.log | jq '.user.username, .requestURI'
