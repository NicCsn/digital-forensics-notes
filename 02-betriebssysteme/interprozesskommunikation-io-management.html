Kurzüberblick

Interprozesskommunikation (IPC) umfasst alle Mechanismen, mit denen Prozesse untereinander Daten austauschen und ihr Verhalten synchronisieren. Das IO-Management kontrolliert, wie das Betriebssystem Ein- und Ausgabeoperationen mit Geräten (Festplatten, Netzwerkkarten, Terminals) plant und ausführt. Beide Konzepte sind fundamental für das Verständnis von Betriebssystemen — und für die forensische Analyse: Malware nutzt IPC für Injection und Kommunikation, während IO-Performance-Muster Hinweise auf Angriffsaktivitäten geben können.

MHB-Mapping: BDF102 (Betriebssysteme) — Vertiefung der Prozesskommunikation und des IO-Subsystems.

Pipes: Anonym und Named (FIFO)

Pipes sind die älteste und einfachste Form der IPC unter Unix/Linux. Sie verbinden die Standardausgabe (stdout) eines Prozesses mit der Standardeingabe (stdin) eines anderen.

Anonyme Pipes

Existieren nur, solange der erzeugende Prozess läuft. Können nur zwischen verwandten Prozessen verwendet werden (Parent-Child, erzeugt via pipe()-Systemaufruf vor einem fork()). Unix-Pipes in der Shell sind anonyme Pipes: cat /var/log/auth.log | grep "Failed" | wc -l.

EigenschaftUnix-Pipe (|)pipe()-Syscall
RichtungUnidirektional (einseitig)Unidirektional; bidirektional nur mit zwei Pipes
SichtbarkeitNur als geöffneter File-Descriptor (fd)Fd[0] = Lese-Ende, fd[1] = Schreib-Ende
Kapazität64 KB (Linux, PIPE_BUF)Gleicher Kernel-Buffer, blockierend bei Überlauf
LebensdauerBis beide Enden geschlossen sindWird vom Kernel freigegeben, wenn keine fd mehr referenzieren

Named Pipes (FIFO)

Haben einen Eintrag im Dateisystem (via mkfifo) und können von unabhängigen Prozessen verwendet werden. Sie erscheinen als reguläre Datei, verhalten sich aber wie eine Pipe: Daten werden nie auf die Platte geschrieben, sondern im Kernel-Buffer gehalten.

# Named Pipe erstellen und verwenden
mkfifo /tmp/forensic_pipe
# Terminal 1: Schreiber
echo "evidence_block_001" > /tmp/forensic_pipe
# Terminal 2: Leser
cat /tmp/forensic_pipe
# Ausgabe: evidence_block_001
# Nach dem Lesen ist die Pipe leer — Daten sind konsumiert
Forensisches Problem: Named Pipes hinterlassen im Dateisystem einen Inode (via mkfifo), aber keine Inhaltsdaten. Der Datenstrom existiert nur im Kernel-Speicher und ist nach dem Lesen verloren. Eine Memory-Dump-Analyse kann unter Umständen Reste des Pipe-Buffers enthalten — aber nur, wenn die Pipe zum Zeitpunkt des Dumps noch Daten enthielt.

Signals: SIGINT, SIGKILL, SIGTERM und mehr

Signals sind asynchrone Unterbrechungen, die das Betriebssystem an einen Prozess sendet. Sie dienen der Benachrichtigung über Ereignisse (Timer-Ablauf, ungültiger Speicherzugriff) oder der Steuerung des Prozesslebenszyklus.

Wichtige Standardsignals

SignalNummerStandardverhaltenKann blockiert werden?Forensischer Kontext
SIGKILL9Sofortige Terminierung, kein CleanupNeinAngreifer killt Forensik-Tool; Prozess kann sich nicht wehren
SIGTERM15Höfliche Bitte um TerminierungJa (kann ignoriert werden)Standard-Kill; Malware kann SIGTERM abfangen und sich verschleiern
SIGINT2Terminierung nach Ctrl+CJaForensik-Tool mit Ctrl+C abbrechen: Datenverlust möglich
SIGSEGV11Core-Dump + TerminierungJa (Handler, dann weiterlaufen ist gefährlich)Buffer-Overflow-Exploit in Forensik-Parser — Core-Dump analysieren!
SIGSTOP19Prozess anhalten (pausieren)NeinDebugger-Trick: Malware pausieren, dann Memory-Dump
SIGCONT18Angehaltenen Prozess fortsetzenNeinNach Memory-Dump fortsetzen; Malware merkt Zeit-Sprung
SIGUSR1/210/12BenutzerdefiniertJaForensik-Tools nutzen SIGUSR1 für Status-Reports während Imaging
# Signal-Handler in Python für forensisches Imaging-Tool
import signal
import sys

def handle_sigterm(signum, frame):
    print("[!] SIGTERM erhalten — schließe Image-Datei sauber...")
    cleanup()
    sys.exit(0)

def handle_sigusr1(signum, frame):
    print(f"[*] Status: {current_sector}/{total_sectors} Sektoren geschrieben")

signal.signal(signal.SIGTERM, handle_sigterm)
signal.signal(signal.SIGUSR1, handle_sigusr1)
Forensik-Strategie: Ein Angreifer, der ein Forensik-Tool mit SIGKILL beendet, verhindert ordnungsgemäßen Cleanup. Deshalb verwenden viele Imaging-Tools Write-Ahead-Logs: Selbst nach SIGKILL kann der nächste Start den unterbrochenen Vorgang wiederherstellen.

Sockets: Unix Domain, TCP, UDP

Sockets sind der universellste IPC-Mechanismus. Sie abstrahieren die Kommunikation über einen Endpunkt — lokal (Unix Domain) oder über das Netzwerk (TCP/UDP). Forensisch relevant, weil sie sowohl legitime (DBus, Systemd, X11) als auch bösartige (C2, Reverse Shell) Verbindungen transportieren.

Socket-TypAdressierungReichweiteForensische Fundstelle
Unix Domain (AF_UNIX)Pfad im Dateisystem (z.B. /var/run/docker.sock)Lokal (gleicher Host)ss -xlp, /proc/net/unix, Inode im Dateisystem
TCP (AF_INET, SOCK_STREAM)IP:Port (z.B. 192.168.1.1:443)Netzwerkweit (verbindungsorientiert)ss -tnp, /proc/net/tcp, PCAP-Dateien
UDP (AF_INET, SOCK_DGRAM)IP:PortNetzwerkweit (verbindungslos)ss -unp, /proc/net/udp, NetFlow-Daten

Unix Domain Sockets in der Forensik

Werden von vielen Systemdiensten genutzt: systemd-journald (Logs), Docker (Container-Management), MySQL (lokale Clients), PHP-FPM. Malware nutzt sie für lokale Kommunikation zwischen Komponenten — z.B. ein Dropper, der mit einem bereits installierten Keylogger kommuniziert.

# Unix Domain Socket Spurensuche
ss -xlp | grep -v "^$"       # Alle aktiven Unix-Sockets mit Prozess-Zuordnung
lsof -U                        # Offene Unix-Sockets auflisten
find / -type s 2>/dev/null    # Alle Socket-Dateien im Dateisystem finden

# Forensisch: Socket-Inode → Prozess-Zuordnung via /proc
ls -la /proc/*/fd 2>/dev/null | grep socket

Shared Memory: shmget/shmat und mmap

Shared Memory ist die schnellste IPC-Methode: Zwei oder mehr Prozesse teilen sich einen physischen Speicherbereich direkt, ohne Kopiervorgänge über den Kernel (Zero-Copy). Linux implementiert Shared Memory über das tmpfs-Dateisystem (meist unter /dev/shm).

System-V Shared Memory

SyscallZweck
shmget(key, size, flags)Erzeugt/öffnet ein Shared-Memory-Segment. Der Key identifiziert es systemweit.
shmat(shmid, addr, flags)Hängt das Segment in den Adressraum des aufrufenden Prozesses ein (attach).
shmdt(addr)Löst das Segment vom Adressraum (detach).
shmctl(shmid, cmd, buf)Kontrolloperationen: Löschen, Berechtigungen ändern, Status abfragen.

mmap-basiertes Shared Memory

Mit mmap(MAP_SHARED) können Prozesse eine Datei (oder anonyme Page) in ihren Adressraum mappen. Änderungen werden für alle Prozesse sichtbar. Dies ist der bevorzugte Weg unter Linux, weil POSIX-konform und flexibler als SysV-SHM.

# Shared Memory Spurensuche (Linux)
ipcs -m                # Alle SysV-Shared-Memory-Segmente auflisten
ls -la /dev/shm/       # POSIX-Shared-Memory (tmpfs-mapped)
cat /proc/*/maps | grep "/dev/shm"  # Welcher Prozess hat was gemappt?

# Relevant: Malware verwendet /dev/shm für temporäre Dateien
# — kein Schreiben auf die Platte, keine Dateisystem-Artefakte!
Forensische Warnung: Shared Memory als Angriffsvektor. DLL-Injection unter Windows nutzt Shared Memory, um Schadcode in legitime Prozesse einzuschleusen. Der Angreifer alloziert Shared Memory, schreibt Shellcode hinein und zwingt einen Ziel-Prozess (via CreateRemoteThread), diesen Code auszuführen. Die forensische Untersuchung solcher Injektionen erfordert Memory-Analyse (Volatility Plugin: malfind, ldrmodules), da der injizierte Code nie auf der Platte existiert hat.

Message Queues: POSIX mq und SysV msg

Message Queues erlauben strukturierte, nachrichtenbasierte Kommunikation mit Prioritätssteuerung. Anders als Pipes sind Nachrichten typisiert und unabhängig voneinander lesbar — kein reiner Byte-Strom.

SysV Message Queues vs. POSIX Message Queues

MerkmalSysV msgPOSIX mq
IdentifikationInteger-Key (IPC_PRIVATE, ftok())Name (String, wie /forensic_queue)
APImsgget(), msgsnd(), msgrcv()mq_open(), mq_send(), mq_receive()
PrioritätenGanze Nachricht priorisiert (msgtyp)Nachrichten-Priorität (0–sysconf)
PersistenzKernel-Persistenz (bis Reboot oder explizites Löschen)Kernel-Persistenz + optional als Datei unter /dev/mqueue
BenachrichtigungKeine native Benachrichtigungmq_notify(): Signal/Thread bei neuer Nachricht
# POSIX Message Queue forensisch inspizieren
mount -t mqueue none /dev/mqueue   # Mqueue-Dateisystem mounten
ls -la /dev/mqueue/                # Alle aktiven Queues anzeigen
cat /dev/mqueue/my_queue           # Queue-Metadaten (kein Inhalt — nur Größe, Max-Nachrichten etc.)

# SysV Message Queue inspizieren
ipcs -q                            # Alle aktiven SysV-Queues
ipcrm -q                    # Queue löschen (während Live-Analyse)

Semaphore: POSIX sem vs. SysV sem

Semaphore sind Synchronisationsprimitive, die den Zugriff auf gemeinsam genutzte Ressourcen koordinieren. Anders als Mutexe (binär: gesperrt/entsperrt) sind Semaphore Zähler: Ein Zählerstand N erlaubt N gleichzeitige Zugriffe.

TypAPIPersistenzSichtbarkeit
SysV Semaphore-Setssemget(), semop(), semctl()Kernel-Persistenz (bis Reboot / explizites Löschen)ipcs -s listet alle Sets
POSIX Named Semaphoresem_open(), sem_wait(), sem_post()Kernel-Persistenz; erscheint unter /dev/shm/sem.*Als tmpfs-Datei im Dateisystem sichtbar
POSIX Unnamed Semaphoresem_init(), sem_wait(), sem_post()Im Shared Memory; nur für verwandte ProzesseNur via Memory-Analyse sichtbar (Thread-Shared)
Forensischer Hinweis: Orphaned Semaphore (Semaphore, deren besitzender Prozess beendet wurde, ohne sie freizugeben) können auf abgestürzte Malware oder fehlgeschlagene Angriffe hindeuten. ipcs -s zeigt Semaphore ohne zugehörigen Prozess — eine Untersuchung wert.

IO-Management: Block/Char/Network und IO-Scheduler

Gerätetypen im Linux-Kernel

TypMajor-NummerBeispieleZugriff
Block Devices (b)8, 259 (nvme)/dev/sda, /dev/nvme0n1, /dev/loop0Wahlfreier Zugriff, gepuffert via Page Cache
Character Devices (c)1, 4, 5/dev/tty, /dev/null, /dev/randomSequenzieller Bytestrom, ungepuffert

IO-Scheduler unter Linux

IO-Scheduler bestimmen die Reihenfolge, in der IO-Requests an Block-Devices gesendet werden. Linux verwendet seit Kernel 5.0 das Multi-Queue-Block-Layer (blk-mq) mit den folgenden Schedulern:

SchedulerPrinzipEinsatzForensische Bedeutung
none / noopFIFO-Reihenfolge, keine OptimierungVMs, SSDs (Hardware-eigenes Scheduling)Keine Request-Reihenfolge-Änderung; gut für deterministische Benchmarks
kyberLatenz-basiert, adaptiv; misst IO-Zeiten und drosselt bei SättigungSSDs, Mischlasten (Desktop/Server)Geringe Latenz für interaktive Anwendungen; erkennbar an gleichmäßiger IO-Last
bfq (Budget Fair Queueing)Garantiert jedem Prozess ein IO-Budget; verhindert, dass ein Prozess das Device monopolisiertHDDs, interaktive DesktopsProzess X schreibt 500 MB/s? BFQ bremst X, damit Prozess Y (Forensik-Tool) nicht blockiert
mq-deadlineDeadline pro Request (Read ~500ms, Write ~5000ms) + Sector-SortingSSDs mit geringer Latenz, ServerFaire Deadline-Verteilung; Writes werden zurückgestellt, bis Reads bedient wurden
# IO-Scheduler auslesen und ändern
cat /sys/block/sda/queue/scheduler
# Ausgabe: [mq-deadline] kyber bfq none
echo bfq > /sys/block/sda/queue/scheduler

# IO-Statistiken für forensische Performance-Analyse
iostat -x 1                    # Laufende IO-Statistiken (await, svctm, %util)
cat /proc/diskstats            # Rohdaten der Disk-Statistiken
ioping -c 10 /dev/sda          # IO-Latenz messen (ähnlich ping für Disk)

DMA und IRQ

DMA (Direct Memory Access): Ermöglicht Peripheriegeräten, direkt in den Hauptspeicher zu schreiben, ohne die CPU für jeden Byte-Transfer zu unterbrechen. Ein DMA-Controller übernimmt den Transfer; die CPU wird nur bei Abschluss per Interrupt benachrichtigt.

IRQ (Interrupt Request): Hardware-Signal, das die CPU auf ein Ereignis aufmerksam macht: "Daten sind angekommen", "Festplatte hat Sektor gelesen", "Netzwerk-Paket im Puffer". Linux verwaltet IRQs über /proc/interrupts. Malware kann IRQ-Handler modifizieren (Kernel-Rootkit), um IO-Operationen zu verbergen.

Forensische Relevanz

IPC als Angriffsvektor — Live-Analyse

IPC-MechanismusAngriffs-SzenarioForensische Spur (Linux)
Shared MemoryDLL-Injection: Schadcode in legitimen Prozess injizieren/proc/PID/maps auf anonyme/verdächtige RWX-Bereiche prüfen; Volatility malfind
Unix Domain SocketsRootkit kommuniziert mit Userland-Komponente über versteckten Socketss -xlp, lsof -U, /proc/net/unix auf unbekannte Sockets untersuchen

IO-Scheduler als Performance-Indikator

Ein plötzlicher Anstieg der IO-Wartezeit (iostat await) kann auf Ransomware hindeuten, die tausende Dateien parallel verschlüsselt. Ein ungewöhnliches Muster aus kurzen Reads und langen Writes deutet auf Datenexfiltration hin (Lesen + Netzwerk-Senden). Die Korrelation von IO-Mustern mit Netzwerkaktivität ist ein starkes forensisches Signal.

Forensik-Tipp: Bei der Live-Analyse immer ipcs -a (alle IPC-Objekte) und ss -tulpn (alle Sockets) ausführen, bevor das System heruntergefahren wird. IPC-Objekte sind flüchtig — sie überleben einen Reboot nicht. Dokumentieren Sie den Zustand mit Screenshots und Log-Dumps.
