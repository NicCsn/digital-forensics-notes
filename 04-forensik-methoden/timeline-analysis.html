Kurzueberblick

Die Timeline-Analyse ist eine der maechtigsten forensischen Methoden. Sie aggregiert samtliche Zeitstempel aus allen verfuegbaren Artefakten eines Systems in einer einzigen, chronologisch sortierten Liste — der Super-Timeline. Entwickelt von Kristinn Gudjonsson, ist Plaso (Plaso Langar Að Safna Öllu — "Plaso moechte alles sammeln") das Open-Source-Standardwerkzeug fuer diese Aufgabe. Es extrahiert Timestamps aus Dateisystemen, Registry, EventLogs, Prefetch, Browser-History, Logdateien und ueber 50 weiteren Artefakttypen und normalisiert sie in ein einheitliches Format.

Super-Timeline vs. einfache Timeline: Eine einfache Timeline zeigt nur Dateisystem-Zeitstempel (MACTIME). Eine Super-Timeline kombiniert Dateisystem-, Registry-, Log-, Netzwerk- und Browser-Artefakte in einer Ansicht. Das ermoeglicht Korrelationen wie: "Datei wurde um 14:03 erstellt, um 14:04 in der Registry referenziert und um 14:07 via Browser heruntergeladen."

Plaso/log2timeline Installation und Grundbefehle

Plaso ist in Python geschrieben und wird als Python-Paket installiert. Es stellt drei Kern-Tools bereit: log2timeline.py (Extraktion), psort.py (Sortierung/Ausgabe) und psteal.py (beides in einem Durchlauf).

Installation

# Via pip (empfohlen)
pip install plaso

# Via Docker (isoliert, keine Abhaengigkeitsprobleme)
docker pull log2timeline/plaso

# Auf Debian/Ubuntu (oft veraltet)
sudo apt install plaso-tools

Kernbefehle

WerkzeugFunktionTypische Verwendung
log2timeline.pyExtrahiert Timestamps aus einem Quell-Image/Verzeichnis und schreibt eine .plaso-Storage-Dateilog2timeline.py timeline.plaso image.dd
psort.pyLiest eine .plaso-Datei und gibt die Events sortiert in verschiedenen Formaten auspsort.py timeline.plaso "date > '2024-01-01'"
psteal.pyKombination aus log2timeline und psort — Extraktion und Ausgabe in einem Befehlpsteal.py --source image.dd -o l2tcsv -w output.csv
Plaso-Storage-Dateien sind binaer: Die .plaso-Datei ist kein Textformat, sondern eine SQLite-Datenbank mit serialisierten Protobuf-Objekten. Sie kann nur mit Plaso-Tools gelesen werden. Fuer den Export in lesbare Formate (CSV, L2TCSV, TLN, etc.) wird psort.py verwendet.

Datenquellen fuer Timelines

Plaso unterstuetzt ueber 50 Parser. Die wichtigsten Datenquellen und ihre forensische Relevanz:

DatenquelleParser in PlasoRelevante Artefakte
NTFS-Dateisystemmft, filestat$MFT (Master File Table) mit $STANDARD_INFO (4 MACB-Zeitstempel) und $FILE_NAME (4 zusaetzliche MACB)
Windows RegistrywinregSAM, SECURITY, SOFTWARE, SYSTEM, NTUSER.DAT — UserAssist, RunMRU, ShimCache, AmCache, USB-Historie
Windows EventLogwinevt, winevtxSecurity.evtx (4624/4625 Login), System.evtx (7036 Dienststart), Application.evtx
PrefetchprefetchAusfuehrungszeitpunkte von Programmen, Run-Count, zuletzt geladene DLLs
Browser-Historychrome_history, firefox_historyChronik-Timestamps, Download-Zeiten, Cookie-Erstellungsdaten
LNK-DateienlnkErstellungs-/Modifikations-/Zugriffszeit der Verknuepfung sowie der Zieldatei, NetBIOS-Name des Ursprungsrechners
RAM-Dump (Volatility)Nicht direkt — via mactime-PluginProzess-Startzeiten, Netzwerkverbindungen mit Timestamps
Linux-Dateisystemfilestatinode-Timestamps (atime, mtime, ctime, crtime auf ext4/XFS)
$STANDARD_INFO vs. $FILE_NAME: Auf NTFS hat jede Datei zwei Sets von MACB-Timestamps. $STANDARD_INFO kann vom Benutzer manipuliert werden (Timestomping). $FILE_NAME wird nur vom Kernel aktualisiert und ist schwerer zu faelschen. Diskrepanzen zwischen beiden sind ein starker Indikator fuer Anti-Forensik.

Mactime-Format und MACB-Timestamps

Das Mactime-Format (auch als bodyfile-Format bekannt) ist der Urvater aller forensischen Timeline-Formate, urspruenglich vom Sleuth Kit (fls -m). Es listet Dateien mit ihren vier Zeitstempeln auf.

MACB — Die vier Zeitstempel

KuerzelNameWindows-EntsprechungBedeutung
mModifiedLastWriteTimeWann wurde der Inhalt zuletzt geaendert?
aAccessedLastAccessTimeWann wurde die Datei zuletzt gelesen? (Seit Win 8 oft deaktiviert)
cChangedChangeTime (MFT Entry Modified)Wann wurden die Metadaten (nicht Inhalt) geaendert?
bBirthCreationTimeWann wurde die Datei auf DIESEM Datentraeger erstellt?
# Mactime/Bodyfile-Beispielzeile
# MD5|Name|Inode|Mode|UID|GID|Size|atime|mtime|ctime|btime
0|/Users/jon/secret.docx|53849|-rw-r--r--|0|0|284672|1706119800|1706119740|1706119740|1706119200

Ein typischer Plaso-Export ins Mactime-Format mit psort:

psort.py -o l2tcsv timeline.plaso > timeline.csv

Die CSV enthaelt dann Spalten wie datetime, timestamp_desc, source, sourcetype, message — jede Zeile ein einzelnes Event, chronologisch sortiert.

Super-Timeline Workflow — Schritt fuer Schritt

Workflow-Uebersicht: Der klassische Plaso-Workflow besteht aus drei Phasen: (1) Extraktion aller Timestamps in eine Storage-Datei, (2) Filtern und Sortieren der relevanten Events, (3) Analyse und Korrelation in Timesketch oder manuell.

Phase 1: Extraktion

# Vollstaendige Extraktion aus einem Image
log2timeline.py --status_view window \
  --storage_file case001.plaso \
  /mnt/evidence/disk_image.E01

# Nur bestimmte Parser verwenden (schneller, gezielter)
log2timeline.py --parsers "winreg,prefetch,winevtx" \
  --storage_file registry_events.plaso \
  /mnt/evidence/disk_image.E01

Phase 2: Filtern und Sortieren

# Alle Events nach Datum filtern
psort.py -o dynamic \
  -q "date >= '2024-06-01 00:00:00' AND date <= '2024-06-15 23:59:59'" \
  timeline.plaso

# Nur Events von bestimmten Parsern
psort.py -o l2tcsv \
  -q "parser == 'prefetch' OR parser == 'winevtx'" \
  timeline.plaso > filtered.csv

# Nach Source-Typ filtern (z.B. nur Registry-Events)
psort.py -o l2tcsv \
  -q "source_short == 'REG'" \
  timeline.plaso

Phase 3: Analyse

Die exportierte CSV kann in Tools wie Timeline Explorer (Eric Zimmerman) oder Timesketch geladen werden. Manuell werden Events korreliert:

  • Welche Programme wurden vor/nach einem verdaechtigen Event ausgefuehrt?
  • Gibt es Dateisystem-Aktivitaet die mit Registry-Aenderungen korreliert?
  • Welche Prefetch-Eintraege tauchen zeitgleich mit Login-Events auf?

Timesketch-Integration

Timesketch ist eine von Google entwickelte Webanwendung zur kollaborativen forensischen Timeline-Analyse. Es baut auf Elasticsearch und Plaso auf und bietet eine grafische Oberflaeche mit Sketch-basierter Analyse.

Timesketch-Architektur

KomponenteFunktion
Timesketch Web UIWeb-Frontend (Flask), in dem Analysten Sketches erstellen, Events durchsuchen, Tags vergeben und Stories (Plots) anlegen
PlasoExtrahiert Timestamps aus Images und schreibt sie als .plaso-Datei
Timesketch Importer (tsctl)Importiert .plaso oder CSV-Dateien in Elasticsearch
ElasticsearchSpeichert und indiziert alle Events, ermoeglicht Volltextsuche und Aggregationen
# Workflow: Plaso → Timesketch
# 1. Timeline mit Plaso extrahieren (wie oben)
# 2. In Timesketch importieren
tsctl import timeline.plaso

# 3. In Timesketch Web-UI: Sketch anlegen, Timeline zuweisen
# 4. Analyse: Suchabfragen, Tags ("suspicious", "malware"), Plots erstellen
Timesketch-Analysepunkte: Analysiere Events mit aggregierten Ansichten (Haeufigkeitsdiagramme), suche nach Ausreissern (einzelne Events in sonst inaktiven Zeitraeumen) und nutze Sigma-Regeln, die Timesketch nativ als Analyzer unterstuetzt.

Elasticsearch Backend

Elasticsearch bildet das Rueckgrat von Timesketch. Jedes Plaso-Event wird als JSON-Dokument indiziert. Die Elasticsearch-Datenstruktur ermoeglicht komplexe Suchabfragen und Zeitreihenanalysen.

FeldTypBeschreibung
datetimedateZeitstempel des Events (ISO 8601)
timestamp_desckeywordBedeutung des Timestamps (z.B. "Creation Time", "Last Written")
messagetextFreitext-Beschreibung des Events
parserkeywordWelcher Plaso-Parser hat das Event erzeugt? (z.B. "winevtx", "mft")
source_shortkeywordKurzes Quell-Kuerzel ("FILE", "REG", "LOG", "WEBHIST")
source_longtextVollstaendiger Quellpfad
Performance: Eine vollstaendige Super-Timeline eines 500-GB-Images kann mehrere Millionen Events enthalten. Elasticsearch braucht ausreichend RAM (mind. 8 GB, besser 16 GB) und Speicherplatz. Fuer grosse Cases: Elasticsearch-Cluster mit mehreren Nodes verwenden.

Praktische Anwendung — Befehle und Workflow

Fallbeispiel: Verdacht auf Datendiebstahl

Szenario: Mitarbeiter verlaesst das Unternehmen am 15.06.2024. Es besteht Verdacht, dass vor dem Austritt grosse Mengen an Firmendaten auf einen USB-Stick kopiert wurden. Image des Rechners liegt als suspect.E01 vor.

# Schritt 1: Komplette Super-Timeline extrahieren
log2timeline.py --status_view window \
  --hashers md5,sha256 \
  --storage_file /cases/data-theft/timeline.plaso \
  /cases/data-theft/suspect.E01

# Schritt 2: Auf den relevanten Zeitraum filtern (01.06.–15.06.2024)
psort.py -o l2tcsv \
  -q "date >= '2024-06-01 00:00:00' AND date <= '2024-06-15 23:59:59'" \
  /cases/data-theft/timeline.plaso \
  -w /cases/data-theft/timeline_juni.csv

# Schritt 3: Nur Datei-Events extrahieren, die auf USB hindeuten
psort.py -o l2tcsv \
  -q "source_short == 'FILE' AND message CONTAINS 'USB'" \
  /cases/data-theft/timeline.plaso \
  -w /cases/data-theft/usb_events.csv

# Schritt 4: Prefetch analysieren — welche Programme liefen?
psort.py -o l2tcsv \
  -q "parser == 'prefetch'" \
  /cases/data-theft/timeline.plaso \
  -w /cases/data-theft/prefetch.csv

# Schritt 5: LNK-Dateien auf USB-Bezug
psort.py -o l2tcsv \
  -q "parser == 'lnk'" \
  /cases/data-theft/timeline.plaso \
  -w /cases/data-theft/lnk.csv

Analyse in Timeline Explorer

Die exportierten CSV-Dateien werden in Timeline Explorer (Eric Zimmerman) geladen. Sortiere nach datetime und markiere auffaellige Muster:

  1. Grosse Dateimengen kurz hintereinander kopiert?
  2. USB-Laufwerksbuchstabe (D:\, E:\) in LNK-Dateien oder Registry?
  3. Programm robocopy.exe oder 7z.exe in Prefetch? (typische Tools fuer Massenkopie)
  4. Zeitgleiche Registry-Eintraege in USBSTOR?

Zeitzonen-Hoelle: Der haeufigste Fehler bei Timeline-Analysen sind inkonsistente Zeitzonen. NTFS speichert alle Timestamps in UTC, aber Plaso kann sie in lokale Zeit konvertieren. Immer dokumentieren, welche Zeitzone verwendet wurde und ob UTC oder lokal. Der Parameter --timezone in Plaso ist dein Freund.
