Kurzüberblick

Die Datenträger-Forensik analysiert die Struktur von Festplatten, SSDs und USB-Sticks unterhalb der Dateiebene. Partitionstabellen, Bootsektoren, Cluster Chains, Slack Space — hier findest du Daten, die kein Dateibrowser zeigt. Die Analyse beginnt immer auf der tiefsten Ebene: dem physischen Sektor.

Grundprinzip: Ein Dateisystem ist eine Datenbank auf Rohdaten. Wer nur den Datei-Explorer benutzt, sieht nur, was die Datenbank aktuell referenziert — nicht das, was tatsächlich auf der Platte liegt.

Partitionstabellen im Detail

MBR (Master Boot Record) — Sektor 0

Der MBR belegt genau 512 Bytes im ersten Sektor (LBA 0) und ist in drei Bereiche aufgeteilt:

OffsetGrößeInhalt
0x000 – 0x1BD446 BytesBootstrap-Code (Bootloader Stage 1)
0x1BE – 0x1FD64 BytesPartitionstabelle (4 Einträge à 16 Bytes)
0x1FE – 0x1FF2 BytesBoot-Signatur 0x55AA

Jeder der 4 Partitionseinträge (16 Bytes) enthält:

  • Byte 0: Boot-Flag (0x80 = aktiv/bootable, 0x00 = inaktiv)
  • Byte 1–3: CHS-Adresse des ersten Sektors
  • Byte 4: Partitionstyp (0x07 = NTFS, 0x83 = Linux, 0x82 = Swap, 0x05 = erweiterte Partition)
  • Byte 5–7: CHS-Adresse des letzten Sektors
  • Byte 8–11: LBA des ersten Sektors (Startsektor)
  • Byte 12–15: Anzahl der Sektoren (Größe)
Forensische Einschränkung MBR: Maximal 2 TB adressierbar (512-Byte-Sektoren × 2³² Sektoren). Größere Platten benötigen zwingend GPT. Gelöschte Partition: Partitionstyp auf 0x00 gesetzt, Startsektor und Größe bleiben meist erhalten → wiederherstellbar.

GPT (GUID Partition Table) — UEFI-Standard

GPT ist der moderne Ersatz für MBR und Teil des UEFI-Standards. Die Struktur verteilt sich über mehrere LBAs:

LBAInhaltDetails
0Protective MBREin MBR-kompatibler Eintrag, der die ganze Platte als eine Partition vom Typ 0xEE markiert — schützt vor älteren Tools, die GPT nicht verstehen
1GPT-HeaderEnthält Signatur ("EFI PART"), Revision, Header-Größe, CRC32, Anzahl der Partitionseinträge, Größe pro Eintrag
2–33Partitionseinträge128 Einträge à 128 Bytes (standardmäßig, konfigurierbar). Jeder Eintrag: Typ-GUID, Unique-GUID, Start/End LBA, Attribute, Name (36 UTF-16 Zeichen)
letzte 34BackupLetzte 33 Sektoren enthalten Backup-Header und Backup-Einträge — Redundanz am Plattenende
Forensisch relevant: GPT speichert Header UND Partitionseinträge doppelt (Anfang + Ende der Platte). Wurde die primäre GPT zerstört, kann das Backup am Plattenende rekonstruiert werden. Tools: gdisk kann Backup wiederherstellen.

TSK (Sleuth Kit) Workflow

Der forensische Analyse-Workflow mit TSK folgt einer festen Reihenfolge — von grob nach fein:

1. mmstat — Welcher Partitionstyp?

mmstat image.dd
# Ausgabe: dos  → MBR-Partitionstabelle
# Ausgabe: gpt  → GPT-Partitionstabelle
# Ausgabe: mac  → Apple Partition Map

Bestimmt den Typ der Partitionstabelle, ohne das Image zu verändern.

2. mmls — Partitionen auflisten

mmls image.dd
# Ausgabe:
# Slot  Start    End      Length   Description
# 00:   Meta     0000000000  0000000000  0000000001  Primary Table (#0)
# 01:   -----    0000000000  0000002047  0000002048  Unallocated
# 02:   00:00    0000002048  0976560127  0976558080  NTFS / exFAT (0x07)
# 03:   00:01    0976560128  1953523711  0976963584  Linux (0x83)

Der entscheidende Wert ist Start — der Offset in Sektoren, ab dem die Partition beginnt (hier: 2048).

3. fsstat — Dateisystem-Details

fsstat -o 2048 image.dd
# Ausgabe u.a.:
# File System Type: NTFS
# Cluster Size: 4096
# Total Clusters: 122069760
# MFT Start: 786432

Liefert Cluster-Größe, MFT-Position, Journal-Status, letzter Check — die Basis für alle weiteren Analysen.

4. fls — Dateien auflisten (auch gelöschte)

# Alle Dateien rekursiv
fls -r -o 2048 image.dd

# Nur gelöschte Dateien
fls -d -r -o 2048 image.dd

# Zeitstempel (m = modified, a = accessed, c = changed)
fls -m "/" -r -o 2048 image.dd > bodyfile

Gelöschte Dateien erkennt man am * (Stern) vor dem Dateinamen oder an der (deleted)-Markierung.

5. icat — Datei-Inhalt extrahieren

# Datei per Inode-Nummer extrahieren (Inode aus fls-Ausgabe)
icat -o 2048 image.dd 42 > wiederhergestellt.txt

# Gelöschte Datei per MFT-Eintrag (NTFS)
icat -o 2048 image.dd 256-128-3 > geloescht.docx

6. istat — Metadaten einer Inode / eines MFT-Eintrags

istat -o 2048 image.dd 42
# Zeigt: Größe, Zeitstempel, Cluster-Runlist, Attribute, Berechtigungen

LVM und RAID Forensik

LVM (Logical Volume Manager) — Linux

LVM abstrahiert physische Partitionen in einer Schichtenarchitektur:

  1. PV (Physical Volume): Physische Partition, markiert mit LVM-Signatur
  2. VG (Volume Group): Fasst mehrere PVs zu einem Pool zusammen
  3. LV (Logical Volume): Wird aus dem VG-Pool alloziert — das eigentliche "Dateisystem-Gerät"
Herausforderung: TSK kann LVM nicht direkt lesen. Du musst LV aus dem Image assemblieren: losetup + vgscan + vgchange -ay oder spezielle Tools wie lvm2 im Read-Only-Modus.
# LVM-Scan auf Image
losetup -f --show -r image.dd    # → /dev/loop0
pvscan
vgscan
vgchange -ay --ignorelockingfailure
lvscan                           # → /dev/vg0/root
# Dann TSK auf /dev/vg0/root anwenden

RAID — Redundant Array of Independent Disks

RAID-LevelStripingForensische Rekonstruktion
RAID 0Blockweise, kein MirrorReihenfolge + Stripe-Größe ermitteln (oft 64 KB oder 128 KB), dann mit mdadm --build assemblieren
RAID 1Kein Striping, MirrorBeide Platten identisch — eine reicht. Achtung: unterschiedliche Versionen möglich
RAID 5Striping + ParitätEine Platte darf fehlen. Stripe-Größe + Rotation (left/right sync/async) ermitteln. mdadm --build
RAID 10Mirror + StripeKombination aus RAID 1 und 0 — zuerst Spiegel, dann Stripe
Kritisch: Falsche Stripe-Größe oder falsche Platten-Reihenfolge führt zu komplett unbrauchbaren Ergebnissen. Vor der Rekonstruktion: Superblock jeder Platte mit mdadm --examine analysieren. Dokumentieren, welche Platte an welchem Port hing.

Slack Space — Daten in der Lücke

File Slack

Dateisysteme allozieren Speicher in Clustern, nicht bytegenau. Eine 1-Byte-Datei auf NTFS mit 4 KB Clustern belegt einen ganzen Cluster (4096 Bytes). Die 4095 ungenutzten Bytes sind File Slack.

Berechnung: File Slack = Cluster-Größe − (Dateigröße mod Cluster-Größe)
Beispiel: Datei 1500 Bytes, Cluster 4096 Bytes → Slack = 4096 − 1500 = 2596 Bytes

RAM Slack

File Slack wird weiter unterteilt:

  • RAM Slack: Der Bereich vom Dateiende bis zum Ende des letzten von der Datei belegten Sektors (512 Bytes). Wird mit Daten aus dem RAM-Puffer des Betriebssystems aufgefüllt.
  • Drive Slack: Die restlichen Sektoren des Clusters, die nie von der Datei beschrieben wurden. Enthält Daten der Vorgängerdatei.
BegriffBereichInhaltForensischer Wert
RAM SlackDateiende bis SektorendeRAM-Schnipsel, PufferresteKann Passwörter, Keys, Klartexte enthalten
Drive SlackSektorende bis ClusterendeDaten der VorgängerdateiGelöschte Dokumente, Fragmente alter Dateien
# Slack Space mit blkls extrahieren
blkls -s image.dd > slackspace.bin

# Dann carving auf der slackspace.bin
photorec slackspace.bin

Orphaned Files & Unallocated Space

Unallocated Space

Der Bereich des Datenträgers, der laut Dateisystem keiner aktiven Datei zugeordnet ist:

  • Nicht partitionierter Bereich: Zwischen Partitionen oder vor der ersten / nach der letzten Partition
  • Nicht allozierte Cluster: Innerhalb einer Partition — Cluster, die nicht von $Bitmap (NTFS) oder der Block-Bitmap (ext4) als belegt markiert sind
  • Enthält: Gelöschte Dateien, temporäre Dateien, alte Partitionen, Druck-Spooler-Daten
# Unallocated Space extrahieren (TSK)
blkls -o 2048 image.dd > unallocated.bin

# Nur den unallocierten Bereich (nicht allozierte Cluster)
blkls -A -o 2048 image.dd > unalloc_only.bin

Orphaned Files

Orphaned Files sind Dateien, deren Inhalt noch auf der Platte existiert, deren Metadaten (MFT-Eintrag, Inode) aber gelöscht wurden. Sie sind nur noch durch File Carving auffindbar:

  • Die Clusters der Datei sind im $Bitmap als "frei" markiert
  • Die Clusters wurden noch nicht überschrieben
  • Carving-Tools durchsuchen den Rohdatenstrom nach Magic Bytes (Header/Footer)
MethodeToolBeschreibung
Header/Footer-CarvingforemostExtrahiert Dateien anhand bekannter Magic Bytes (FFD8 für JPEG, 25 50 44 46 für PDF)
File-Structure-CarvingscalpelWie foremost, aber mit konfigurierbarer Datenbank
Semantisches Carvingbulk_extractorExtrahiert semantische Daten (E-Mails, URLs, Kreditkartennummern) unabhängig vom Dateiformat
Fragment-CarvingPhotoRecRekonstruiert fragmentierte Dateien durch Validierung interner Strukturen

Praktische Anwendung: Vollständiger TSK-Workflow

Ein forensisches Image asservat01.dd wurde erstellt und per SHA-256 verifiziert. Die Analyse läuft wie folgt:

# Schritt 1: Partitionstyp erkennen
mmstat asservat01.dd
# → dos

# Schritt 2: Partitionen auflisten
mmls asservat01.dd
# Slot  Start    End      Length   Description
# 02:   00:00    0000002048  0976560127  0976558080  NTFS (0x07)
# 03:   00:01    0976560128  1953523711  0976963584  Linux (0x83)

# Schritt 3: Dateisystem-Info (NTFS-Partition bei Offset 2048)
fsstat -o 2048 asservat01.dd
# Cluster Size: 4096, MFT Start: 786432

# Schritt 4: Dateiliste erstellen (inkl. Zeitstempel)
fls -m "/" -r -o 2048 asservat01.dd > bodyfile_ntfs.txt

# Schritt 5: Timeline generieren
mactime -b bodyfile_ntfs.txt -z UTC 2025-03-15 > timeline.csv

# Schritt 6: Gelöschte Dateien identifizieren
fls -d -r -o 2048 asservat01.dd > geloescht.txt

# Schritt 7: Verdächtige Dateien extrahieren
# (Angenommen Inode 55555 ist eine verdächtige .docx)
icat -o 2048 asservat01.dd 55555 > verdacht.docx

# Schritt 8: Unallocated Space extrahieren für Carving
blkls -o 2048 asservat01.dd > unalloc.bin
foremost -t all -o carved_output/ unalloc.bin

# Schritt 9: Zweite Partition (ext4 bei Offset 976560128)
fsstat -o 976560128 asservat01.dd
fls -m "/" -r -o 976560128 asservat01.dd >> bodyfile_linux.txt
Praxis-Tipp: Immer zuerst fsstat aufrufen, um die Cluster-Größe zu erfahren. Viele TSK-Befehle verwenden intern Clusternummern — wenn du die falsche Cluster-Größe annimmst, kalkulierst du falsche Offsets.
