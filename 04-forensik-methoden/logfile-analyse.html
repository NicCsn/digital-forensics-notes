Kurzüberblick

Logs sind das Gedächtnis eines Systems — sie protokollieren Ereignisse mit Zeitstempel. In der Forensik sind sie oft der rote Faden, der eine Angriffskette rekonstruiert: von der initialen Infektion über laterale Bewegung bis zur Exfiltration. Ein einzelner Log-Eintrag mag unbedeutend erscheinen — die Korrelation mehrerer Log-Quellen über die Zeitachse ist das, was Fälle löst.

Grundsatz: Ein System ohne Logs ist ein Tatort ohne Zeugen. Je mehr Log-Quellen du hast, desto schwerer ist es für einen Angreifer, alle Spuren zu verwischen.

Zeitnormalisierung

Zeit ist das fundamentale Problem der Log-Analyse. Zwei Log-Einträge von verschiedenen Systemen lassen sich nur dann korrelieren, wenn ihre Zeitstempel auf dieselbe Zeitzone normalisiert sind.

Zeitzonenfallen

QuelleTypisches ZeitformatZeitzonen-Problem
Linux syslogMar 15 03:17:22Keine Zeitzone im String (nur System-Locale)
Apache access.log[15/Mar/2025:03:17:22 +0200]Hat Zeitzone, Apache-eigener Format-String
Windows EventLogIntern UTC, Anzeige lokalEventLog speichert UTC, wevtutil zeigt lokal
Syslog (RFC 5424)2025-03-15T03:17:22ZISO 8601 mit UTC-Marker — ideal!
Firewall (pfSense)Mar 15 03:17:22 PDTZeitzonen-Kürzel mehrdeutig (PDT ≠ PST)
Sommerzeit-Falle: Ein Angriff findet am Tag der Zeitumstellung statt (z.B. 3:15 MESZ). Ein Server loggt in UTC (1:15), der andere in Lokalzeit (3:15). Ohne Normalisierung: 2 Stunden Differenz = inkorrekte Timeline = Angriffskette falsch rekonstruiert.

Normalisierungsstrategie

  1. Alle Zeitstempel nach UTC konvertieren. Keine Ausnahmen. UTC hat keine Sommerzeit, keinen TZ-Wechsel.
  2. Original-Zeitstempel behalten. Den UTC-konvertierten Wert zusätzlich speichern, nicht ersetzen.
  3. Zeitzone jeder Quelle dokumentieren: Welche TZ hatte das System? War Sommerzeit aktiv? NTP-Synchronisation?
# Python: Normalisierung verschiedener Formate nach UTC
import dateutil.parser
import pytz

def normalize_to_utc(timestr, source_tz='UTC'):
    dt = dateutil.parser.parse(timestr)
    if dt.tzinfo is None:
        dt = pytz.timezone(source_tz).localize(dt)
    return dt.astimezone(pytz.UTC).isoformat()

print(normalize_to_utc('Mar 15 03:17:22', 'Europe/Berlin'))
# → 2025-03-15T01:17:22+00:00 (CET = UTC+1)

print(normalize_to_utc('15/Mar/2025:03:17:22 +0200'))
# → 2025-03-15T01:17:22+00:00 (+0200 = CEST)

Linux Auth Log Muster

Erfolgreiche SSH-Anmeldung

# Mit Passwort:
sshd[1234]: Accepted password for root from 10.0.0.42 port 54321 ssh2

# Mit Public-Key:
sshd[1234]: Accepted publickey for ubuntu from 10.0.0.42 port 54321 ssh2: RSA SHA256:abc123...

# Session-Start:
sshd[1234]: Starting session: shell on pts/0 for root

# Session-Ende:
sshd[1234]: Received disconnect from 10.0.0.42 port 54321:11: disconnected by user

Fehlgeschlagene SSH-Anmeldung (Brute-Force)

sshd[1234]: Failed password for root from 203.0.113.42 port 54321 ssh2
sshd[1234]: Failed password for root from 203.0.113.42 port 54322 ssh2
sshd[1234]: Failed password for root from 203.0.113.42 port 54323 ssh2
#              ... hunderte Einträge ...
sshd[1234]: Accepted password for root from 203.0.113.42 port 54399 ssh2
#     ^^^ ERFOLGREICH! Der Angreifer hat das Passwort erraten.

sudo:   frank : TTY=pts/0 ; PWD=/home/frank ; USER=root ; COMMAND=/usr/bin/apt install nmap

Log-Grep-Patterns für schnelle Triage

Top-Brute-Force-IPs (auth.log)

grep "Failed password" auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head -20

Top-404-URLs (access.log) — Reconnaissance

grep " 404 " access.log | awk '{print $7}' | sort | uniq -c | sort -rn | head -20

SQL-Injection-Versuche

grep -iE "(UNION.*SELECT|SELECT.*FROM|INSERT.*INTO|'.*OR.*'='|'.*--)" access.log

XSS-Versuche

grep -iE "(<script|javascript:|onerror=|onload=)" access.log

Path-Traversal

grep -E "\.\./\.\." access.log

Erfolgreiche Logins nach Fehlversuchen (evtl. erfolgreicher Brute-Force)

# Alle IPs, die erst Failed und dann Accepted hatten:
grep "sshd" auth.log | grep -E "(Failed password|Accepted)" | \
  awk '{print $(NF-3)}' | uniq -c | awk '$1 > 5'

Zeitfenster-Isolation

# Alle Events zwischen 03:00 und 04:00 Uhr (am 15. März)
grep "Mar 15 03:" auth.log

# systemd-journal (Linux)
journalctl --since "2025-03-15 03:00:00" --until "2025-03-15 04:00:00"

Apache: Top-IPs nach Request-Volumen (DDoS/Scraping)

awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -20

Praktische Anwendung

Tool: lnav (Log Navigator)

# Interaktiver Log-Viewer mit automatischer Format-Erkennung
lnav /var/log/apache2/access.log /var/log/auth.log
# Syntax-Highlighting, Timeline-Ansicht, SQL-Abfragen auf Live-Logs

Tool: LogParser (Windows)

# SQL-artige Abfragen auf EventLogs und CSV-Dateien
LogParser.exe "SELECT TimeGenerated, EventID, Message
  INTO report.csv
  FROM Security.evtx
  WHERE EventID IN (4624; 4625)
  ORDER BY TimeGenerated DESC"

Tool: chainsaw (Sigma-Rules auf EventLogs)

# EventLog-Datei mit Sigma-Regeln scannen
chainsaw hunt /mnt/evidence/Windows/System32/winevt/Logs/ -s sigma/rules/ --mapping mappings/sigma-event-logs-all.yml

Tool: Hayabusa (EventLog Timeline)

# Windows EventLog Timeline mit Hayabusa generieren
hayabusa.exe csv-timeline -d C:\evidence\Windows\System32\winevt\Logs\ -o timeline.csv

Vollständiger Log-Analyse-Workflow

# 1. Zeitzone des loggenden Systems ermitteln
cat /etc/timezone                    # Linux
tzutil /g                            # Windows
# → Dokumentiere: Welche TZ war zur Tatzeit aktiv?

# 2. Zeitfenster eingrenzen
# Tatzeit: 15.03.2025 01:00 - 04:00 UTC
grep "Mar 15 0[1-4]:" auth.log > auth_tatzeit.txt

# 3. Brute-Force-Check
grep "Failed password" auth_tatzeit.txt | \
  awk '{print $(NF-3)}' | sort | uniq -c | sort -rn

# 4. Erfolgreiche Logins im Zeitfenster
grep "Accepted" auth_tatzeit.txt

# 5. Sudo/Su-Nutzung nach Login
grep "sudo:" auth_tatzeit.txt
grep "su:" auth_tatzeit.txt

# 6. Web-Logs auf Angriffe scannen
grep -E " 50[0-9] " access.log | awk '{print $1, $4, $7}'

# 7. Korrelation: Firewall-Log + Server-Log
# FW: "ACCEPT src=10.0.0.42 dst=10.0.0.100 dport=443"
# Web: "10.0.0.42 ... POST /upload.php ... 200"
# → Daten wurden erfolgreich exfiltriert
Praxis-Tipp: Verwende immer sort | uniq -c | sort -rn als erste Triage. Die größten Ausreißer (die am häufigsten vorkommenden IPs, URLs, User-Agents) zeigen dir sofort, wo du tiefer graben musst. 90 % der Log-Analyse ist Zählen und Sortieren.
Log-Integrität: Logs können manipuliert sein. Achte auf: Lücken in der Timeline (fehlende Stunden/Tage), inkonsistente PID-Sequenzen, Log-Rotation mitten im Angriffszeitraum, fehlende session closed-Einträge nach session opened. All das sind Hinweise auf Log-Wiping.
