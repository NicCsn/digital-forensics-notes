Kurzüberblick

Forensisches Imaging ist der erste Schritt jeder Untersuchung: eine bit-genaue Kopie des Original-Datenträgers. Jeder Fehler hier kann die gesamte Untersuchung gefährden. Das forensische Image ist der Goldstandard — jede Abweichung muss explizit dokumentiert und begründet werden (ACPO Prinzip 2).

Goldene Regel: Niemals am Original arbeiten. Das Original-Image wird einmal erstellt und dann versiegelt. Alle Analysen laufen auf Kopien der Kopie.

Image-Formate im Detail

RAW / DD-Format

EigenschaftBeschreibung
StrukturBit-für-Bit-Kopie, keine Metadaten, keine Kompression. Das Image enthält exakt jeden Sektor in linearer Reihenfolge.
Endungen.dd, .raw, .img, .bin, .001
VorteileUniversell von allen Tools lesbar, kein Overhead, direkt als Loop-Device mountbar
NachteileKeine eingebetteten Metadaten (Fallnummer, Prüfer), keine Integritätsprüfung, keine Kompression
GrößeExakt gleich der Quell-Disk — 1 TB Disk = 1 TB Image

E01 / Expert Witness Format (EWF)

Von EnCase (Guidance Software) entwickelt, heute De-facto-Standard in der Strafverfolgung:

EigenschaftBeschreibung
StrukturHeader + Daten-Chunks + CRC-Prüfsumme pro Chunk. Jeder Chunk kann komprimiert sein.
Endungen.E01, .E02, .E03 … (gesplittet nach Größe, z.B. 2 GB pro File)
MetadatenFallnummer, Ermittler, Gerätemodell, Seriennummer, Hash (MD5 + SHA-1 im Header), Zeitstempel der Erstellung, Erfassungs-Software
KompressionDeflate (zlib) — typisch 30–50 % Platzersparnis, abhängig vom Datentyp
VerschlüsselungAES-256 möglich (Passwort-basiert), dann Endung .Ex01 statt .E01
IntegritätCRC pro Chunk + MD5/SHA-1 des Gesamt-Images im Header. Jeder Chunk-Fehler wird sofort erkannt.
E01-Chunk-Struktur: Ein E01-Image besteht aus: Header-Sektion (Case-Info, Hash) → Volume-Sektion → Table-Sektion → Chunk 1 (64 Sektoren) → Chunk 2 → … → letzter Chunk. Jeder Chunk hat einen eigenen CRC.

AFF4 (Advanced Forensic Format 4)

  • Nachfolger von AFF, entwickelt mit Fokus auf Cloud-Forensik und Big Data
  • Stream-basierte Architektur: Ein Image kann aus vielen Streams bestehen (z.B. Partition 1, Partition 2, MFT-Extrakt, Registry-Extrakt)
  • Unterstützt Dedupulierung, Verschlüsselung (AES), Signierung (RSA/ECDSA)
  • Container-Format: ZIP-basiert (.aff4 = .zip mit spezifischen Streams)
  • Tool-Support: pmem, Velociraptor, rekall

SMART (ASR Data)

  • Proprietäres Format für sehr große Datensätze (Firmen-RAIDs, Server)
  • Unterstützt paralleles Imaging (mehrere Worker schreiben gleichzeitig)
  • Integrierte Remote-Fähigkeit (Image über Netzwerk streamen)
  • Selten außerhalb von Enterprise-DFIR

Hash-Verifikation

MD5 ist tot — fast. MD5 hat bekannte Kollisionen (zwei verschiedene Inputs ergeben denselben Hash) und ist für forensische Zwecke nicht mehr als alleiniger Hash ausreichend. ABER: MD5 reicht als Filter (bekannte gute/böse Dateien), weil die Angriffe gezielt und nicht zufällig sind.
AlgorithmusBitsEinsatz in der ForensikStatus
MD5128NSRL-Datenbank, Hash-Set-Filtering, bekanntes-Malware-MatchingAls Filter OK, nicht für Integrität
SHA-1160E01-Header (Legacy), Git-CommitsÜbergangsweise, SHAttered-Angriff existiert
SHA-256256Standard für forensische Integrität. Gerichtsverwertbar.Empfohlen, kein praktischer Angriff
SHA-512512Für hochsensible Fälle, zusätzliche SicherheitsmargeZukunftssicher
# Hash des Originals (VOR dem Imaging, durch Write-Blocker)
sha256sum /dev/sdb > original.sha256

# Imaging durchführen (dd, dcfldd, guymager...)

# Hash des Images
sha256sum image.dd > image.sha256

# Vergleich (muss identisch sein!)
diff original.sha256 image.sha256 && echo "Hashes stimmen überein"

# Alternative: dcfldd hasht während des Kopierens
dcfldd if=/dev/sdb hash=sha256 hashlog=hash.log of=image.dd
Achtung: Der Hash des Originals MUSS durch den Write-Blocker hindurch berechnet werden. Direktes Anschließen des Asservats ohne Write-Blocker verändert den Hash (Mount-Zeitstempel, Journal-Updates).

Guymager GUI Workflow

Guymager ist das grafische Imaging-Tool für Linux, speziell für forensische Workflows entwickelt:

  1. Guymager starten: sudo guymager (benötigt Root-Rechte für Block-Devices)
  2. Device auswählen: Links erscheint die Liste aller erkannten Geräte (Modell, Größe, Seriennummer)
  3. Rechtsklick → "Acquire Image": Öffnet den Konfigurationsdialog
  4. Format wählen: DD (Raw) oder E01 (Expert Witness)
  5. Fall-Metadaten eingeben: Case Number, Evidence Number, Examiner Name, Description, Notes
  6. Zielpfad setzen: Wohin wird das Image geschrieben? (Muss genügend freien Speicher haben)
  7. Hash-Algorithmen wählen: SHA-256 (empfohlen) + optional SHA-1 / MD5
  8. Split-Größe: Bei E01 z.B. 2 GB (zwei .E01/.E02/.E03…) oder 0 für ein File
  9. Start: Guymager zeigt Fortschrittsbalken, Geschwindigkeit (MB/s), geschätzte Restzeit
  10. Nach Abschluss: Verifikation läuft automatisch (Image wird zurückgelesen und Hash mit Original verglichen)
  11. Info-Datei prüfen: Guymager schreibt eine .info-Datei mit allen Metadaten und Hash-Werten
Guymager-Feature: Im Gegensatz zu dd hasht Guymager während des Kopierens und verifiziert nach dem Kopieren automatisch. Der Verifikations-Pass liest das Image zurück und vergleicht — Fehler werden sofort gemeldet.

Logical vs. Physical Imaging

MerkmalPhysical ImagingLogical Imaging
Was wird kopiert?Jeder Sektor (inkl. Bootsektor, Partitionstabelle, Slack Space, unallociert)Nur aktive Dateien innerhalb einer Partition
VolständigkeitVollständig — alle Daten, auch gelöschteUnvollständig — nur das, was der Datei-Explorer sieht
EinsatzStrafverfolgung, GerichtsverwertungSchnelle Triage, eDiscovery, Unternehmens-DFIR
GeschwindigkeitLangsamer (ganze Disk)Schneller (nur belegte Dateien)
GrößeExakt Disk-GrößeNur belegter Speicher (oft 30-70 % der Disk-Größe)
Gelöschte DateienEnthalten (in unallocierten Bereichen)Nicht enthalten
Wann Logical Imaging? Nur wenn Zeit oder Speicherplatz extrem knapp ist und gelöschte Daten nicht relevant sind — z.B. eDiscovery mit Fokus auf aktive Dokumente. Für kriminalistische Ermittlungen immer Physical Imaging.

SSD-spezifische Probleme

TRIM / UNMAP

TRIM ist ein ATA-Befehl, mit dem das OS der SSD mitteilt, welche Blöcke nicht mehr benötigt werden. Die SSD löscht diese Blöcke dann im Hintergrund — die Daten sind unwiederbringlich verloren.

Wear Leveling

SSDs verteilen Schreibzugriffe gleichmäßig über alle Flash-Zellen (Wear Leveling). Das bedeutet:

  • Logische Blockadresse (LBA) ≠ physischer Flash-Speicherort
  • Der Flash Translation Layer (FTL) übersetzt Adressen dynamisch
  • Alte Versionen eines logischen Blocks können in anderen physischen Zellen noch existieren
  • Aber: Diese "Overprovisioning"-Bereiche sind für forensische Tools nicht direkt zugänglich

Garbage Collection

Der SSD-Controller löscht im Leerlauf Blöcke, die per TRIM markiert oder nicht mehr referenziert sind. Dies geschieht ohne Zutun des Betriebssystems — einfach durch das Anschließen der SSD an Strom.

SSD-Falle: Eine SSD, die als Beweismittel sichergestellt wird und nur 30 Minuten an Strom hängt (auch ohne Datenzugriff!), kann bereits Garbage Collection ausführen und Beweisdaten vernichten. Maßnahmen: SSD sofort vom Strom trennen oder spezielle Forensik-Bridges mit ATA-Pass-Through verwenden, die TRIM blockieren.

Image-Splitting, Kompression, Verschlüsselung (E01)

Image-Splitting

E01-Images können automatisch in Segmente fester Größe aufgeteilt werden:

# ftkimager (Windows)
ftkimager \\.\PhysicalDrive1 C:\evidence --e01 --frag 2GB

# guymager: Split-Größe im GUI einstellen (default: 2GB)
# libewf (Linux):
ewfacquire -t evidence -S 2G /dev/sdb

Sinnvoll für: FAT32-Ziele (max. 4 GB pro Datei), Netzwerkübertragung, Backup-Tapes.

Kompression (Deflate)

StufeGeschwindigkeitPlatzersparnisEinsatz
Keine (0 / fast)MaximalKeineBereits komprimierte Daten (ZIP, JPEG, Video)
Mittel (1–5)Gut~30 %Standard-Workstation
Maximal (6–9 / best)Langsam (CPU-gebunden)~50 %Textlastige Datenträger, Langzeitarchivierung
Tipp: Kompression ist doppelt sinnvoll: Sie spart Platz UND beschleunigt das Imaging, weil weniger Daten auf das Ziel-Laufwerk geschrieben werden. Moderne CPUs komprimieren schneller, als die meisten USB-3-Ziellaufwerke schreiben können.

Verschlüsselung (AES-256)

# E01 verschlüsselt erstellen (Endung .Ex01)
ewfacquire -t evidence -E password123 /dev/sdb

# Verschlüsseltes Image öffnen
ewfmount -p password123 evidence.Ex01 /mnt/ewf/

Einsatz: Transport von Images über unsichere Kanäle, Speicherung in Cloud-Umgebungen, Compliance-Vorgaben.

Praktische Anwendung: Vollständiger Imaging-Workflow

Kompletter Ablauf von der Asservatannahme bis zum versiegelten Image:

# ===== 1. ASSERVAT IDENTIFIZIEREN =====
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT,MODEL,SERIAL
fdisk -l /dev/sdb

# ===== 2. WRITE-BLOCKER PRÜFEN =====
blockdev --getro /dev/sdb          # muss 1 sein
dd if=/dev/zero of=/dev/sdb bs=512 count=1 2>&1 | grep "Operation not permitted"

# ===== 3. HASH DES ORIGINALS =====
sha256sum /dev/sdb | tee original_sha256.txt

# ===== 4. IMAGING =====
# Variante A: dcfldd
dcfldd if=/dev/sdb hash=sha256 hashlog=hash_dcfldd.log \
       of=/mnt/evidence/Fall001_Asservat01.dd bs=4M status=on

# Variante B: ddrescue (bei fehlerhaften Medien)
ddrescue -d -r3 /dev/sdb /mnt/evidence/Fall001_Asservat01.dd \
         /mnt/evidence/Fall001_Asservat01.map

# Variante C: ewfacquire (E01 mit Metadaten)
ewfacquire -c best -S 2G -t Fall001 \
           -m removable -M physical \
           -E "Name des Ermittlers" \
           -N "Notizen: Asservat sichergestellt am 15.03.2025" \
           /dev/sdb

# ===== 5. HASH-VERIFIKATION =====
sha256sum /mnt/evidence/Fall001_Asservat01.dd | tee image_sha256.txt
diff original_sha256.txt image_sha256.txt && echo "=== VERIFIKATION BESTANDEN ==="

# Bei E01:
ewfverify Fall001.E01

# ===== 6. IMAGE SICHERN (ZWEITE KOPIE) =====
cp /mnt/evidence/Fall001_Asservat01.dd /mnt/backup/Fall001_Asservat01.dd
sha256sum /mnt/backup/Fall001_Asservat01.dd | tee backup_sha256.txt
diff image_sha256.txt backup_sha256.txt && echo "=== BACKUP VERIFIZIERT ==="

# ===== 7. ORIGINAL VERSIEGELN =====
# Asservat in antistatische Tüte, Siegel aufkleben,
# Siegelnummer im Chain-of-Custody-Formular vermerken
Dokumentations-Checkliste: 1. Device-Identifikation (Modell, Seriennummer, Größe), 2. Write-Blocker-Modell + Testprotokoll, 3. Hash-Algorithmus + Werte, 4. Imaging-Tool + Version, 5. Start-/Endzeitpunkt, 6. Lesefehler (Anzahl, Position), 7. Zielpfad, 8. Hash-Vergleich bestanden/fehlgeschlagen.
