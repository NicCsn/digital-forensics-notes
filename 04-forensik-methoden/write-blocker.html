Kurzüberblick

Ein Write-Blocker verhindert jeglichen schreibenden Zugriff auf ein Speichermedium. Er ist das entscheidende Werkzeug, um die Integrität des Originalbeweismittels zu wahren — und damit die gerichtliche Verwertbarkeit der gesamten Analyse sicherzustellen.

Niemals ohne! Das Anschließen eines Asservats ohne Write-Blocker ist ein fataler Fehler. Windows mountet automatisch und schreibt System Volume Information. Linux aktualisiert Mount-Zeitstempel und ext4-Journals. Jeder Schreibzugriff macht das Originalbeweismittel angreifbar vor Gericht.

Hardware-Write-Blocker

Wie sie funktionieren

Hardware-Write-Blocker sitzen physisch zwischen Asservat und Analyse-Rechner. Sie operieren auf der ATA/SCSI-Protokollebene und blockieren Schreib-Kommandos, bevor sie das Gerät erreichen:

  1. Analyse-Rechner sendet einen ATA/SCSI-Befehl an den Write-Blocker
  2. Write-Blocker prüft den Opcode: Ist es READ (0x20, 0x08)? Ist es WRITE (0x30, 0x0A)?
  3. READ-Befehl: Wird transparent an das Asservat weitergeleitet, Daten fließen zurück
  4. WRITE-Befehl: Wird blockiert. Der Write-Blocker sendet eine Erfolgsmeldung (ACK) zurück, damit das OS keinen Fehler meldet. Der Schreibbefehl erreicht nie das Gerät.
  5. Einige Modelle blockieren auch TRIM/DISCARD, SECURITY ERASE, SANITIZE — also Kommandos, die Daten zerstören könnten

Vorteile von Hardware-Write-Blockern

  • Betriebssystem-unabhängig: Funktioniert mit Windows, Linux, macOS — der Rechner "sieht" ein normales Laufwerk, weiss aber nicht, dass ein Blocker dazwischen sitzt
  • Praktisch nicht umgehbar: Kein Userspace-Tool und kein Kernel-Modul kann den Blocker überwinden (ausser Firmware-Angriffe)
  • Forensisch anerkannt: Gerichte akzeptieren Hardware-Write-Blocker als verlässlichen Schutz. Software-Write-Blocker werden oft kritischer hinterfragt
  • TRIM/Garbage-Collection-Schutz: Moderne Blocker blockieren auch ATA-TRIM und NVMe-Deallocate — kritisch für SSDs

Software-Write-Blocker

Linux: udev-Regeln für forensischen Schreibschutz

Linux kann Block-Devices zuverlässig auf Read-Only setzen — auf Kernel-Ebene. Die beste Methode ist eine udev-Regel:

# /etc/udev/rules.d/99-forensic-writeblock.rules
# Alle neu angeschlossenen SCSI/USB-Geräte automatisch auf RO setzen

ACTION=="add", SUBSYSTEM=="block", KERNEL=="sd[a-z]*", ATTR{ro}="1"
ACTION=="add", SUBSYSTEM=="block", KERNEL=="sd[a-z]*", RUN+="/usr/bin/logger Forensic: %k auf read-only gesetzt"

Prüfen nach dem Anschließen:

cat /sys/block/sdb/ro        # → 1 (read-only aktiv)
blockdev --getro /dev/sdb    # → 1
blockdev --setro /dev/sdb    # Manuelles setzen falls udev nicht griff
udev allein reicht nicht: Die udev-Regel schützt nur vor nachfolgenden Schreibzugriffen, nicht vor dem Mount-Vorgang selbst. Zusätzlich muss mount -o ro,noexec,noatime verwendet werden. Empfehlung: Forensik-Distribution (CAINE, Paladin, Kali) nutzen, die bereits vollständig vorkonfiguriert ist.

Windows: Registry-Write-Protect

; Registry-Pfad:
; HKLM\SYSTEM\CurrentControlSet\Control\StorageDevicePolicies
; DWORD-Wert: WriteProtect = 1

# Per PowerShell setzen:
New-Item -Path "HKLM:\SYSTEM\CurrentControlSet\Control\StorageDevicePolicies" -Force
New-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\StorageDevicePolicies" `
    -Name "WriteProtect" -Value 1 -PropertyType DWORD -Force
Windows-Software-Write-Blocker unsicher: Der Windows-Registry-Write-Protect funktioniert nur für USB-Massenspeicher und nur auf Usermode-Ebene. Kernel-Treiber, direkte ATA-Kommandos und SATA-Geräte sind nicht geschützt. Für forensische Zwecke unzureichend. Immer Hardware-Write-Blocker bevorzugen.

Write-Blocker-Validierungstest

Jeder Write-Blocker muss vor dem Einsatz getestet werden — und das Ergebnis dokumentiert. Der Testablauf:

# ===== SCHRITT 1: Test-Datei auf Test-Datenträger schreiben =====
# (Ohne Write-Blocker, Test-Datenträger direkt angeschlossen)
echo "forensischer-test" | dd of=/dev/sdc bs=512 count=1

# Hash berechnen
sha256sum /dev/sdc > hash_vorher.txt
# Ausgabe: a1b2c3d4... /dev/sdc

# ===== SCHRITT 2: Write-Blocker dazwischenschalten =====
# Physisch: Write-Blocker zwischen /dev/sdc und Rechner
# Software: blockdev --setro /dev/sdc
# Test-Datenträger aus- und wieder einstecken

# ===== SCHRITT 3: Schreibversuch =====
dd if=/dev/zero of=/dev/sdc bs=512 count=1 2>&1
# Erwartung: "Operation not permitted" oder "Read-only file system"

# ===== SCHRITT 4: Hash erneut berechnen =====
sha256sum /dev/sdc > hash_nachher.txt

# ===== SCHRITT 5: Hash-Vergleich =====
diff hash_vorher.txt hash_nachher.txt
echo $?
# 0 = Hashes identisch → Write-Blocker funktioniert
# ≠ 0 = Hash verändert → WRITE-BLOCKER DEFEKT! NICHT einsetzen!
NIST-CFTT-Verfahren: Das NIST Computer Forensics Tool Testing Program hat einen formalisierten Testprozess für Write-Blocker. Jeder Blocker muss: 1. Alle Schreibkommandos blockieren, 2. Alle Lesekommandos korrekt durchleiten, 3. Keine Daten verändern (selbst nicht schreiben).

Kein Ersatz: Eine 15-Euro-SATA-USB-Bridge ist KEIN Write-Blocker. Sie ist nur ein Adapter. Ohne Write-Blocker verändert jedes moderne OS das Beweismittel beim Anschließen.

NVMe-Falle: NVMe hat ein komplett anderes Kommandoprotokoll als SATA. Ein SATA-Write-Blocker kann NVMe-Geräte nicht schützen, weil er die NVMe-Kommandos gar nicht versteht. Für M.2-NVMe-SSDs brauchst du zwingend einen NVMe-Write-Blocker (500–1000 €).

Praktische Anwendung: Linux-Write-Blocker via udev

# ===== 1. STATISCHE UDEV-REGEL EINRICHTEN =====
sudo tee /etc/udev/rules.d/99-forensic-writeblock.rules << 'EOF'
# Alle Block-Devices automatisch auf Read-Only setzen
# Gilt für alle neu angeschlossenen Geräte (Hotplug)

ACTION=="add", SUBSYSTEM=="block", KERNEL=="sd[a-z]", \
  ATTR{ro}="1", \
  RUN+="/usr/bin/logger 'Forensic Write-Block: %k auf Read-Only'"

# Wenn du nur bestimmte Ports blocken willst:
# ACTION=="add", SUBSYSTEM=="block", KERNEL=="sd[b-z]", ATTR{ro}="1"
EOF

# ===== 2. UDEV NEU LADEN =====
sudo udevadm control --reload-rules
sudo udevadm trigger

# ===== 3. MOUNT-OPTIONEN (für gemountete Images) =====
# /etc/fstab-Eintrag oder manuell:
mount -o ro,noexec,noatime,noauto /dev/sdb1 /mnt/evidence/

# ===== 4. VERIFIKATIONSSKRIPT =====
cat << 'SCRIPT' > ~/verify-writeblock.sh
#!/bin/bash
DEVICE="$1"
if [ -z "$DEVICE" ]; then
  echo "Usage: $0 /dev/sdX"
  exit 1
fi

# Read-Only-Status prüfen
RO=$(cat /sys/block/$(basename $DEVICE)/ro 2>/dev/null)
if [ "$RO" != "1" ]; then
  echo "WARNUNG: $DEVICE ist NICHT read-only! (ro=$RO)"
  echo "Setze manuell: sudo blockdev --setro $DEVICE"
  exit 1
fi

# Schreibtest
dd if=/dev/zero of="$DEVICE" bs=512 count=1 2>&1
if [ $? -ne 0 ]; then
  echo "OK: Schreibtest fehlgeschlagen (write-blocking aktiv)"
else
  echo "FEHLER: Schreibtest erfolgreich - Write-Blocker defekt!"
  exit 1
fi

# Hash berechnen
echo "Berechne SHA-256 von $DEVICE..."
HASH=$(sha256sum "$DEVICE" | cut -d' ' -f1)
echo "SHA-256: $HASH"
echo "Write-Blocker-Test bestanden."
SCRIPT
chmod +x ~/verify-writeblock.sh

Hardware-Write-Blocker-Validierung (ausführlich)

  1. Test-Datenträger vorbereiten: Einen bekannten, unwichtigen Datenträger mit Testdaten beschreiben. Hash berechnen und notieren.
  2. Blocker anschließen: Write-Blocker an den Analyse-Rechner anschließen. Test-Datenträger an den Write-Blocker anschließen. Stromversorgung sicherstellen.
  3. Identifikation prüfen: Wird das Gerät korrekt erkannt? Stimmen Modell, Seriennummer und Kapazität?
  4. Schreibversuch 1: dd if=/dev/zero of=/dev/sdX bs=4096 count=1024 — muss fehlschlagen
  5. Hash 1: sha256sum /dev/sdX und mit dem bekannten Hash vergleichen — muss identisch sein
  6. Schreibversuch 2: Verschiedene Block-Größen testen (512, 4096, 65536, 1048576 Bytes) — alle müssen fehlschlagen
  7. Hash 2: Erneut Hashen und vergleichen — muss identisch sein
  8. Read-Test: Alle Sektoren auslesen: dd if=/dev/sdX of=/dev/null bs=1M — muss ohne Fehler durchlaufen
  9. Hash 3: Nach dem vollständigen Read-Test erneut Hashen — muss immer noch identisch sein
  10. Dokumentation: Alle Testergebnisse mit Zeitstempel, Hash-Werten und Fehlermeldungen protokollieren
Praxis-Tipp: Führe den Validierungstest vor JEDEM neuen Fall durch und bewahre das Testprotokoll auf. Vor Gericht kann die Frage kommen: "Woher wissen Sie, dass der Write-Blocker an DIESEM Tag funktionierte?" — Das Testprotokoll ist die Antwort.
