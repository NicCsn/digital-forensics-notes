Szenario

Ein Forensik-Anfänger hat einen RAM-Dump eines Windows-Systems erhalten. Darin ist eine Flag versteckt. Die Herausforderung ist einfach: Finde die Flag mit grundlegenden Memory-Forensik-Techniken. Es gibt keine komplexe Malware, keine obfuskierten Daten — nur einen sauberen Einstieg in die Welt der Speicheranalyse mit Volatility.

Diese Challenge ist ideal, um die wichtigsten Volatility-Befehle zu erlernen und ein Gefühl für die Struktur von RAM-Dumps zu entwickeln. Ziel ist es, den gesamten Workflow einer Memory-Analyse einmal von Anfang bis Ende durchzugehen: Profilbestimmung, Prozessanalyse, Dateiextraktion und String-Suche.

Perfekt für Einsteiger: MemLabs ist eine der beliebtesten Memory-Forensics-CTF-Reihen auf GitHub. Die Challenges sind didaktisch aufgebaut und decken alle wichtigen Techniken ab — von einfachen Flag-Suchen bis zu komplexer Malware-Analyse.

Walkthrough — Schritt für Schritt

Schritt 1: OS-Profil mit imageinfo / windows.info bestimmen

Jede Memory-Analyse beginnt mit der Bestimmung des Betriebssystem-Profils. Volatility muss wissen, welche Windows-Version und Architektur vorliegt, um die Kernel-Datenstrukturen korrekt zu interpretieren:

# Volatility 3 — System-Informationen
vol -f memdump.raw windows.info

# Ausgabe interpretieren:
# - NT Build: z. B. 7601 → Windows 7 / Server 2008 R2
# - NtMajorVersion / NtMinorVersion: 6.1 = Windows 7
# - SystemRoot: C:\Windows
# - Anzahl der Prozessoren, RAM-Grösse usw.
NT BuildWindows-Version
7600 / 7601Windows 7 / Server 2008 R2
9200Windows 8 / Server 2012
9600Windows 8.1 / Server 2012 R2
10240Windows 10 (1507)
14393Windows 10 1607 / Server 2016
17763Windows 10 1809 / Server 2019

Schritt 2: Prozessliste analysieren (pslist)

Der nächste logische Schritt: Welche Prozesse liefen zum Zeitpunkt der Dump-Erstellung? Wir suchen nach ungewöhnlichen Prozessen, die nicht zum normalen Windows-Start gehören:

# Alle laufenden Prozesse auflisten
vol -f memdump.raw windows.pslist

# Prozessbaum für Parent-Child-Beziehungen
vol -f memdump.raw windows.pstree

# Achten auf:
# - cmd.exe, powershell.exe → Was wurde hier gestartet?
# - notepad.exe, mspaint.exe → Enthalten möglicherweise die Flag!
# - Unbekannte Prozesse mit kryptischen Namen
Worauf achten? Notepad.exe im Speicher = jemand hat eine Textdatei geöffnet. Paint.exe = jemand hat ein Bild bearbeitet. Browser-Prozesse = mögliche Downloads oder gespeicherte Passwörter. All diese Prozesse können Hinweise auf die Flag enthalten.

Schritt 3: Command-Line-Argumente prüfen (cmdline)

Die Command-Line eines Prozesses verrät, wie und mit welchen Argumenten er gestartet wurde. Hier finden sich oft direkte Hinweise auf die Flag oder den Dateipfad:

# Command-Line aller Prozesse
vol -f memdump.raw windows.cmdline

# Oder spezifisch filtern
vol -f memdump.raw windows.cmdline | grep -iE "notepad|cmd|powershell|flag|hint"

# Beispiel-Ausgabe:
# notepad.exe pid: 1234
# CommandLine: "C:\Windows\system32\NOTEPAD.EXE" C:\Users\User\Desktop\important.txt

In MemLabs-1 findet man typischerweise einen cmd.exe-Prozess oder ein notepad.exe, das eine Datei mit einem Hinweis oder der Flag selbst geöffnet hat. Die Command-Line enthält den Dateipfad.

Schritt 4: Filescan und Dumpfiles — Dateien aus dem RAM extrahieren

Volatility kann Dateien, die zum Zeitpunkt des Dumps im Speicher waren (weil sie von einem Prozess verwendet wurden), identifizieren und extrahieren:

# Alle im Speicher referenzierten Dateien auflisten
vol -f memdump.raw windows.filescan

# Nach interessanten Dateien filtern
vol -f memdump.raw windows.filescan | grep -iE "\.txt$|flag|hint|secret|important|desktop"

# Datei anhand der Offset-Adresse dumpen
vol -f memdump.raw -o ./extracted/ windows.dumpfiles --virtaddr 0x1234567890

# Alternativ: Alle .txt- und .png-Dateien dumpen
vol -f memdump.raw windows.filescan | grep -E "\.txt|\.png|\.jpg" > interesting_files.txt
# Dann jede Offset-Adresse manuell dumpen
DateitypWarum relevant?
.txtEnthält möglicherweise die Flag im Klartext.
.png / .jpgFlag könnte in ein Bild eingebettet (Steganografie) oder als Text im Bild dargestellt sein.
.zip / .rarPasswortgeschützte Archive — Flag könnte darin sein.
.exe (unbekannt)Malware? Könnte die Flag als String eingebettet haben.

Schritt 5: Strings-Suche — Die letzte Eskalation

Wenn filescan keine klaren Hinweise liefert, durchsuchen wir den gesamten RAM-Dump nach Klartext-Strings. Dies ist die umfassendste, aber auch rechenintensivste Methode:

# Alle ASCII-Strings mit Mindestlänge 6 extrahieren
strings -n 6 memdump.raw > all_strings.txt

# Nach Flag-Format suchen (typische CTF-Formate)
grep -iE "flag\{|ctf\{|insec\{|htb\{|memlabs\{|FLAG:" all_strings.txt

# Oder mit Volatility 3
vol -f memdump.raw windows.strings | grep -iE "flag\{|ctf\{"

# Auch nach Base64-encodierten Strings suchen
strings memdump.raw | grep -E "^[A-Za-z0-9+/]{20,}={0,2}$" > b64_strings.txt
Flag-Formate in CTFs: Übliche Formate sind flag{...}, CTF{...}, insec{...}, HTB{...} oder einfach FLAG: gefolgt vom eigentlichen Text. Kennt man das Format, kann man gezielt danach suchen.

Schritt 6: Versteckte Daten in Prozess-Memory-Bereichen

Manchmal ist die Flag nicht als Datei, sondern direkt im Speicher eines Prozesses versteckt — z. B. in einem offenen Notepad-Fenster oder in der Kommandozeile eines Terminals:

# Prozess-Speicher dumpen
vol -f memdump.raw -o ./procdump/ windows.memmap --pid 1234 --dump

# Strings auf dem Prozess-Dump
strings ./procdump/pid.1234.dmp | grep -iE "flag|ctf|hint"

# Alternativ: Alle Prozesse dumpen und batchweise nach Flag suchen
for pid in $(vol -f memdump.raw windows.pslist --quiet); do
  vol -f memdump.raw -o ./procdump/ windows.memmap --pid $pid --dump
  strings ./procdump/pid.$pid.dmp | grep -i "flag"
done

Flag und Erklärung

Die Flag in MemLabs-1 findet sich typischerweise auf einem der folgenden Wege:

  1. Variante A — Flag in Command-Line: Ein cmd.exe-Prozess wurde mit einem echo-Befehl gestartet, der die Flag ausgibt. Die Command-Line in windows.cmdline enthält die Flag direkt.
  2. Variante B — Flag in Notepad: Notepad.exe hat eine Textdatei mit der Flag geöffnet. Über windows.memmap den Notepad-Speicher dumpen und mit strings durchsuchen.
  3. Variante C — Flag in Datei: Eine .txt-Datei auf dem Desktop enthält die Flag. Mit windows.filescan finden und mit windows.dumpfiles extrahieren.
  4. Variante D — Flag in Bild: Eine PNG-Datei zeigt die Flag als Text im Bild. Mit filescan finden, dumpen und öffnen.
Erklärung des Lösungswegs: Memory-Forensik funktioniert, weil das Betriebssystem ALLE Daten im RAM hält — auch solche, die nie auf die Festplatte geschrieben wurden. Ein offenes Notepad-Fenster, ein laufender Befehl in cmd.exe, ein angezeigtes Bild — all das existiert als Klartext (oder in dekodierbarer Form) im Arbeitsspeicher. Volatility ermöglicht es, diese flüchtigen Daten zu extrahieren und zu analysieren.

Lessons Learned

  • Der RAM vergisst nie — solange der Strom an ist: Selbst Daten, die nie auf Disk gespeichert wurden, sind im RAM-Dump vollständig vorhanden. Das macht Memory-Forensik so mächtig.
  • Strings sind dein bester Freund: Der Befehl strings ist das Schweizer Taschenmesser der Memory-Forensik. Er findet Klartext-Daten, die in keinem strukturierten Tool auftauchen.
  • Command-Lines sind Gold wert: windows.cmdline zeigt exakt, wie ein Prozess gestartet wurde — oft inklusive Passwörtern, Dateipfaden und Flags.
  • Volatility 3 ist der Standard: Für alle grundlegenden Analysen reicht Volatility 3 völlig aus. Es ist schneller, besser dokumentiert und aktiv entwickelt.
  • Der Analyse-Workflow ist immer gleich: Profil → Prozesse → Command-Lines → Dateien → Strings → Prozess-Memory. Dieses Schema funktioniert für 90 % aller Memory-Forensik-Aufgaben.
  • CTFs lehren echte Forensik: MemLabs bildet realistische Szenarien ab. Die Techniken, die du hier lernst, sind direkt auf echte Incident-Response-Fälle übertragbar.
