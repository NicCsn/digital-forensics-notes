Szenario

Ein Unternehmen vermutet, dass sein Windows-Domain-Controller kompromittiert wurde. Das IR-Team hat einen RAM-Dump des betroffenen Domain-Controllers erstellt und übergibt ihn dir zur Analyse. Deine Aufgabe ist es, den Angriff zu rekonstruieren: Welches Betriebssystem läuft? Welche verdächtigen Prozesse sind aktiv? Welche Kerberos-Tickets wurden extrahiert? Und vor allem: Wurde via Kerberoasting ein Service-Account kompromittiert?

Da es sich um einen Domain-Controller handelt, erwarten wir eine Vielzahl von lsass.exe-gespeicherten Credentials, Kerberos-Tickets im Speicher und möglicherweise Mimikatz-Artefakte. Die Analyse erfolgt primär mit Volatility und ergänzend mit Hashcat zur Passwortwiederherstellung.

Wichtiger Hinweis: Memory-Forensik auf Domain-Controllern ist besonders kritisch, da lsass.exe sämtliche Kerberos-TGTs und Service-Tickets im Speicher hält. Ein Angreifer mit Administrator-Rechten kann diese Tickets dumpen und für Pass-the-Ticket- oder Kerberoasting-Angriffe nutzen.

Walkthrough — Schritt für Schritt

Schritt 1: OS-Profil und Image-Info bestimmen

Der erste Schritt in jeder Memory-Analyse: Welches Betriebssystem läuft? Volatility braucht das korrekte Profil (Vol2) oder Symbol-Tabelle (Vol3), um Datenstrukturen korrekt zu interpretieren.

# Volatility 3 — Windows-Info abrufen
vol -f campfire1.raw windows.info

# Ausgabe (Beispiel):
# Kernel Base: 0xf80271c18000
# DTB: 0x1aa000
# Symbol Table: ntkrnlmp.pdb
# NT Build: 14393.amd64fre.rs1_release.161110-2025
# OS: Windows 10 / Server 2016

# Bei Volatility 2: Profil ermitteln
vol.py -f campfire1.raw imageinfo

# Gängige Profile:
# Win10x64_14393  → Windows 10 / Server 2016 Version 1607
# Win10x64_19041  → Windows 10 / Server 2019 Version 2004
# Win2016x64_14393 → Windows Server 2016
Build-Nummern als Referenz: Build 14393 = Windows 10 1607 / Server 2016. Build 17763 = Windows 10 1809 / Server 2019. Build 19041 = Windows 10 2004. Die Build-Nummer ist kritisch für die Auswahl des korrekten Profils in Volatility 2.

Schritt 2: Prozessliste analysieren

Wir listen alle laufenden Prozesse und deren Parent-Child-Beziehungen. Verdächtig sind: Prozesse mit ungewöhnlichen Namen, Prozesse die von cmd.exe oder powershell.exe gestartet wurden, sowie bekannte Malware-Prozesse.

# Volatility 3 — Prozessliste
vol -f campfire1.raw windows.pslist

# Prozessbaum (Parent-Child-Beziehungen)
vol -f campfire1.raw windows.pstree

# Nach verdächtigen Prozessen filtern
vol -f campfire1.raw windows.pslist | grep -iE "mimikatz|psexec|powershell|cmd|wmic|rundll32|regsvr32"

# Command-Line-Argumente jedes Prozesses
vol -f campfire1.raw windows.cmdline
ProzessForensische Bedeutung
lsass.exeSpeichert Credentials, NTLM-Hashes, Kerberos-Tickets. Ziel Nr. 1 für Angreifer.
powershell.exeBase64-kodierte Command-Lines sind hochverdächtig (PowerShell-Empire, Cobalt Strike).
cmd.exeChild-Prozesse prüfen — welche Programme wurden von cmd aus gestartet?
rundll32.exeWird oft für DLL-Sideloading und C2-Kommunikation missbraucht.
mshta.exeFührt HTA/VBS aus — fast immer bösartig auf Servern.
wsmprovhost.exeHost-Prozess für WinRM — deutet auf Lateral Movement hin.

Schritt 3: Netzwerkverbindungen prüfen

Offene Netzwerkverbindungen geben Aufschluss über C2-Kommunikation, Lateral Movement und Datenexfiltration:

# Netzwerkverbindungen (Volatility 3)
vol -f campfire1.raw windows.netstat

# Nur ESTABLISHED-Verbindungen
vol -f campfire1.raw windows.netstat | grep ESTABLISHED

# DNS-Cache (welche Domains wurden aufgelöst?)
vol -f campfire1.raw windows.dnscache

Schritt 4: Kerberos-Tickets aus LSASS extrahieren

Der Domain-Controller hält Kerberos-Tickets im Speicher des lsass.exe-Prozesses. Volatility 2 bietet hierfür das kerberos-Plugin, Volatility 3 hat keinen direkten Ersatz — wir dumpen den LSASS-Speicher und analysieren ihn manuell:

# Volatility 2 — Kerberos-Tickets auflisten
vol.py -f campfire1.raw --profile=Win2016x64_14393 kerberostickets

# LSASS-Speicher dumpen (Volatility 3)
vol -f campfire1.raw -o ./dump/ windows.memmap --pid $(pgrep lsass) --dump

# Aus dem LSASS-Dump Tickets mit Mimikatz extrahieren
mimikatz.exe
# sekurlsa::kerberos /export

# Volatility 2 — Mimikatz-Plugin für direkte Extraktion
vol.py -f campfire1.raw --profile=Win2016x64_14393 mimikatz
Volatility 2 vs. 3: Volatility 3 hat kein natives Kerberos-Plugin. Für Kerberos-Forensik ist Volatility 2 mit dem mimikatz-Plugin oft die bessere Wahl. Alternativ kann man lsass.exe dumpen und mit pypykatz offline analysieren.

Schritt 5: Kerberoasting — TGS-Tickets analysieren

Kerberoasting extrahiert das TGS (Ticket Granting Service) für einen Service-Account aus dem Kerberos-Ticket. Das TGS ist mit dem NTLM-Hash des Service-Account-Passworts verschlüsselt und kann offline gecrackt werden. Die entscheidenden Indikatoren sind:

AttributBedeutung
ServiceNameDer angefragte Service (z. B. HTTP/webserver.domain.local)
TargetDomainNameDie Domain des Service-Accounts
TicketEncryptionTypischerweise RC4-HMAC (0x17) oder AES256-CTS-HMAC-SHA1-96 (0x12)
UserNameDer Benutzer, der das Ticket angefordert hat
# Kerberoastbare Tickets identifizieren
# TGS-REP mit RC4-Verschlüsselung (0x17) = crackbar mit Hashcat Modus 13100
# TGS-REP mit AES256 (0x12) = mit AES256 verschlüsselt, schwerer crackbar

# Hash-Extraktion für Hashcat
# Format: $krb5tgs$23$*USERNAME$DOMAIN$SPN$HASH
# Der Hash wird aus dem TGS-Ticket berechnet

Schritt 6: Passwörter cracken mit Hashcat

Die extrahierten Kerberos-TGS-Hashes werden mit Hashcat und einer geeigneten Wordlist gecrackt:

# Hashcat-Modus 13100 = Kerberos 5 TGS-REP etype 23 (RC4)
hashcat -m 13100 -a 0 tgs_hashes.txt /usr/share/wordlists/rockyou.txt --force

# Mit OneRuleToRuleThemAll (effizientere Regel-basierte Attacke)
hashcat -m 13100 -a 0 tgs_hashes.txt /usr/share/wordlists/rockyou.txt -r /usr/share/hashcat/rules/OneRuleToRuleThemAll.rule --force

# Bruteforce als Fallback
hashcat -m 13100 -a 3 tgs_hashes.txt ?a?a?a?a?a?a?a?a --increment --force
Kerberoasting-Prävention: Service-Account-Passwörter sollten mindestens 25 Zeichen lang, komplex und regelmässig rotiert werden. Group Managed Service Accounts (gMSA) verhindern Kerberoasting vollständig, da Windows das Passwort automatisch verwaltet und es für Angreifer praktisch unknackbar ist.

Schritt 7: Zusätzliche Credentials und Artefakte extrahieren

# NTLM-Hash-Dump
vol -f campfire1.raw windows.hashdump

# Cached Domain Credentials
vol -f campfire1.raw windows.cachedump

# Registry-Hives aus dem Speicher dumpen
vol -f campfire1.raw -o ./registry/ windows.registry.hivelist
vol -f campfire1.raw -o ./registry/ windows.registry.printkey --key "Software\Microsoft\Windows\CurrentVersion\Run"

# Strings nach "password", "pwd", "credential" suchen
strings campfire1.raw | grep -iE "password[=:]|pwd[=:]|credential[=:]" > strings_creds.txt

Exkurs: Kerberoasting — So funktioniert der Angriff

Kerberoasting ist eine Angriffstechnik gegen Service-Accounts in Active-Directory-Umgebungen. Der Ablauf ist wie folgt:

  1. Authentifizierung: Der Angreifer authentisiert sich mit einem beliebigen kompromittierten Domain-User-Account (keine Admin-Rechte nötig).
  2. TGS-Anfrage: Der Angreifer fragt ein Service-Ticket (TGS) für einen Service Principal Name (SPN) an — z. B. MSSQLSvc/sql01.domain.local.
  3. Ticket-Empfang: Der Domain-Controller stellt das TGS aus, verschlüsselt mit dem NTLM-Hash des Service-Account-Passworts.
  4. Offline-Cracking: Der Angreifer extrahiert das TGS aus dem Speicher und crackt den Hash offline mit Hashcat oder John the Ripper.
  5. Privilege Escalation: Der Service-Account hat oft höhere Rechte (z. B. Domain-Admin) — der Angreifer eskaliert.
Erkennung von Kerberoasting: Windows Event-ID 4769 (Kerberos Service Ticket Request) mit TicketEncryptionType=0x17 (RC4) ist ein Indikator. Viele TGS-Anfragen von einem einzelnen Benutzer in kurzer Zeit = Alarm!

Lessons Learned

  • Volatility-Profile sind entscheidend: Ohne das korrekte OS-Profil sind alle Analyseergebnisse unzuverlässig. Immer mit imageinfo / windows.info starten.
  • lsass.exe ist das Kronjuwel: Der LSASS-Prozess enthält NTLM-Hashes, Kerberos-Tickets, Klartext-Passwörter (bei WDigest) und DPAPI-Keys. Der Dump von LSASS ist immer der erste Schritt eines Angreifers.
  • Service-Accounts sind die Schwachstelle: Kerberoasting funktioniert, weil Service-Accounts oft schwache Passwörter haben. gMSA ist die Lösung.
  • RC4-Tickets sind crackbar: TGS-Tickets mit RC4-Verschlüsselung (etype 23) sind trivial mit Hashcat zu cracken. AES256 (etype 18) ist deutlich resistenter.
  • Memory-Dumps enthalten alles: Netzwerkverbindungen, Prozesse, Registry-Keys, Datei-Handles — der RAM-Dump ist das vollständigste Artefakt in der forensischen Analyse.
  • Volatility 2 + 3 kombiniert nutzen: Vol3 ist moderner und schneller, Vol2 hat mehr community-Plugins und das mimikatz-Modul. Beide Tools ergänzen sich.
