Kurzueberblick

Velociraptor ist eine agentenbasierte Open-Source-DFIR-Plattform, entwickelt von Mike Cohen und dem Velociraptor-Team. Anders als klassische Forensik-Tools, die ein statisches Image analysieren, arbeitet Velociraptor live auf Endpoints und kann zeitgleich hunderte oder tausende Systeme durchsuchen (Threat Hunting, IR-Triage, Compliance-Checks).

EigenschaftBeschreibung
EntwicklerMike Cohen / Velociraptor (CNCF Sandbox Projekt)
LizenzAGPLv3 (Open Source)
SpracheGo (Server/Client), VQL (Query Language), Vue.js (Frontend)
PlattformWindows, Linux, macOS
ArchitekturServer — Client — Web-Frontend
Offline-ModusJa (Offline Collector)
Abgrenzung: Velociraptor ersetzt KEIN klassisches Forensik-Image. Es ist ein Live-Response- und Threat-Hunting-Werkzeug, das auf laufenden Systemen arbeitet. Fuer die gerichtsfeste Analyse wird zusaetzlich ein forensisches Image benoetigt.

Architektur

Velociraptor besteht aus drei Kernkomponenten:

KomponenteRolleTechnologie
ServerZentraler Knoten: Empfaengt Client-Verbindungen, speichert Ergebnisse, stellt GUI bereitGo-Binary, eingebettete Datenbank
ClientAgent auf jedem Endpoint: Fuehrt VQL-Queries aus, sammelt Artefakte, sendet ErgebnisseGo-Binary (Windows-Service, Linux/macOS-Daemon)
FrontendWeb-GUI: Queries schreiben, Ergebnisse visualisieren, Notebooks, Admin-FunktionenVue.js SPA, im Server integriert

Die Kommunikation laeuft ueber einen einzigen HTTPS-Port (Default: 8000). Clients initiieren die Verbindung zum Server (Outbound), Firewall-Regeln muessen also nur server-seitig eingerichtet werden. Das Protokoll ist verschluesselt und TLS-secured.

# Vereinfachte Architektur
[Clients]  ---HTTPS:8000--->  [Velociraptor Server]
                                  |
                                  +--- Web-GUI (Port 8889)
                                  +--- Interne DB (File- oder MySQL)
Port-Konfiguration: Port 8000 = Client-Server-Kommunikation (muss von Clients erreichbar sein). Port 8889 = GUI (sollte nur intern/durch Admin erreichbar sein, nicht oeffentlich).

Deployment

Server-Setup (Linux/macOS/Windows)

# 1. Binary herunterladen
wget https://github.com/Velocidex/velociraptor/releases/latest/download/velociraptor-linux-amd64

# 2. Config generieren (interaktiv)
./velociraptor config generate -i

# 3. Server starten (Frontend + Listener)
./velociraptor --config server.config.yaml frontend -v

# 4. Als systemd-Service (optional)
sudo ./velociraptor --config server.config.yaml service install

Client-Installation (Windows)

# Client-Binary per Gruppenrichtlinie, SCCM oder manuell verteilen
velociraptor.exe --config client.config.yaml service install

# Service starten
velociraptor.exe service start

# Pruefen, ob Client verbunden ist
velociraptor.exe --config client.config.yaml query "select * from info()"

Client-Installation (Linux/macOS)

# Linux
sudo ./velociraptor --config client.config.yaml service install
sudo systemctl start velociraptor

# macOS
sudo ./velociraptor --config client.config.yaml service install
sudo launchctl load /Library/LaunchDaemons/org.velociraptor.plist
Client-Konfiguration: Die client.config.yaml enthaelt die Server-Adresse und den CA-Pin. Sie wird beim ersten config generate automatisch erstellt. Ohne gueltige Config kann der Client keine Verbindung zum Server aufbauen.

Artifacts

Artifacts sind vordefinierte Sammlungen von VQL-Queries, die eine bestimmte forensische Frage beantworten. Velociraptor bringt ueber 200 Built-in Artifacts mit, deckt Windows, Linux und macOS ab.

Custom VQL Artifacts

Eigene Artifacts werden als YAML definiert und koennen per GUI oder CLI importiert werden. Die Community pflegt eine grosse Sammlung unter der Artifact Exchange (docs.velociraptor.app/exchange).

# Custom Artifact Definition (vereinfacht)
name: Custom.ProcessList
description: Listet alle Prozesse mit Binary-Pfad
type: CLIENT
parameters:
  - name: ProcessFilter
    default: .
sources:
  - query: |
      SELECT Name, Pid, Exe, CommandLine
      FROM pslist()
      WHERE Name =~ ProcessFilter

VQL — Velociraptor Query Language

VQL ist die Abfragesprache von Velociraptor. Sie aehnelt SQL, ist aber speziell fuer forensische Abfragen auf Betriebssystem-Daten konzipiert. VQL kann Dateien lesen, Registry-Schluessel auswerten, Prozesse auflisten und Netzwerkverbindungen analysieren.

# Basis-VQL — alle laufenden Prozesse
SELECT Name, Pid, Exe FROM pslist() WHERE Name =~ "cmd|powershell"

# Event-Logs parsen (letzte 24 Stunden)
SELECT System.TimeCreated, System.EventID, System.Computer
FROM parse_evtx(filename="C:\\Windows\\System32\\winevt\\Logs\\Security.evtx")
WHERE System.TimeCreated > now() - 86400

# Registry auslesen
SELECT Key, Value FROM read_reg_key(
  globs="HKEY_LOCAL_MACHINE\\Software\\Microsoft\\Windows\\CurrentVersion\\Run\\**")

# Dateien mit Pfad-Filter suchen
SELECT FullPath, Size, Mtime FROM glob(
  globs="C:\\Users\\*\\AppData\\Local\\Google\\Chrome\\**\\History")

# MFT parsen
SELECT FileName, ParentPath, Created0x10, LastModified0x10
FROM parse_mft(filename="C:\\$MFT")
WHERE FileName =~ "\.exe$"
VQL-PluginFunktion
pslist()Laufende Prozesse auflisten
glob()Dateien per Pfad-Pattern suchen
parse_evtx()Windows-EventLogs parsen
parse_mft()MFT (Master File Table) parsen
read_reg_key()Registry-Keys auslesen
yara()YARA-Scan auf Dateien/Speicher
parse_csv()CSV-Dateien einlesen
chain()Mehrere Queries verketten
VQL lernen: Die Velociraptor-GUI hat einen eingebauten VQL-Editor mit Autovervollstaendigung. Das offizielle VQL-Reference-Handbuch (docs.velociraptor.app/vql_reference/) dokumentiert alle Plugins und Funktionen.

Offline-Kollektor

Der Offline Collector ist ein stand-alone Binary, das ohne Server-Verbindung auf einem Zielsystem laeuft. Ideal fuer Incident Response vor Ort, wenn kein Velociraptor-Server verfuegbar ist oder das System isoliert werden muss.

# 1. Collector Binary erstellen (auf dem Server)
velociraptor --config server.config.yaml tools collector \
  --artifacts "Windows.Forensics.Prefetch,Windows.Sys.Users" \
  --output collector.exe

# 2. collector.exe auf Zielsystem kopieren und ausfuehren
collector.exe

# 3. Ergebnis ist eine ZIP-Datei, die auf dem Server importiert wird
velociraptor --config server.config.yaml imports collector_results.zip

Der Offline-Kollektor ist der wichtigste Use-Case fuer Incident Responder: Er sammelt alle definierten Artefakte, speichert sie in einem ZIP-Archiv, und hinterlaesst minimale Spuren auf dem Zielsystem (keine Installation, kein persistenter Service).

Hunting-Notebooks

Hunting-Notebooks sind interaktive Analyse-Dokumente in der Velociraptor-GUI, die VQL-Queries, Ergebnisse und Notizen in einem Dokument vereinen. Sie sind vergleichbar mit Jupyter-Notebooks, aber speziell fuer DFIR konzipiert.

Notebook-TypZweck
HuntVordefinierte Query auf vielen Clients ausfuehren (z.B. "Suche auf allen 500 Endpoints nach dieser IOC-Liste")
CollectionEinmalige Collection von Artefakten auf einem bestimmten Client
Event MonitoringDauerhafte Ueberwachung von Events (z.B. neue Prozesse, Registry-Aenderungen)
Vorsicht bei Hunts: Ein Hunt auf 1000+ Clients mit einer teuren Query (z.B. YARA-Scan aller EXE-Dateien) kann die Endpoints stark belasten. Immer zuerst auf einem einzelnen Client testen und CPU/IO-Limits in VQL setzen.

Praktische Anwendung: Offline-Collection von Windows-Artefakten

Szenario: Ein Windows-Server zeigt verdaechtige Netzwerkverbindungen. Du sollst eine forensische Collection durchfuehren, ohne einen Velociraptor-Server im Netzwerk zu haben.

  1. Collector vorbereiten (auf deinem Analyse-Rechner mit Velociraptor-Server): 
    velociraptor --config server.config.yaml tools collector \
  --artifacts Windows.Forensics.Prefetch,\
Windows.Sys.Users,\
Windows.EventLogs.Powershell,\
Windows.Network.NetstatEnriched,\
Windows.Detection.Yara.Process,\
Windows.Forensics.Usn \
  --output win_collector.exe
  2. win_collector.exe auf USB-Stick kopieren und zum Zielserver bringen.
  3. Als Administrator ausfuehren: win_collector.exe
  4. Ergebnis-ZIP wird im aktuellen Verzeichnis erstellt, z.B. Collection-2026-07-02.zip
  5. ZIP auf deinen Analyse-Rechner uebertragen und importieren: 
    velociraptor --config server.config.yaml imports Collection-2026-07-02.zip
  6. In der GUI analysieren: Unter "Collected Artifacts" die Ergebnisse einsehen, filtern und in Notebooks pivotieren.
Admin-Rechte: Fuer den Zugriff auf viele Artefakte (MFT, bestimmte Registry-Hives, raw disk access) sind Admin-Rechte auf dem Zielsystem erforderlich. Der Collector meldet bei fehlenden Rechten eine Warnung, fuehrt aber die restlichen Queries trotzdem aus.
