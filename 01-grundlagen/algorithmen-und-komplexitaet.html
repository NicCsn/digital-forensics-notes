Kurzüberblick

Ein Algorithmus ist eine präzise Schritt-für-Schritt-Anweisung zur Lösung eines Problems. In der Forensik brauchen Sie Algorithmen z.B. für File Carving, Duplikatsuche, Log-Analyse oder Passwort-Hashing. Die Komplexität (O-Notation) sagt Ihnen, wie ein Algorithmus skaliert — entscheidend bei großen Datenmengen von mehreren Terabyte.

Forensischer Grundsatz: Ein 8-TB-Festplattenimage mit einem O(n²)-Algorithmus zu analysieren ist keine Geduldsfrage — es ist praktisch unmöglich. Die Wahl des richtigen Algorithmus entscheidet, ob eine Analyse Minuten oder Jahrzehnte dauert.

O-Notation (Landau-Notation)

Was beschreibt die O-Notation?

Die O-Notation beschreibt das Laufzeit- oder Speicherverhalten eines Algorithmus im Worst Case in Abhängigkeit der Eingabegröße n. Sie abstrahiert von Hardware, Programmiersprache und konstanten Faktoren.

NotationNameBeispieln = 1 TB (10^12 Bytes)
O(1)KonstantArray-Zugriff per Index, Hash-Table Lookupsofort
O(log n)LogarithmischBinäre Suche in sortierter Dateiliste~40 Schritte
O(n)LinearLineare Suche, Hash-Berechnung~1 Stunde (I/O-gebunden)
O(n log n)QuasilinearMergesort, Heapsort, Timsort~2 Stunden
O(n²)QuadratischBubblesort, naive Duplikatsuche~Jahre (10^24 Operationen!)
O(2^n)ExponentiellBrute-Force-Passwort (n = Schlüssellänge)nicht praktikabel

Forensik-Beispiele für jede Komplexitätsklasse

KlasseForensik-BeispielWarum relevant?
O(1)Hash-Table-Lookup: "Ist Hash X in NSRL?"80 Mio. Hashes in konstanter Zeit prüfen
O(log n)Binäre Suche in sortiertem Event-Log nach ZeitstempelStatt 1 Mrd. Events linear zu scannen, in ~30 Vergleichen finden
O(n)sha256sum image.dd, strings image.ddJedes Byte einmal lesen — optimal für vollständige Scans
O(n log n)Timeline sortieren: 100 Mio. Events → Super-Timelineplaso/log2timeline sortiert alle extrahierten Zeitstempel
O(n²)Jeden Hash mit jedem anderen vergleichen (Duplikatsuche)Nie so implementieren! Stattdessen: sortieren O(n log n) + linear scannen
O(2^n)Brute-Force: Alle 8-Zeichen-Passwörter durchprobieren95^8 ≈ 6,6 × 10^15 Kombinationen — ohne GPU nicht machbar

Grafik: Wachstumsvergleich

# Vergleich der Komplexitätsklassen (Python)
import math
n_values = [10, 100, 1000, 10000, 100000, 1000000]

print(f"{'n':>10} {'O(1)':>6} {'O(log n)':>10} {'O(n)':>10} {'O(n log n)':>12} {'O(n²)':>12}")
for n in n_values:
    print(f"{n:>10} {1:>6} {math.log2(n):>10.1f} {n:>10} {n*math.log2(n):>12.1f} {n**2:>12}")

# Ausgabe zeigt: Bei n=1.000.000 ist O(n)=1M, O(n²)=1 Billion — der Unterschied ist dramatisch!
Merkhilfe: Ein O(n)-Algorithmus auf 8 TB Daten ist technisch machbar (einmal alles lesen). Ein O(n²)-Algorithmus auf denselben Daten würde ~10^25 Operationen benötigen — das Universum ist nicht alt genug dafür. In der Forensik bedeutet das: sortieren, dann scannen (O(n log n) + O(n)) statt naiv vergleichen (O(n²)).

Suchalgorithmen für die Forensik

Lineare Suche (Grep)

  • Komplexität: O(n)
  • Prinzip: Jedes Element der Reihe nach prüfen
  • Forensik-Nutzung: grep, strings, xxd, Rohdaten-Scanning
  • Optimal für: Unsortierte Daten, einmalige Volltextsuche, Pattern-Matching auf Rohdaten
  • Grenzen: Bei 10 TB pro Suchmuster wird es ineffizient — besser: einmal indizieren (Aho-Corasick)
# Lineare Suche in der Praxis
$ grep -r "password" /mnt/forensic_image/
$ strings image.dd | grep -i "admin@"

Binäre Suche (Sortierte Daten)

  • Komplexität: O(log n)
  • Voraussetzung: Daten müssen sortiert sein (Sortierung kostet O(n log n) upfront)
  • Forensik-Nutzung: Zeitstempel-Suche in sortierten Event-Logs, Suchen in sortierten Hash-Listen (NSRL), MFT-Record-Nummern (sind aufsteigend)
# Binäre Suche in sortierter Datei
# Angenommen: hashes.txt ist sortiert, 80 Mio. Zeilen
$ grep -n "^5d41402abc4b2a76b9719d911017c592" hashes.txt  # lineare Suche (langsam)
# Besser: lookup in vorgefertigter Hash-Table (O(1))

Hash-basierte Suche (O(1) Lookup)

  • Komplexität: O(1) durchschnittlich, O(n) worst-case bei vielen Kollisionen
  • Prinzip: Hash-Funktion berechnet Index im Array, direkter Zugriff
  • Forensik-Nutzung: NSRL-Abgleich, Hash-Sets (HashKeeper), Deduplizierung in Forensic-Tools
  • IDEAL für: Known-Good/Known-Bad Hash-Abgleich in Echtzeit
# Python: Hash-Set für O(1) Lookup
known_bad = set()
with open("malware_hashes.txt") as f:
    for line in f:
        known_bad.add(line.strip().lower())  # O(1) insert

# Später: O(1) lookup
if file_hash in known_bad:
    print(f"MALWARE FOUND: {file_hash}")

Aho-Corasick: Multi-Pattern-Suche

Was ist Aho-Corasick?

Aho-Corasick ist ein String-Matching-Algorithmus, der alle Patterns in einem einzigen Durchlauf durch den Text findet.

  • Komplexität: O(n + m + z) wobei n = Textlänge, m = Summe aller Pattern-Längen, z = Anzahl Treffer
  • Funktionsweise: Baut einen Trie (Präfixbaum) aus allen Such-Patterns, ergänzt um Fehler-Links (Failure-Links) — ein endlicher Automat
  • Schlüsselvorteil: Die Anzahl der Patterns hat (fast) keinen Einfluss auf die Suchzeit!
Tool/SystemVerwendung von Aho-CorasickForensische Relevanz
YARAPattern-Matching-Engine basiert auf Aho-CorasickMalware-Scanning mit Tausenden von Regeln gleichzeitig
ClamAVViren-Signatur-MatchingOpen-Source AV-Scanner — scannt Millionen Signaturen in einem Durchlauf
Snort/SuricataIDS/IPS Pattern MatchingNetzwerkverkehr in Echtzeit auf Tausende IOCs prüfen
bulk_extractorMulti-Pattern-Extraktion aus RohdatenE-Mail-Adressen, URLs, Kreditkartennummern gleichzeitig extrahieren
strings (GNU)Kein AC, aber ähnliches PrinzipASCII/Unicode-Strings aus Binärdaten extrahieren

Forensik-Szenario: IOC-Scanning

Sie haben eine Liste mit 50.000 verdächtigen IPs, Domains und Hashes aus Threat Intelligence Feeds. Sie müssen ein 500 GB forensisches Image danach durchsuchen.

# Naiver Ansatz (Tage!):
for ioc in $(cat iocs.txt); do
    grep -r "$ioc" /mnt/image/     # 50.000 × 500 GB = 25 PB I/O!
done

# Aho-Corasick-Ansatz (Minuten!):
# Einmal 500 GB lesen, alle 50.000 IOCs gleichzeitig matchen
# Tools: bulk_extractor, YARA mit IOC-Liste, oder pyahocorasick
python3 -c "
import ahocorasick
A = ahocorasick.Automaton()
for ioc in open('iocs.txt'):
    A.add_word(ioc.strip(), ioc.strip())
A.make_automaton()
# → scannt 500 GB in O(n) mit allen 50.000 Patterns gleichzeitig
"
Performance-Vergleich: 1.000 Patterns × 100 GB Logs mit Grep ≈ ~28 Stunden (1.000 separate lineare Scans). Mit Aho-Corasick ≈ ~10 Minuten (ein linearer Scan). Faktor > 150× schneller!

Bloom-Filter: Probabilistische Datenstruktur

Konzept

Ein Bloom-Filter ist eine platzsparende Datenstruktur, die Mengenzugehörigkeit prüft — mit einer bewussten Fehlertoleranz:

  • "Definitiv nicht in der Menge" → 100% korrekt (keine Falsch-Negative)
  • "Vielleicht in der Menge" → mit einstellbarer Falsch-Positiv-Rate (typisch 0,1–1 %)
  • Speicherverbrauch: ~1 Byte pro Element (statt 32+ Bytes für Hash-Set)

Forensische Anwendung: NSRL-Abgleich

Die NSRL (National Software Reference Library) enthält ~80 Millionen Hash-Werte bekannter, unverdächtiger Dateien (Windows-Systemdateien, Standard-Software).

AnsatzSpeicherLookup-ZeitFalsch-Positive
Hash-Set (MD5, 16 Bytes × 80M)~10 GB RAMO(1)0%
Bloom-Filter (1% FP-Rate)~100 MB RAMO(k) ≈ O(1)~1%
Datenbank (SQLite, indiziert)~5 GB DiskO(log n)0%
Workflow: 1) Bloom-Filter sagt "nein" → Datei ist definitiv nicht in NSRL → verdächtig, analysieren! 2) Bloom-Filter sagt "vielleicht" → exakter Hash-Lookup in vollständiger NSRL-Datenbank → bestätigt oder nicht. Der Bloom-Filter eliminiert 99% der irrelevanten Dateien mit nur 100 MB RAM.

Weitere Bloom-Filter-Anwendungen in der Forensik

  • Passwort-Cracking: Bloom-Filter für bereits getestete Passwörter (verhindert doppelte Arbeit)
  • Carving: Bereits bekannte Sektoren überspringen (Deduplizierung im Voraus)
  • Netzwerkforensik: Bereits gesehene IP-Adressen/Flows tracken (Cuckoo Sandbox)
  • Browser-Forensik: Besuchte URLs in Bloom-Filter (Chrome Safe Browsing)

Hashing: Kryptografisch vs. nicht-kryptografisch

Kryptografische Hash-Funktionen

HashLängeSicherheitForensische Verwendung
MD5128 BitGebrochen (Kollisionen)Legacy, File-Integrity (nicht für Sicherheit), NSRL-Einträge
SHA-1160 BitGebrochen (SHAttered)Git (intern), Legacy-Zertifikate
SHA-256256 BitSicherForensische Images verifizieren, Beweismittel-Integrität, Blockchain
SHA-3VariabelSicherNeuere Standard-Hashes, Post-Quantum-resistent (kein SHA-2-Nachfolger notwendig)

Nicht-kryptografische Hash-Funktionen

Diese sind auf Geschwindigkeit optimiert, nicht auf Sicherheit:

  • Fowler-Noll-Vo (FNV): Extrem schnell, gute Verteilung, für Hash-Tables. FNV-1a ist die verbreitetste Variante
  • MurmurHash: Sehr schnell, verwendet in vielen Datenbanken und NoSQL-Systemen
  • xxHash: Aktuell einer der schnellsten nicht-kryptografischen Hashes, verwendet in zstd, LZ4
  • CityHash/FarmHash: Von Google für interne Hash-Tables optimiert

Hash-Kollisionen und forensische Bedeutung

  • Geburtstagsparadoxon: Bei einem 128-Bit-Hash braucht man ~2^64 (~1,8 × 10^19) Dateien für eine 50%-Kollisionswahrscheinlichkeit — praktisch irrelevant für forensische Duplikatsuche
  • MD5 in der Forensik: Obwohl kryptografisch gebrochen, ist MD5 für reine Duplikaterkennung immer noch sicher (Kollisionen können nicht zufällig entstehen, sondern müssen aktiv konstruiert werden)
  • Best Practice: Immer SHA-256 verwenden, MD5 höchstens zusätzlich für Kompatibilität mit Legacy-Tools

Sortieralgorithmen und ihre Relevanz

Vergleich der wichtigsten Sortieralgorithmen

AlgorithmusAverageWorstStabilForensische Verwendung
QuicksortO(n log n)O(n²)NeinSchnellster In-Memory-Sort, in vielen C-Standardbibliotheken
MergesortO(n log n)O(n log n)JaExternes Sortieren (Festplatte), stabil = wichtig für mehrstufiges Sortieren
TimsortO(n log n)O(n log n)JaPython/Java Standard-Sort! Optimal für teilweise vorsortierte Daten (Logs!)
HeapsortO(n log n)O(n log n)NeinGarantiertes O(n log n), in-place — für eingebettete Systeme

Warum Timsort für die Forensik relevant ist

Timsort (der Standard-Sort in Python und Java) ist ein hybrider Sortieralgorithmus, der von Mergesort und Insertion Sort abstammt:

  • Optimal für realweltliche Daten: Logdateien, Zeitstempel und Event-Streams sind oft teilweise vorsortiert. Timsort erkennt und nutzt existierende "Runs"
  • Stabil: Mehrstufiges Sortieren (z.B. erst nach IP, dann nach Zeitstempel) behält die Reihenfolge gleicher Elemente bei
# Python: Timsort ist der Standard
events = load_timeline()  # Liste von (timestamp, source, event)
events.sort()             # Timsort — O(n log n), nutzt existierende Runs

# Mehrstufig sortieren (stabil)
events.sort(key=lambda e: e.event)    # Erst nach Event-Typ
events.sort(key=lambda e: e.timestamp) # Dann nach Zeit — behält Event-Reihenfolge!

Externes Sortieren bei großen Datenmengen

Wenn Daten größer als RAM sind, wird externes Sortieren benötigt:

# Linux: sort kann extern sortieren (Mergesort auf Festplatte)
$ sort -S 50% --parallel=8 -o sorted.txt huge_log.txt
# -S 50%: Nutzt 50% des RAM als Puffer
# --parallel=8: 8 Threads
# Mergesort-Phasen: sortierte Chunks schreiben → mergen

Regular Expressions: DFA vs NFA, ReDoS

Theoretische Grundlagen

Reguläre Ausdrücke werden von zwei Arten endlicher Automaten verarbeitet:

EigenschaftDFA (Deterministisch)NFA (Nicht-deterministisch)
ZuständeGenau ein Folgezustand pro EingabeMehrere mögliche Folgezustände
BacktrackingKein BacktrackingBacktracking bei Fehlschlag
LaufzeitO(n) — garantiert linear!O(2^n) worst-case — exponentiell!
KompilierungszeitLangsam, großer SpeicherSchnell, kompakt
FeaturesKeine BackreferencesBackreferences, Lookahead/Lookbehind
EnginesRE2 (Google), Hyperscan (Intel), Rust regexPCRE (Perl, Python, grep -P), JavaScript, .NET

ReDoS (Regular Expression Denial of Service)

ReDoS tritt auf, wenn eine Regex-Engine mit NFA-Backtracking auf bestimmte Eingaben exponentiell viele Pfade evaluiert:

# ReDoS-anfällige Regex (katastrophales Backtracking):
# /^(a+)+$/  —  matcht "aaaa...a", aber NICHT "aaaa...X"
# Eingabe "aaaaaaaaaaaaaaaaaaaaX" (20× a + X)
# Die Engine probiert ALLE Aufteilungen der a's → 2^20 Pfade → evtl. Jahre!

# Sichere Alternative (kein verschachteltes Quantifier):
# /^a+$/  —  macht dasselbe, aber O(n)!
Forensik-Gefahr: Wenn Sie eine selbstgeschriebene Regex auf 10 GB Logs anwenden, die ReDoS-anfällig ist, kann Ihre Analyse scheinbar hängen. Die CPU ist bei 100%, aber es kommt kein Fortschritt. Regeln: 1) Kein verschachteltes Quantifiering ((a+)+), 2) Nicht mehrere Alternativen mit gemeinsamen Präfix ((a|ab)+), 3) Im Zweifel RE2/Hyperscan statt PCRE verwenden.

Praktische Forensik-Grep-Muster

# E-Mail-Adressen (vereinfacht, O(n), DFA-sicher)
grep -oP '[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}' file.txt

# IPv4-Adressen (DFA-sicher)
grep -oP '\b(?:(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\.){3}(?:25[0-5]|2[0-4][0-9]|[01]?[0-9][0-9]?)\b'

# Kreditkartennummern (Visa/MC/Amex — Luhn-Prüfung separat nötig)
grep -oP '\b(?:4[0-9]{12}(?:[0-9]{3})?|5[1-5][0-9]{14}|3[47][0-9]{13})\b'

# Base64-Strings (lange Zeichenfolgen mit typischem Alphabet)
grep -oP '[A-Za-z0-9+/]{40,}={0,2}'

bulk_extractor-Prinzip

bulk_extractor ist ein forensisches Tool von Simson Garfinkel, das auf Geschwindigkeit optimiert ist:

  • Multi-Pattern-Scanning: Scannt gleichzeitig nach E-Mails, URLs, Kreditkarten, GPS-Koordinaten, Domain-Namen usw.
  • Optimierungen: Nutzt Aho-Corasick-ähnliche Automaten, vermeidet Backtracking, arbeitet in 16 MB-Blöcken für Cache-Lokalität
  • Kein Dateisystem-Parsing: Arbeitet direkt auf Rohdaten (Byte-Strom) — findet Daten auch in gelöschten/Fragmentierten Bereichen
  • Parallelisierung: Multi-Threaded, skaliert linear mit CPU-Kernen
# bulk_extractor: Forensik-Schweizer-Taschenmesser
$ bulk_extractor -o output/ image.dd
# Extrahiert: E-Mails, URLs, Telefonnummern, GPS, EXIF, JSON, Kreditkartennummern
# Ausgabe in output/: url.txt, email.txt, telephone.txt, ccn.txt, ...

Praktische Anwendung / Befehle

Hashing und Deduplizierung

# Alle Dateien hashen und Duplikate finden
find /mnt/image/ -type f -exec sha256sum {} \; | sort | uniq -w64 -dD

# Fuzzy-Hashing (ähnliche Dateien finden)
ssdeep -r /mnt/image/ > hashes.ssdeep
ssdeep -m hashes.ssdeep suspicious_file.exe

Sortieren und Filtern

# Große Datei sortieren (externes Mergesort)
sort -S 50% --parallel=8 -o sorted.txt huge_log.txt

# Häufigkeitsanalyse
sort log.txt | uniq -c | sort -rn | head -20

# Mehrstufig sortieren (Unix)
sort -k2,2 -k1,1nr data.txt  # Nach Spalte 2 alphabetisch, dann Spalte 1 numerisch absteigend

IOC-Scanning und Pattern-Matching

# Fixed-String-Matching (schneller als Regex!)
grep -F -f iocs.txt access.log

# YARA-Regeln kompilieren und scannen
yara -r rules.yar /mnt/image/

# bulk_extractor für Rohdaten-Extraktion
bulk_extractor -o output/ image.dd
