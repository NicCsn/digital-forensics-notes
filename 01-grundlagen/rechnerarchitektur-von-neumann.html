Kurzueberblick

Jeder Computer folgt einem Architekturprinzip. Die zwei fundamentalen Modelle sind die Von-Neumann-Architektur (gemeinsamer Speicher fuer Code und Daten, alle Desktop-PCs) und die Harvard-Architektur (getrennte Speicher, Mikrocontroller, ARM Cortex-M).

Fuer die Digitale Forensik ist das Verstaendnis der Rechnerarchitektur essenziell, weil sie bestimmt: wo Daten gespeichert werden, welche Spuren fluechtig sind und wie Malware auf Hardware-Ebene agiert (Cache Poisoning, DMA-Angriffe, Firmware-Rootkits).

Von-Neumann-Architektur

Die Von-Neumann-Architektur (1945, John von Neumann) ist das dominierende Design fuer Universalrechner. Ihr Kernmerkmal: Programme und Daten teilen sich denselben Speicher. Das macht sie flexibel (jedes Programm kann geladen werden), aber auch anfaellig fuer Buffer-Overflow-Angriffe (Code kann als Daten eingeschleust und dann ausgefuehrt werden).

Komponenten

KomponenteFunktionForensische Relevanz

Von-Neumann-Flaschenhals: CPU und Speicher teilen sich EINEN Bus → die CPU wartet oft auf Daten aus dem RAM. Caches und Prefetching wurden entwickelt, um diesen Flaschenhals zu umgehen — genau diese Mechanismen sind forensisch relevant (Cache-Artefakte, Prefetch-Dateien).

Harvard-Architektur

Getrennte Speicher und Busse fuer Programmcode und Daten. Vorteil: Code kann nicht versehentlich ueberschrieben werden (Sicherheit). Nachteil: weniger flexibel, mehr Hardware-Aufwand.

Von Neumann

  • 1 gemeinsamer Bus fuer Code + Daten
  • Flexibel, Software-definiert
  • Anfaellig fuer Buffer-Overflows
  • Typisch fuer: x86-64 Desktop

Harvard

  • 2 getrennte Busse (Code + Daten)
  • Hoehere Sicherheit, weniger Flexibilitaet
  • Schutz vor Code-Injection
  • Typisch fuer: ARM Cortex-M, AVR, PIC
Modifiziertes Harvard: Moderne Desktop-CPUs (Intel, AMD) nutzen ein modifiziertes Harvard-Design: Getrennte L1-Caches fuer Instruktionen und Daten, aber gemeinsamer Hauptspeicher. Forensisch: Cache-Spuren koennen nach Prozessor-Generation unterschiedlich sein (Intel vs AMD Cache-Hierarchie).

Pipelining und Superskalaritaet

Moderne CPUs fuehren Instruktionen nicht sequenziell aus, sondern in einer Pipeline — mehrere Instruktionen gleichzeitig in verschiedenen Stufen:

Ergebnis in Register schreiben

  • Superskalar: Mehrere Pipelines parallel → mehrere Instruktionen pro Taktzyklus
  • Out-of-Order Execution: CPU ordnet Instruktionen fuer optimale Auslastung um → forensisch relevant: Zeitstempel von Instruktionen entsprechen nicht der Programmreihenfolge
  • Branch Prediction: CPU errät, welcher Code-Pfad als naechstes kommt → Spectre (CVE-2017-5753) nutzt falsche Vorhersagen zum Auslesen von Kernel-Speicher
  • Speculative Execution: CPU fuehrt Instruktionen SPEKULATIV aus, bevor sicher ist, dass sie gebraucht werden → Meltdown (CVE-2017-5754) liest geschuetzten Speicher ueber spekulative Ausfuehrung
Spectre/Meltdown (2018): Hardware-Schwachstellen, die die Isolation zwischen User-Mode und Kernel-Mode aufheben. Forensisch: Ein Angreifer kann ueber JavaScript sensible Daten aus dem Browser-Prozess auslesen. Patch-Status ueber /sys/devices/system/cpu/vulnerabilities/ pruefen.

Register~1 KB, <1 ns
L1-Cache~64 KB, ~1 ns
L2-Cache~512 KB, ~4 ns
L3-Cache~16 MB, ~12 ns
RAM~16 GB, ~100 ns
SSD / NVMe~1 TB, ~100 us
zunehmende Groesse, abnehmende Geschwindigkeit → ← fluechtig → ← persistent →
Cold-Boot-Attacke: RAM-Inhalte ueberleben einen kurzen Stromausfall (2-5 Sekunden ohne Kuehlung, Minuten mit Kuehlspray). Angreifer kann RAM aus einem laufenden System ausbauen und in einem anderen Rechner auslesen → Verschluesselungsschluessel (BitLocker, FileVault) koennen extrahiert werden. Gegenmassnahme: TPM + Secure Boot + RAM-Verschluesselung (Intel TME, AMD SME).

Warnung: RAM-Dump-Analyse auf ARM benoetigt zwingend das richtige Volatility-Profil und eine ARM-spezifische Symboltabelle. Ein x86-Profil auf einem ARM-Dump liefert strukturell falsche Ergebnisse — nicht nur unvollstaendig, sondern aktiv irrefuehrend!

Bootprozess: BIOS vs UEFI

Der Bootprozess ist forensisch relevant, weil er bestimmt, welche Sicherheitsmechanismen (Secure Boot, TPM) aktiv sind und wo sich Angreifer in der Boot-Kette einnisten koennen (Bootkits).

PhaseLegacy BIOSUEFI (Modern)
Firmware16-Bit Real Mode, MBR-basiert32/64-Bit, GPT-basiert, EFI-Partition
BootloaderErste 446 Bytes des MBR/EFI/BOOT/BOOTX64.EFI
Secure BootNicht vorhandenValidiert Signatur jedes Boot-Schrittes
ForensikMBR leicht manipulierbar (Bootkits)Signaturpruefung via TPM, Boot-Logs in EFI-Variablen
IntegritaetspruefungKeine (ausser TPM PCR)Measured Boot: jeder Schritt liefert Hash an TPM
Forensische Artefakte im UEFI: EFI-Variablen enthalten Boot-Order, Secure-Boot-Konfiguration und Boot-Logs. Auslesen via efivarfs unter Linux oder Get-FirmwareEnvironmentVariable unter Windows. Manipulierte Boot-Order ist ein starkes Indiz fuer ein Bootkit.

Forensische Relevanz — Zusammenfassung

  • RAM ist fluessig aber entscheidend: Prozesse, Netzwerkverbindungen, Entschluesselungsschluessel existieren NUR im RAM. RAM-Dump vor System-Shutdown hat hoechste Prioritaet (Order of Volatility)
  • Cache als versteckte Quelle: L1/L2/L3-Caches enthalten Daten, die nie auf die Festplatte geschrieben wurden. Forensisch extrem schwer zu extrahieren, aber theoretisch moeglich (Cold-Boot, JTAG-Debugging)
  • DMA als Angriffsvektor: Thunderbolt, FireWire, PCIe — Geraete mit DMA-Zugriff umgehen die CPU und koennen direkt RAM lesen/schreiben. Forensisch: Pruefen, ob DMA-faehige Geraete angeschlossen waren
  • Spectre/Meltdown-Klasse: CPU-Seitenkanalangriffe, die Prozess-Isolation unterlaufen. Erkennung nur ueber Mikrocode-Patch-Level und ungewoehnliche Cache-Zugriffsmuster (extrem schwer forensisch nachweisbar)
  • ARM-Forensik im Kommen: Apple Silicon Macs, Smartphones, IoT, Embedded Systems — alle ARM-basiert. Forensische Tools muessen ARM-Profile unterstuetzen
