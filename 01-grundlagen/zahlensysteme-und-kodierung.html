Kurzüberblick

Computer speichern alle Daten als Bit-Folgen. Um Rohdaten forensisch zu interpretieren, muss man die zugrunde liegenden Zahlensysteme verstehen: Binär (Basis 2), Oktal (Basis 8), Dezimal (Basis 10) und Hexadezimal (Basis 16). Kodierungen wie ASCII, UTF-8 und Base64 übersetzen zwischen Bitmustern und menschenlesbaren Zeichen. In der Forensik ist dieses Wissen unverzichtbar — vom Hex-Dump über Timestamp-Dekodierung bis zur Malware-Analyse.

Forensischer Grundsatz: Jedes Byte in einem forensischen Image ist potenziell ein Beweismittel. Ohne Verständnis der Zahlensysteme und Kodierungen können Sie Rohdaten nicht korrekt interpretieren und übersehen oder verfälschen Beweise.

Zahlensysteme: Dezimal, Binär, Oktal, Hexadezimal

Übersicht der Systeme

SystemBasisZiffernPräfixBeispielForensischer Kontext
Dezimal100–9—255Menschenlesbare Darstellung von Größen (Bytes, KB, MB)
Binär20, 10b0b11111111Rohdaten, Bitmasken, File-Permissions, Flags
Oktal80–70 (C/Unix)0377Unix-Dateirechte (chmod 755), PDP-11 Legacy
Hexadezimal160–9, A–F0x0xFFHex-Dumps, Speicheradressen, Magic Bytes, Hash-Werte

Binärsystem im Detail

  • Ein Bit hat zwei Zustände: 0 oder 1
  • n Bits können 2^n verschiedene Werte darstellen
  • 8 Bit = 1 Byte, das auf den meisten modernen Architekturen die kleinste adressierbare Einheit ist
  • Ein Nibble sind 4 Bit, also ein halbes Byte — entspricht genau einer Hex-Ziffer
  • Ein Word ist architekturabhängig: 16 Bit (x86 real mode), 32 Bit (x86 protected mode), 64 Bit (x86-64)
Merkhilfe: 2^10 = 1024 ≈ 1 KB, 2^20 ≈ 1 MB, 2^30 ≈ 1 GB, 2^40 ≈ 1 TB. Diese Zweierpotenzen tauchen überall in der Forensik auf — von Sektorgrößen (512 = 2^9) bis zu NTFS-Clustergrößen (4096 = 2^12).

Hexadezimalsystem im Detail

  • Hex: Ziffern 0–9 + A–F (Basis 16). Zwei Hex-Ziffern = 1 Byte
  • Präfix 0x (z.B. 0x4A = 74 dezimal) oder Suffix h in manchen Kontexten
  • Warum Hex dominierte: 1 Byte = genau 2 Hex-Ziffern — perfekte 1:1-Abbildung. Bei Oktal braucht man 3 Ziffern mit Überhang
  • Hex in der Forensik: Magic Bytes (z.B. JPEG = FF D8 FF E0), File Signaturen, Speicheradressen, Hash-Werte (SHA-256 = 64 Hex-Zeichen)

Oktalsystem im Detail

  • Oktal: Ziffern 0–7 (Basis 8)
  • Präfix 0 in C/Unix (z.B. 0755 = rwxr-xr-x)
  • Forensisch relevant: Unix-Dateirechte, PDP-11-Architektur (historisch), einige Dateiformate (tar-Archive speichern oktale Größen)

Umrechnungsmethoden

  • Dezimal → Binär: Sukzessive Division durch 2, Reste von unten nach oben lesen
  • Binär → Hex: Von rechts nach links in 4er-Gruppen teilen, jede Gruppe in Hex-Ziffer umwandeln
  • Hex → Dezimal: Jede Stelle mit 16^Position multiplizieren und summieren
# Python: Zahlensystem-Umrechnungen
>>> hex(255)       # '0xff'
>>> bin(42)        # '0b101010'
>>> oct(64)        # '0o100'
>>> int('0x4A', 16) # 74
>>> int('1010', 2)  # 10
>>> 0xFF            # 255 — Python interpretiert Hex-Literale direkt
Forensik-Tipp: In Hex-Editoren wie HxD oder 010 Editor sehen Sie alle drei Darstellungen nebeneinander: Hex-Werte (Mitte), ASCII-Interpretation (rechts), und die Rohdaten. Lernen Sie, häufige Byte-Muster auf einen Blick zu erkennen — 0x00 (Null), 0xFF (oft Padding), 0x20 (Space), 0x0D0A (CRLF).

2er-Komplement: Negative Zahlen und Overflow

Wie funktioniert das 2er-Komplement?

Moderne Computer verwenden das 2er-Komplement zur Darstellung negativer Ganzzahlen. Der Vorteil: Addition und Subtraktion benötigen dieselbe Hardware — es gibt keine Sonderfälle für negative Zahlen.

  • Bildung des 2er-Komplements: Alle Bits invertieren (1er-Komplement), dann +1 addieren
  • Wertebereich 8-Bit (signed): −128 bis +127
  • Wertebereich 32-Bit (signed): −2.147.483.648 bis +2.147.483.647
  • Erkennung: Das höchstwertige Bit (MSB) ist das Vorzeichenbit: 0 = positiv, 1 = negativ
Dezimal8-Bit 2er-KomplementHex (signed byte)
+1270111 11110x7F
+10000 00010x01
00000 00000x00
−11111 11110xFF
−1281000 00000x80

Overflow und seine forensische Bedeutung

Ein Overflow tritt auf, wenn das Ergebnis einer Operation den darstellbaren Bereich überschreitet:

# 8-Bit signed Overflow-Beispiel
 0111 1111  (+127)
+0000 0001  (+1)
───────────
 1000 0000  (−128)  ← Overflow! 127 + 1 ergibt nicht +128, sondern −128
Forensische Relevanz — Timestamp-Overflow: Unix-Timestamps sind 32-Bit signed Integer (Sekunden seit 01.01.1970). Der Maximalwert ist 2.147.483.647, was dem 19. Januar 2038, 03:14:07 UTC entspricht. Danach springt der Zähler auf −2.147.483.648 (= 13. Dezember 1901). Dies ist das Year-2038-Problem — analog zu Y2K, aber für 32-Bit-Systeme. Forensiker müssen dies bei der Analyse alter Systeme und Embedded-Geräte berücksichtigen.
Andere Integer-Typen in der Forensik: Windows FILETIME ist ein 64-Bit unsigned Integer (100-Nanosekunden-Intervalle seit 01.01.1601). NTFS-Zeitstempel nutzen dieses Format. Das verhindert Overflow-Probleme, erfordert aber spezielle Konvertierung für die forensische Analyse.

Gleitkommazahlen: IEEE 754

IEEE 754 — Der Standard für Gleitkommazahlen

IEEE 754 definiert, wie Dezimalzahlen mit Nachkommastellen binär gespeichert werden. Das Format besteht aus drei Komponenten:

  • Vorzeichen (Sign): 1 Bit (0 = positiv, 1 = negativ)
  • Exponent: Gespeichert mit Bias (Single: Bias 127, Double: Bias 1023)
  • Mantisse (Significand): Die Nachkommastellen, normalisiert mit implizierter führender 1
TypBits gesamtSignExponentMantisseBiasDezimalstellen
Single (float)321823127~7
Double64111521023~16

Normalisierte vs. Denormalisierte Zahlen

  • Normalisiert: Exponent ≠ 0 und ≠ alle 1en. Die Mantisse hat eine implizite führende 1 (1.xxx). Dies ist der Normalfall
  • Denormalisiert (subnormal): Exponent = 0, Mantisse ≠ 0. Die implizite führende Ziffer ist 0 (0.xxx). Erlaubt die Darstellung sehr kleiner Zahlen nahe Null
  • Null: Exponent = 0, Mantisse = 0. Es gibt +0 und −0 (unterscheidbar!)
  • Unendlich: Exponent = alle 1en, Mantisse = 0. +∞ und −∞
  • NaN (Not a Number): Exponent = alle 1en, Mantisse ≠ 0. Zeigt ungültige Operationen an (z.B. 0/0)
# IEEE 754 Single Precision — Beispiel: −12.625
# −12.625 dezimal = −1100.101 binär = −1.100101 × 2^3
# Sign:    1 (negativ)
# Exponent: 3 + 127 (Bias) = 130 = 1000 0010
# Mantisse: 100101 0000 0000 0000 0000 00 (23 Bit, führende 1 implizit)
# Hex:     0xC14A0000
Forensische Relevanz — GPS-Koordinaten: GPS-Koordinaten werden häufig als IEEE 754 Double gespeichert. Breitengrad und Längengrad in EXIF-Daten, GPX-Dateien und Navigationssystemen nutzen 64-Bit-Floats. Bei der Analyse von RAM-Dumps oder gelöschten Dateien können Sie Float-Muster suchen: 52.5200° N (Berlin) wäre als 0x404A3D70A3D70A3D gespeichert. Tools wie bulk_extractor können GPS-Koordinaten aus Rohdaten extrahieren, indem sie nach Float-Mustern in plausiblen Wertebereichen suchen.
Python-Tipp: struct.unpack('<d', bytes.fromhex('404A3D70A3D70A3D'))[0] liest einen Little-Endian-Double aus einem Hex-String. struct.pack('<d', 52.52).hex() macht das Gegenteil.

Zeichenkodierung: ASCII, UTF-8, UTF-16, UTF-32

ASCII (American Standard Code for Information Interchange)

  • 7-Bit-Kodierung, 128 Zeichen (0–127)
  • 32 Steuerzeichen (0–31, 127) + 96 druckbare Zeichen (32–126)
  • Forensisch relevant: Strings-Kommando erkennt standardmäßig ASCII-Sequenzen ≥ 4 Zeichen. Viele Logs, Config-Dateien und Protokolle (HTTP-Header) sind ASCII

UTF-8 (Unicode Transformation Format — 8 Bit)

  • Variable Länge: 1–4 Bytes pro Zeichen
  • 1 Byte: 0xxxxxxx (ASCII-kompatibel, 0–127)
  • 2 Bytes: 110xxxxx 10xxxxxx (128–2047, z.B. Umlaute)
  • 3 Bytes: 1110xxxx 10xxxxxx 10xxxxxx (2048–65535, z.B. CJK-Zeichen)
  • 4 Bytes: 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx (Emoji, seltene Zeichen)
  • Dominant in: Linux, Web (HTML5 Default), JSON, moderne APIs

UTF-16

  • 2 oder 4 Bytes pro Zeichen (meist 2)
  • Zeichen im BMP (Basic Multilingual Plane, U+0000–U+FFFF): 2 Bytes direkt
  • Zeichen außerhalb BMP: Surrogatpaare (4 Bytes)
  • UTF-16 LE (Little Endian): Windows-native Kodierung! Registry, NTFS-Dateinamen, Windows-API
  • UTF-16 BE (Big Endian): Selten, aber in einigen Netzwerkprotokollen

UTF-32

  • 4 Bytes fest pro Zeichen — jedes Unicode-Zeichen passt direkt
  • In der Praxis selten wegen Speicherverschwendung (4× ASCII)
  • Wird in einigen internen APIs und Datenbanken verwendet

BOM (Byte Order Mark)

Die BOM ist ein optionales Unicode-Zeichen (U+FEFF) am Anfang einer Datei, das die Byte-Reihenfolge anzeigt:

KodierungBOM (Hex)BOM (Bytes)
UTF-8EF BB BF3 Bytes
UTF-16 LEFF FE2 Bytes
UTF-16 BEFE FF2 Bytes
UTF-32 LEFF FE 00 004 Bytes
UTF-32 BE00 00 FE FF4 Bytes
Forensik — Windows ist UTF-16 LE: Die Windows-Registry, NTFS-Dateinamen, Windows Event Logs und viele Windows-APIs verwenden UTF-16 LE. Wenn Sie mit strings auf einem Windows-Image arbeiten, müssen Sie strings -e l image.dd verwenden, um UTF-16 LE Strings zu extrahieren. Andernfalls sehen Sie nur ASCII und übersehen einen Großteil der Windows-spezifischen Strings (Registry-Pfade, Dateinamen, Benutzernamen).

Base64 und Hex-Encoding

Base64

  • Kodiert Binärdaten in 64 druckbare ASCII-Zeichen: A–Z, a–z, 0–9, +, / (und = als Padding)
  • 3 Eingabe-Bytes → 4 Ausgabe-Zeichen (33 % Overhead)
  • Padding: Endet mit = (1 Byte übrig) oder == (2 Bytes übrig)
KontextBase64-VerwendungForensische Relevanz
E-Mail (MIME)Anhänge, Bilder in HTML-MailsBase64-dekodierte Anhänge in .eml-Dateien
PowerShellpowershell -enc <Base64>Malware führt obfuskierten Code aus — dekodieren mit base64 -d
Data URIsdata:image/png;base64,...Eingebettete Bilder in Webseiten und Phishing-Mails
JSON/APIsBinäre Felder als Base64API-Logs, OAuth-Tokens, JWT
ZertifikatePEM-Format (—–BEGIN CERTIFICATE—–)TLS-Zertifikate in PCAPs und Logs
# Base64 in Malware erkennen und dekodieren
$ echo "powershell -enc SABlAGwAbABvAA==" | grep -oP '(?<=-enc )[A-Za-z0-9+/=]+' | base64 -d
# → Hello (UTF-16 LE, daher viele 00-Bytes)

# Base64 in Logfiles finden (typische Merkmale: lange Strings mit +=/)
$ grep -oP '[A-Za-z0-9+/=]{40,}' suspicious.log | while read b64; do
    echo "$b64" | base64 -d 2>/dev/null && echo "---"
done

Hex-Encoding

  • Jedes Byte wird als zwei Hex-Ziffern dargestellt
  • 100 % Overhead (1 Byte → 2 Zeichen)
  • Verwendung: Hex-Dumps, Hash-Ausgaben (sha256sum), URL-Encoding (%20 = Space)
  • Forensik: Hex-Editing von Magic Bytes, Manipulation von File-Signaturen in Anti-Forensik

Endianness: Big-Endian vs Little-Endian

Konzept

Endianness definiert die Byte-Reihenfolge bei der Speicherung von Werten, die mehr als ein Byte belegen:

  • Big-Endian (BE): Höchstwertiges Byte zuerst („Big End first"). Netzwerk-Byte-Order (TCP/IP), Motorola 68000, SPARC
  • Little-Endian (LE): Niedrigstwertiges Byte zuerst („Little End first"). Intel x86/x86-64 — die dominierende Architektur auf Desktops und Servern
  • Bi-Endian: ARM-Architektur kann beide Modi (standardmäßig LE, aber umschaltbar). Relevant für mobile Geräte und Embedded Systems

Erkennung in Hex-Dumps

# Beispiel: 32-Bit-Wert 0x12345678 im Speicher

# Big-Endian (Netzwerk-Byte-Order):
# Adresse:  N    N+1   N+2   N+3
# Wert:    0x12  0x34  0x56  0x78

# Little-Endian (Intel x86):
# Adresse:  N    N+1   N+2   N+3
# Wert:    0x78  0x56  0x34  0x12
KontextEndiannessForensische Bedeutung
TCP/IP-HeaderBig-EndianIP-Adressen, Ports in PCAPs — müssen als BE gelesen werden
NTFS $MFTLittle-EndianZeitstempel, Dateigrößen, Cluster-Nummern sind LE
JPEG/EXIFBig-Endian (meist)EXIF-Header beginnt mit 'MM' (Motorola = BE) oder 'II' (Intel = LE)
ELF (Linux Binaries)LE oder BEByte 5 im ELF-Header: 1 = LE, 2 = BE
PE (Windows Binaries)Little-EndianWindows läuft nur auf x86/x86-64 → immer LE
ARM (Mobile/Embedded)Bi-Endian (meist LE)Android/iOS sind LE, aber Netzwerkdaten auf ARM können BE sein
Häufiger Forensik-Fehler: Rohdaten mit falscher Endianness lesen führt zu völlig falschen Werten. Ein 64-Bit-NTFS-Zeitstempel als Big-Endian gelesen ergibt eine unsinnige Zeit im Jahr 30.000. Immer die Architektur des Quellsystems berücksichtigen.
Praktischer Test: Lesen Sie einen bekannten Wert (z.B. Dateigröße) aus einem Hex-Dump. Wenn 00 00 10 00 4096 ergibt, ist es Big-Endian. Wenn 00 10 00 00 4096 ergibt, ist es Little-Endian. Python: int.from_bytes(b'\x00\x10\x00\x00', 'big') vs 'little'.

Bit-Operationen: AND, OR, XOR, NOT, Shift

Grundoperationen

OperationSymbol (C/Python)WahrheitstabelleTypische Anwendung
AND&1 & 1 = 1, sonst 0Bitmasken: Flag prüfen, Bit auslesen
OR|0 | 0 = 0, sonst 1Flag setzen, Bits kombinieren
XOR^unterschiedlich = 1, gleich = 0Verschlüsselung (Malware), Checksummen, Toggling

Forensische Relevanz von XOR

XOR ist DIE zentrale Bit-Operation in der Malware-Analyse:

  • XOR-Verschlüsselung: Einfachste Form der Obfuskation. Malware XORt Strings und Payloads mit einem Ein-Byte-Schlüssel (data[i] ^ key)
  • Erkennung: XOR-verschlüsselte Daten haben oft eine gleichmäßige Byte-Verteilung (keine dominierenden 0x00-Bytes wie bei Klartext)
  • Entschlüsselung: Da XOR selbstinvers ist (A ^ K ^ K = A), genügt dasselbe XOR erneut
# XOR-String-Dekodierung in Python (Malware-Analyse)
data = bytes.fromhex("2E0A1F00411C130600411B0415001B")
# Häufigster Byte-Wert in englischen Texten ist 0x20 (Space) oder 0x00 (Null)
# Brute-Force: Alle 256 möglichen Schlüssel probieren
for key in range(256):
    decoded = bytes(b ^ key for b in data)
    if decoded.isascii() and decoded.isprintable():
        print(f"Key 0x{key:02X}: {decoded.decode()}")

Bitmasken in Dateisystemen

Dateisysteme nutzen Bitmasken extensiv für Flags und Attribute:

  • NTFS $MFT Attribute: Bit 0 = Read-Only, Bit 1 = Hidden, Bit 2 = System, Bit 5 = Archive
  • Unix-Dateirechte: rwxr-xr-x = 0755 oktal = 111 101 101 binär. Jedes 3er-Bit-Tripel = owner/group/other
  • Windows-Registry: REG_DWORD-Werte mit Bitfeldern (z.B. 0x00000001 = Enabled)
# Bitmasken-Analyse in Python
attr = 0x22  # NTFS-Attribut: 0010 0010 (Hidden + Archive)

# Flags prüfen
is_readonly = bool(attr & 0x01)  # False
is_hidden   = bool(attr & 0x02)  # True
is_system   = bool(attr & 0x04)  # False
is_archive  = bool(attr & 0x20)  # True

# Alle gesetzten Bits auflisten
set_bits = [f"Bit {i}" for i in range(8) if attr & (1 << i)]
print(f"Gesetzte Flags: {set_bits}")  # ['Bit 1', 'Bit 5']

Praktische Anwendung / Befehle

Hex-Dump-Analyse

# Hex-Dump mit ASCII-Vorschau
xxd suspicious.bin | head -50

# Hex-Dump ohne Adressen (roh)
xxd -p image.dd | head -c 256

# Bestimmten Offset anspringen (Byte 512)
xxd -s 512 -l 64 image.dd

# Hex-Editoren: HxD (Windows), hexedit (Linux), 010 Editor (kommerziell, mit Templates)

Base64-Analyse

# Dekodieren
echo "SGVsbG8gV2VsdA==" | base64 -d

# PowerShell -enc entschlüsseln
echo "powershell -enc SABlAGwAbABvAA==" | sed 's/.*-enc //' | base64 -d

# Base64-Strings in Datei finden
grep -oP '[A-Za-z0-9+/]{20,}={0,2}' datei.txt | while read b; do echo "$b" | base64 -d 2>/dev/null; done

Endianness-Konvertierung

# Python: Endianness-Konvertierung
import struct

# Little-Endian 32-Bit-Wert aus Bytes lesen
value = struct.unpack('<I', b'\x78\x56\x34\x12')[0]  # → 0x12345678

# In Big-Endian umwandeln
big_endian = struct.pack('>I', value)  # → b'\x12\x34\x56\x78'

# 64-Bit Windows FILETIME → Unix Timestamp
filetime = struct.unpack('<Q', data[offset:offset+8])[0]
unix_ts = (filetime - 116444736000000000) / 10_000_000

Zahlensystem-Umrechnung CLI

# bash: Schnelle Umrechnungen
echo $((16#FF))        # Hex → Dez: 255
echo $((2#1010))       # Bin → Dez: 10
printf '%x\n' 255      # Dez → Hex: ff
printf '%o\n' 64       # Dez → Oktal: 100

# Python: Interaktiver Rechner
python3 -c "print(hex(255), bin(42), oct(64))"
python3 -c "print(0xFF, 0b1010, 0o377)"
