Kurzüberblick

Software-Tests und Qualitätssicherung (QS) bilden das Rückgrat jeder professionellen Softwareentwicklung. Sie stellen sicher, dass Code nicht nur funktional korrekt ist, sondern auch unter unerwarteten Bedingungen zuverlässig arbeitet. Für digitale Forensiker ist Qualitätssicherung essenziell: Ein Hash-Tool, das in 0,01% der Fälle falsche Werte liefert, kann Beweise entwerten. Ein Imaging-Werkzeug, das Sektoren überspringt, macht Ermittlungen angreifbar. Die Prinzipien des Software-Testens helfen Forensikern, ihre eigenen Werkzeuge zu validieren und die Glaubwürdigkeit ihrer Ergebnisse vor Gericht zu verteidigen.

MHB-Mapping: BDF103 (Programmierung) — Qualitätssicherung von Software, Testmethoden und forensische Anwendung.

Unit-Tests: JUnit und pytest

Unit-Tests prüfen die kleinste testbare Einheit einer Software — typischerweise eine einzelne Methode oder Funktion — isoliert von ihrer Umgebung. Ziel ist es, Fehler frühzeitig und lokalisiert zu erkennen, bevor sie sich in der Integration ausbreiten.

Arrange-Act-Assert (AAA-Muster)

Das AAA-Muster strukturiert jeden Test in drei klar getrennte Phasen:

PhaseBeschreibungBeispiel
ArrangeTestumgebung vorbereiten, Objekte initialisieren, Eingabedaten festlegenNeues User-Objekt mit Test-E-Mail erstellen
ActDie zu testende Operation ausführenuser.changeEmail("neu@test.de") aufrufen
AssertErgebnis prüfen, Soll-Ist-Vergleich durchführenassertEquals("neu@test.de", user.email)

Python: pytest mit AAA-Pattern

# test_hash_utils.py
import hashlib
from forensics.hash_utils import compute_sha256

class TestComputeSHA256:
    def test_empty_input(self):
        # Arrange
        data = b""
        expected = hashlib.sha256(b"").hexdigest()

        # Act
        result = compute_sha256(data)

        # Assert
        assert result == expected

    def test_known_string(self):
        # Arrange
        data = b"forensic evidence"
        expected = hashlib.sha256(b"forensic evidence").hexdigest()

        # Act
        result = compute_sha256(data)

        # Assert
        assert result == expected
        assert len(result) == 64

Java: JUnit 5 mit AAA-Pattern

// SectorReaderTest.java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

class SectorReaderTest {

    @Test
    void testSectorAlignment() {
        // Arrange
        SectorReader reader = new SectorReader("/dev/sda", 512);
        long offset = 1024L;

        // Act
        byte[] sector = reader.readSector(offset);

        // Assert
        assertNotNull(sector, "Sektor darf nicht null sein");
        assertEquals(512, sector.length, "Sektorgröße muss 512 Bytes betragen");
    }
}

Mocking und Stubbing

Mocks und Stubs ersetzen reale Abhängigkeiten durch kontrollierte Test-Doubles. Ein Stub gibt vorprogrammierte Werte zurück (z.B. eine Fake-Datenbank). Ein Mock registriert zusätzlich, wie er aufgerufen wurde: Welche Methoden mit welchen Parametern?

# pytest mit unittest.mock
from unittest.mock import Mock, patch

def test_disk_imager_with_mocked_device():
    # Arrange: Mock des physischen Block-Devices
    mock_device = Mock()
    mock_device.read.return_value = b"\x00" * 512

    imager = DiskImager(mock_device)

    # Act
    checksum = imager.create_image("output.dd")

    # Assert
    mock_device.read.assert_called()  # Wurde read() überhaupt aufgerufen?
    assert checksum is not None       # Prüfsumme muss berechnet sein
Forensik-Praxis: Mocking von Hardware-Abhängigkeiten. Ein Imaging-Tool, das direkt auf /dev/sda zugreift, ist im Test schwer prüfbar. Mit Mocks simulieren Sie das Verhalten einer Festplatte — inklusive Fehlerfällen wie fehlerhaften Sektoren oder Timeout beim Lesevorgang. So testen Sie das Fehlerhandling, ohne eine echte Platte zu gefährden.

Integrationstests: Top-down, Bottom-up, Sandwich

Integrationstests prüfen das Zusammenspiel mehrerer Module. Während Unit-Tests Einzelkomponenten isolieren, decken Integrationstests Fehler in den Schnittstellen auf: Falsche Parameter, abweichende Datenformate oder unerwartete Seiteneffekte.

Teststrategien

StrategieVorgehenVorteilNachteil
Top-downStartet beim Hauptmodul. Untermodule werden durch Stubs ersetzt und schrittweise durch echte Module ersetzt.Früher Überblick über das Gesamtverhalten, GUI-getriebene Tests früh möglichBraucht viele Stubs, die selbst fehleranfällig sein können
Bottom-upStartet bei den Basismodulen (z.B. Datenbank-Treiber, File-I/O). Treiber rufen sie auf und prüfen Ergebnisse.Keine Stubs nötig, da unterste Schicht echt ist; gute FehlerlokalisierungGesamtsystem erst spät sichtbar; System-Level-Bugs zeigen sich verzögert
SandwichKombiniert Top-down und Bottom-up: Gleichzeitig von oben (mit Stubs) und unten (mit Treibern) integrieren, bis sie sich treffen.Parallelisierbar, nutzt Vorteile beider Ansätze, gute TestabdeckungKomplexere Koordination nötig; mehr Overhead im Testmanagement

Test-Doubles im Detail

TypVerhaltenForensik-Beispiel
DummyWird nur als Platzhalter übergeben, aber nie verwendet.Ein Log-Level-Parameter, der in diesem Test nicht relevant ist
StubLiefert fest codierte Antworten auf Aufrufe.Stub einer Hash-Library, die immer "abc123" zurückgibt
MockRegistriert Aufrufe und erlaubt Assertions auf Interaktion.Mock eines Write-Blockers: Wurde preventWrite() aufgerufen?
FakeLeichtgewichtige Arbeitsimplementierung (z.B. In-Memory-DB statt PostgreSQL).In-Memory-Dateisystem statt eines physischen E01-Images
SpyEchtes Objekt, das Aufrufe mitprotokolliert.Echter SQLite-Treiber, der alle Queries in eine Liste loggt
Forensik-Integrationstest: Ein Imaging-Tool muss nicht nur korrekte Sektoren lesen (Unit-Test), sondern auch Prüfsummen über mehrere Sektoren akkumulieren (Integrationstest mit Mock-Device), das E01-Format schreiben (Integrationstest: Imager + E01-Encoder) und schließlich Kompression und Metadaten korrekt speichern (End-to-End-Test).

Test-Driven Development (TDD)

TDD kehrt die klassische Reihenfolge um: Erst der Test, dann der Code. Der Zyklus: Red → Green → Refactor.

  1. Red: Schreibe einen fehlschlagenden Test (rote Ampel). Der Test definiert das gewünschte Verhalten, scheitert aber, weil die Implementierung fehlt.
  2. Green: Schreibe minimalen Code, der den Test bestehen lässt (grüne Ampel). Nicht mehr Code als nötig — keine Vorab-Optimierung, keine "könnte man später brauchen"-Features.
  3. Refactor: Verbessere den Code (ohne Verhalten zu ändern): Duplikate entfernen, Lesbarkeit erhöhen, Performance optimieren. Die Tests schützen vor Regression.
# TDD-Beispiel: Funktion zur Validierung einer MD5-Prüfsumme

# --- Red-Phase: Test schreiben ---
def test_valid_md5_checksum():
    """Ein gültiger 32-stelliger Hex-String wird akzeptiert."""
    assert validate_md5("d41d8cd98f00b204e9800998ecf8427e") == True

def test_invalid_md5_checksum():
    """Ein String mit falscher Länge wird abgelehnt."""
    assert validate_md5("abc123") == False   # Zu kurz
    assert validate_md5("z" * 32) == False   # Ungültige Hex-Zeichen

# --- Green-Phase: Minimaler Code ---
import re
def validate_md5(s: str) -> bool:
    return bool(re.match(r'^[a-f0-9]{32}$', s))

# --- Refactor-Phase: Code verbessern ---
import re
from typing import Final

MD5_PATTERN: Final = re.compile(r'^[a-f0-9]{32}$', re.IGNORECASE)

def validate_md5(s: str) -> bool:
    """Prüft, ob ein String eine gültige MD5-Hex-Prüfsumme ist."""
    if not isinstance(s, str):
        return False
    return bool(MD5_PATTERN.match(s))
TDD für Forensik-Tools: Schreiben Sie zuerst den Test für ein neues Carving-Signatur-Modul (Red), bevor Sie die Signatur-Datenbank erweitern. Der Test definiert, welche Header/Footer-Kombination erkannt werden muss. Der Green-Code implementiert exakt diese Signatur — und der Refactor entfernt Duplikate zu bereits existierenden Signaturen. TDD zwingt Sie, das Problem vor der Lösung zu verstehen.

Testabdeckung (Coverage)

Coverage misst, welcher Anteil des Quellcodes durch Tests ausgeführt wurde. Sie ist ein notwendiges, aber nicht hinreichendes Qualitätsmerkmal: 100% Coverage bedeutet nicht, dass der Code fehlerfrei ist — nur dass jede Zeile mindestens einmal durchlaufen wurde.

Coverage-Typen

MetrikWas gemessen wirdBeispiel
Line CoverageWelche Code-Zeilen wurden ausgeführt?Zeile x = a / b wurde ausgeführt, aber keiner hat b=0 getestet
Branch CoverageWurden alle Zweige von Bedingungen abgedeckt? (if/else, switch/case)Ein if hat zwei Branches. Beide müssen getestet werden
Statement CoverageAnzahl der ausgeführten Statements im Verhältnis zur GesamtzahlGröbere Metrik als Line; eine Zeile kann mehrere Statements haben
Function CoverageWelche Funktionen wurden überhaupt aufgerufen?Eine Hilfsfunktion wurde nie getestet und hat möglicherweise Bugs
Path CoverageAlle möglichen Pfade durch den Code (exponentiell viele)In der Praxis selten 100% erreichbar — teuer und umständlich

Tools

ToolSpracheBesonderheit

Forensik-Warnung: 100% Line Coverage ist im Forensik-Kontext nicht genug. Sie müssen auch Edge Cases testen: Was passiert bei einem 4-GB-File mit 0-Byte-Inhalt? Was bei einer corrupten E01-Metadaten-Sektion? Was bei Unicode-Pfaden auf Windows? Coverage misst Quantität, nicht Qualität. Ein einzelner ungetesteter Branch in einem Hash-Algorithmus kann katastrophal sein.

Statische vs. dynamische Analyse

Statische Analyse (Compile-Time)

Analysiert den Quellcode oder Bytecode ohne Ausführung des Programms. Findet potenzielle Fehler, Sicherheitslücken, Coding-Style-Verstöße und toten Code.

ToolSpracheErkennt
SonarQubeMulti (25+ Sprachen)Code Smells, Bugs, Sicherheitslücken (OWASP-Top-10-Regeln), Technical Debt
ESLintJavaScript/TypeScriptStyle-Verstöße, potenzielle Runtime-Fehler, Best-Practice-Deviationen
pylint / ruffPythonPEP-8-Abweichungen, unbenutzte Variablen, zu komplexe Funktionen (McCabe)
SpotBugsJava-BytecodeNull-Pointer-Dereferenzen, Resource Leaks, Concurrency-Bugs
SemgrepMultiBenutzerdefinierte Regeln ("Finde alle Stellen, wo subprocess mit Shell=True aufgerufen wird")

Dynamische Analyse (Runtime)

Analysiert das Programm während der Ausführung — beobachtet Speicher, Aufrufe, Abhängigkeiten. Dazu gehören auch Fuzzing und Sanitizer.

Tool/KonzeptEinsatzForensik-Nutzen
Valgrind (memcheck)C/C++: Speicherlecks, Use-after-FreeImaging-Tool auf Memory-Corruption prüfen
AddressSanitizer (ASan)C/C++: Buffer overflows, Stack overflowsForensik-Parser auf Pufferüberläufe testen
AFL / libFuzzerC/C++: Fuzzing mit Coverage-Guided MutationJPEG-Carver fuzzen: Korrumpierte Header erkennen?
Python Profiler (cProfile)Python: Hotspots identifizierenWarum dauert die Timeline-Analyse 3 Stunden?
Zusammenspiel in der Forensik: Statische Analyse prüft den Code auf Schwachstellen, bevor er im Feld eingesetzt wird. Dynamische Analyse validiert das Laufzeitverhalten und deckt Fehler auf, die statisch nicht sichtbar sind (Rennbedingungen, Speicherkorruption). Beide sind komplementär und müssen in der QS forensischer Tools kombiniert werden.

Code-Reviews und Pair Programming

Code-Reviews sind systematische Prüfungen von Quellcode durch andere Entwickler, bevor Änderungen in den Hauptzweig gemerged werden. Sie sind eine der effektivsten Methoden zur Fehlerfindung.

Review-Tiefe

StufeFokusDauer
WalkthroughAutor führt den Code vor, Reviewer stellen Fragen. Informell.30–60 min
Technical ReviewTechnische Korrektheit, Architektur-Entscheidungen, Performance.1–2 h
Formal InspectionStrukturierte Sitzung mit Checkliste, Rollen (Moderator, Autor, Reviewer), Metriken.2–4 h
Pull-Request-ReviewAsynchrones Review über GitHub/GitLab. Inline-Kommentare, CI-Checks.Variabel

Forensische Perspektive: Warum Code-Reviews für Forensik-Tools unverzichtbar sind

  • Beweisintegrität: Ein Bug in der Hash-Berechnung kann im Gerichtssaal fatal sein. Reviews fangen Logikfehler, die Tests übersehen.
  • Toolchain-Transparenz: Die Verteidigung kann nach der Code-Historie fragen. Dokumentierte Reviews zeigen Sorgfalt.
  • Wissensverteilung: Wenn nur eine Person den MFT-Parser versteht, ist das ein Single Point of Failure. Reviews verteilen Wissen im Team.
  • Compliance: ISO 17025 (Forensik-Labore) verlangt dokumentierte Validierungsprozesse — Code-Reviews sind Teil davon.

Pair Programming

Zwei Entwickler an einem Rechner: Driver (schreibt Code) und Navigator (denkt strategisch, prüft jede Zeile). Studien zeigen 15% weniger Fehler bei nur 15% mehr Zeitaufwand — eine positive Kosten-Nutzen-Bilanz.

Forensischer Bezug

Forensisches ToolKritischer TestWarum essenziell
Hash-Generator (sha256deep)Leere Datei, 4-GB-Datei, Unicode-Pfad, Symlink, Device-FileFalscher Hash = Beweis wertlos; Daubert/Frye-Standard fordert validierte Tools
Imaging-Tool (dd, dc3dd)Alle Sektoren kopiert? Prüfsumme nach Imaging? Fehler-Sektoren korrekt geloggt?Ein nicht kopierter Slack-Space kann den einzigen Hinweis enthalten
Carving-Tool (foremost, scalpel)Falsch-Positive? Falsch-Negative? Fragmentierte Dateien? Korrumpierte Header?Ein falsch gecarvtes JPEG kann zu einer falschen Verdächtigung führen
Log-Parser (Plaso, log2timeline)Zeitzonen-Konvertierung? Zeilenumbruch in Einträgen? Unterschiedliche Log-Formate?Zeitstempel sind das Rückgrat jeder Timeline-Analyse
Memory-Analyse (Volatility 3)Prozessliste vollständig? Versteckte Prozesse (DKOM)? DLL-Injection erkannt?Rootkits manipulieren aktiv die Datenstrukturen im RAM

Tests sind im forensischen Kontext nicht optional — sie sind die Grundlage der gerichtlichen Verwertbarkeit. Nach dem Daubert-Standard (US) und den ACPO-Richtlinien (UK) müssen forensische Methoden und Werkzeuge validiert sein. Ein nicht getestetes Tool ist ein nicht validiertes Tool — und damit vor Gericht potenziell unbrauchbar.
