SIEM und Security Monitoring Mittel

Inhaltsverzeichnis
  1. Kurzüberblick
  2. Details
  3. Praktische Anwendung
  4. Quellen / Weiterführendes

Kurzüberblick

SIEM-Systeme (Security Information and Event Management) sammeln, korrelieren und analysieren Log-Daten aus dem gesamten Netzwerk in Echtzeit. Aus der Forensik-Perspektive sind sie die zentrale Anlaufstelle: Hier findest du die Timeline eines Angriffs über alle Systeme hinweg.

Details

SIEM-Architektur

[Server]    ──┤
[Firewalls] ──┼──> [Log Collector] ──> [Parser/Normalisierung] ──> [Korrelations-Engine] ──> [Dashboard / Alerting]
[IDS/IPS]   ──┤
[DNS]       ──┘
  • Collection: Agenten (Winlogbeat, Filebeat, Syslog) sammeln Logs
  • Parsing / Normalisierung: Unterschiedliche Log-Formate → einheitliches Schema (z.B. ECS — Elastic Common Schema)
  • Korrelation: Regelbasiert ("10 fehlgeschlagene Logins + 1 erfolgreiches" = mögliche Kompromittierung)
  • Alerting: E-Mail, Slack, PagerDuty — Eskalation auslösen
  • Dashboard: Live-Ansichten (Logins pro Stunde, Top-Firewall-Drops)

SIEM in der forensischen Untersuchung

  • Zeitraum eingrenzen: Wann begann der Vorfall?
  • IP des Angreifers pivotieren: Auf welchen anderen Systemen taucht die Angreifer-IP auf?
  • User-Account pivotieren: Welche Aktionen führte der kompromittierte Account aus?
  • Timeline erstellen: Alle Events in chronologischer Reihenfolge

Splunk SPL (Search Processing Language)

| stats count by src_ip, user
| where count > 10
| sort - count

Elasticsearch / Kibana (KQL)

| stats count() by user.name

Praktische Anwendung

  • ELK Stack aufsetzen: docker-compose up → Elasticsearch + Kibana + Logstash
  • Windows-Logs einsammeln: Winlogbeat konfigurieren → SIEM
  • Sigma-Regeln: Plattformunabhängige Detection-Rules (Sigma → Splunk/Kibana/... konvertieren)

AAA — Authentifizierung, Autorisierung und Accounting

AAA-Framework

  • Authentication: Wer bist du? (Username/Password, Zertifikat, Token)
  • Authorization: Was darfst du? (Zugriffsrechte, VLAN-Zuweisung, ACLs)
  • Accounting: Was hast du getan? (Login-Zeit, übertragene Bytes, ausgeführte Kommandos)
  • Forensischer Wert: Accounting-Logs sind die präzisesten Benutzer-Aktivitäts-Logs im Netzwerk

RADIUS (Remote Authentication Dial-In User Service)

  • Protokoll: UDP/1812 (Authentication), UDP/1813 (Accounting)
  • Paket-Struktur: Code, Identifier, Length, Authenticator, Attributes (AV-Pairs)
  • Sicherheit: MD5-Hash für Passwort, RADIUS-Secret shared key
  • RADIUS Accounting Start/Interim-Update/Stop — zeigt exakte Login/Logout-Zeiten!
  • Forensisch: RADIUS-Accounting-Logs beantworten: Welcher User war wann auf welchem System?

TACACS+ (Terminal Access Controller Access Control System Plus)

  • Protokoll: TCP/49, entwickelt von Cisco
  • Trennt Authentication, Authorization, Accounting in separate Flows (RADIUS kombiniert Auth+Auth)
  • Komplette Paket-Verschlüsselung (nicht nur Passwort wie bei RADIUS)
  • Detailliertes Command-Authorization: Welche IOS-Kommandos darf ein Admin ausführen?
  • Command-Accounting: JEDES ausgeführte Command wird geloggt! → Forensiker-Traum bei Switch/Router-Investigations

802.1X — Port-Based Network Access Control

  • IEEE 802.1X: Supplicant (Client) ↔ Authenticator (Switch/WLAN-AP) ↔ Authentication Server (RADIUS)
  • EAP (Extensible Authentication Protocol): EAP-TLS (Zertifikate), PEAP-MSCHAPv2 (Username/Password)
  • Forensischer Nutzen: 802.1X-Logs zeigen exakt, welches Gerät (MAC) mit welcher Identität wann am Switch-Port angemeldet war
  • NAC (Network Access Control): Dynamische VLAN-Zuweisung basierend auf Geräte-Health

Netzwerk-Monitoring mit NetFlow und sFlow

NetFlow / IPFIX

  • Flow-Definition: 5-Tuple (SrcIP, DstIP, SrcPort, DstPort, Protocol) + Timestamp
  • NetFlow v5: feste Felder, 7 Schlüsselfelder
  • NetFlow v9: Template-basiert, erweiterbar (Flexible NetFlow)
  • IPFIX (IP Flow Information Export): IETF-Standard (RFC 7011), basiert auf NetFlow v9
  • Typische Felder: Bytes, Packets, TCP-Flags, ToS, Input/Output-Interface, Next-Hop, BGP-AS
  • Forensische Nutzung: Wer hat mit wem kommuniziert? Wie viel Daten? Wann? — auch ohne Payload extrem wertvoll!
  • Speicherdauer: Typisch 30-90 Tage → historische Netzwerkkommunikation rekonstruierbar

sFlow

  • Sampling-basiert (nicht jeder Flow, sondern 1 von N Paketen)
  • sFlow Agent (in Switch/Router-Hardware) → sFlow Collector
  • Weniger granular als NetFlow, aber skalierbarer bei High-Throughput (100 Gbps+)
  • Forensisch: Besser als NICHTS, aber NetFlow bevorzugen wenn verfügbar (Flow-basiert = vollständiger)
