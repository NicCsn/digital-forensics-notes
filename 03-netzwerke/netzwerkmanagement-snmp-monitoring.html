Kurzüberblick

Netzwerkmanagement umfasst die Uberwachung, Steuerung und Konfiguration von Netzwerkkomponenten (Router, Switches, Firewalls, Server) uber standardisierte Protokolle. SNMP (Simple Network Management Protocol) ist das dominierende Monitoring-Protokoll seit 1988. NetFlow/IPFIX und sFlow liefern granulare Verkehrsdaten auf Flow-Ebene. Fur die digitale Forensik sind diese Protokolle und die durch sie gesammelten Daten von unschatzbarem Wert: Sie rekonstruieren Netzwerkverhalten, identifizieren Angriffsmuster und liefern gerichtsfeste Beweise in Form von Metadaten.

MHB-Mapping: BDF208 (Netzwerke) — Protokolle und Methoden des Netzwerkmanagements und ihre forensische Anwendung.

SNMPv1/v2c/v3: Manager, Agent, MIB, OID

SNMP arbeitet nach dem Manager-Agent-Modell. Der Manager (NMS — Network Management Station) fragt Daten von Agents ab, die auf den verwalteten Geraten laufen. Ein Agent kann auch unaufgefordert Traps an den Manager senden.

SNMP-Architektur

KomponenteRolleBeispiel
NMS (Manager)Fragt Agents ab, empfangt Traps, visualisiert DatenLibreNMS, PRTG, Zabbix mit SNMP-Modul
AgentLauft auf verwaltetem Gerat, beantwortet Queries, sendet Trapssnmpd (Linux), Windows SNMP Service
MIB (Management Information Base)Datenbank-Schema: Definiert, welche Daten abfragbar sindIF-MIB (Interfaces), TCP-MIB, HOST-RESOURCES-MIB
OID (Object Identifier)Adresse eines Datenpunkts in der MIB-Hierarchie1.3.6.1.2.1.1.3.0 = sysUpTime

SNMP-Versionen im Vergleich

MerkmalSNMPv1 (RFC 1157)SNMPv2c (RFC 1901)SNMPv3 (RFC 3410)
AuthentifizierungCommunity-String (Klartext)Community-String (Klartext)USM: MD5/SHA + Benutzername + optional Verschlusselung
VerschlusselungKeineKeineAES/DES (Privatsphare)

Wichtige OIDs fur die Forensik

# System-Informationen (System MIB)
1.3.6.1.2.1.1.1.0    # sysDescr  — Geratetyp, OS-Version
1.3.6.1.2.1.1.3.0    # sysUpTime — Wie lange lauft das Gerat schon?
1.3.6.1.2.1.1.5.0    # sysName   — Hostname

# Interface-Statistiken (IF-MIB)
1.3.6.1.2.1.2.2.1.10  # ifInOctets  — Empfangene Bytes pro Interface
1.3.6.1.2.1.2.2.1.16  # ifOutOctets — Gesendete Bytes pro Interface
1.3.6.1.2.1.2.2.1.8   # ifOperStatus — Up/Down/Testing

# TCP-Verbindungen (TCP-MIB)
1.3.6.1.2.1.6.13.1.1  # tcpConnState — Aktive TCP-Verbindungen

# Abfrage-Beispiel mit snmpget und snmpwalk
snmpget -v2c -c public 192.168.1.1 1.3.6.1.2.1.1.1.0
snmpwalk -v2c -c public 192.168.1.1 1.3.6.1.2.1.2.2.1.2
Sicherheitswarnung: SNMPv1/v2c-Community-Strings werden im Klartext ubertragen. Der Default-Community-String public (read) und private (write) ist auf Millionen Geraten weltweit aktiv. Ein Angreifer mit Zugriff auf UDP/161 kann komplette Netzwerk-Topologien auslesen — dies ist ein klassischer Reconnaissance-Vektor und in forensischen Untersuchungen einer der ersten Prufpunkte.

SNMPv3 Security: USM und VACM

SNMPv3 fuhrt ein mehrschichtiges Sicherheitsmodell ein, das die gravierenden Schwachen von v1/v2c adressiert. Forensisch relevant, weil Angreifer oft ungesicherte SNMPv1/v2c-Installationen ausnutzen — die Existenz von SNMPv3 auf einem Gerat zeigt Security-Awareness, das Fehlen dagegen ein potenzielles Einfallstor.

USM (User-based Security Model)

ParameterFunktion
UserNameIdentifiziert den Benutzer (z.B. "forensic_monitor")
Authentication ProtocolMD5 oder SHA — stellt Integritat und Authentizitat sicher (HMAC)
Authentication KeyMindestens 8 Zeichen; wird lokal gehasht und im Request als Digest gesendet
Privacy ProtocolDES (veraltet) oder AES — verschlusselt den Payload
Privacy KeyAhnlich wie Auth-Key; zur Verschlusselung des SNMP-Payloads
Engine IDEindeutige ID des SNMP-Agents; Teil der Key-Lokalisierung

VACM (View-based Access Control Model)

Definiert, welcher Benutzer welche OIDs lesen oder schreiben darf — granular auf MIB-View-Ebene. Eine MIB-View ist ein Subtree des OID-Baums (z.B. "alle OIDs unter 1.3.6.1.2.1.2" = nur Interface-Daten). VACM erlaubt so, einem Auditor nur Lese-Zugriff auf Traffic-Statistiken zu geben.

NetFlow/IPFIX: Flow-basiertes Monitoring

NetFlow (Cisco, 1996) und sein Nachfolger IPFIX (IETF, RFC 7011) zeichnen nicht jedes einzelne Paket auf, sondern aggregierte Flows — definiert durch das 5-Tupel: SrcIP, DstIP, SrcPort, DstPort, Protocol. Ein Flow ist eine unidirektionale Sequenz von Paketen mit gleichem 5-Tupel innerhalb eines Zeitfensters.

NetFlow-Record (v5) — Typische Felder

Src IP:     10.0.0.5
Dst IP:     203.0.113.42
Src Port:   49152
Dst Port:   443 (HTTPS)
Protocol:   6 (TCP)
Packets:    1,847
Bytes:      2,301,442
Start Time: 2026-07-04 14:22:09.123
End Time:   2026-07-04 14:22:11.890
Input If:   Gi0/1
Output If:  Gi0/3
TCP Flags:  0x1b (SYN+ACK+FIN+PSH)
Next Hop:   192.168.1.1

IPFIX (NetFlow v9/v10)

IPFIX verwendet Templates: Der Exporter definiert, welche Felder ein Record enthalt. Das ermoglicht flexible Erweiterungen (z.B. HTTP-Host-Header, DNS-Queries, VLAN-Tags) ohne Protokollanderung. Forensisch enorm wertvoll, weil auch Layer-7-Daten im Flow-Kontext sichtbar werden.

sFlow: Sampling-basierte Alternative

sFlow (sampled Flow) sampelt Pakete (z.B. jedes 1000. Paket) statt alle Flows zu erfassen. Das reduziert die Last auf dem Gerat drastisch und ist statistisch fur Traffic-Analyse und Anomalieerkennung ausreichend. Allerdings eignen sich gesampelte Daten weniger fur forensische Beweisfuhrung, da einzelne Verbindungen nicht luckenlos dokumentiert sind.

AspektNetFlow/IPFIXsFlow
ErfassungsmethodeFlow-basiert: Jeder Flow wird erfasstSampling: Nur ein Teil der Pakete
Genauigkeit100% (vollstandig)Statistisch (je nach Sample-Rate)
RessourcenverbrauchMittel bis hoch (je nach Traffic-Volumen)Gering (nur Sample-Daten)
Forensische EignungHoch: Jede Verbindung ist dokumentiertEingeschrankt: Einzelne Verbindungen konnen fehlen
VerbreitungCisco-dominierte UmgebungenData-Center (HP, Arista, Juniper), Switches
Forensik-Tipp: NetFlow-Daten sind eine unvergleichliche Timeline-Quelle. Sie zeigen nicht nur, wer wann mit wem kommuniziert hat, sondern auch Traffic-Volumen, Session-Dauer und TCP-Flag-Anomalien. Kombinieren Sie NetFlow mit DNS-Logs und Firewall-Logs, um eine vollstandige Kommunikations-Timeline zu rekonstruieren.

Syslog im Netzwerk-Kontext (RFC 5424)

Syslog ist das universelle Logging-Protokoll fur Netzwerkgerate. Praktisch jeder Router, Switch, jede Firewall und jeder Server kann Logs per Syslog (UDP/514 oder TCP/601) an einen zentralen Collector senden.

Syslog-Meldungsformat (RFC 5424)

PRI VERSION TIMESTAMP HOSTNAME APP-NAME PROCID MSGID [SD] MESSAGE

# Beispiel einer authentifizierten SSH-Anmeldung:
134 1 2026-07-04T14:22:09.123Z fw01.example.com SSH - -
[login user="root" src_ip="10.0.0.5"]
Accepted password for root from 10.0.0.5 port 49152 ssh2

Priority (PRI) = Facility x 8 + Severity

FacilityWertSeverityWert
kern (Kernel)0Emergency0
auth (Authentifizierung)4Alert1
daemon3Critical2
syslog5Error3
local0–local716–23Warning4
——Notice5
——Info6
——Debug7

Fur die Forensik ist besonders die auth-Facility (4) relevant — sie enthalt Login-Versuche, sudo-Nutzung, SSH-Sessions. Ein klassisches Angriffs-Indiz ist eine Folge von "authentication failure"-Meldungen (Severity 5, Notice) gefolgt von einer "Accepted" Meldung — ein Hinweis auf erfolgreiches Brute-Forcing.

Syslog-Trap und SNMP-Trap: Ein SNMP-Trap ist eine unaufgeforderte Nachricht vom Agent an den Manager — "Interface down", "CPU uber 90%", "Authentifizierungsfehler". Syslog-Traps werden oft via SNMP-Trap-Infrastruktur transportiert. Fur SIEM-Systeme sind SNMP-Traps wertvolle Security-Events: Ein linkDown-Trap an Interface eth0 kann auf physische Manipulation (Kabelziehen) oder einen DoS-Angriff hindeuten.

NMS-Tools: Zabbix, Nagios, PRTG, LibreNMS

Network Management Systems (NMS) sind die grafische und datenbankgestutzte Schicht uber SNMP, NetFlow und Syslog. Sie speichern historische Daten, generieren Alarme und visualisieren Netzwerkzustande.

ToolTypStarkenForensische Relevanz

Nagios / IcingaOpen SourceModular (Plugins), bewahrt in GrossunternehmenState-History: Wann ging Interface down? Wann wurde eine ACL geandert?
PRTGKommerziell (Paessler)Sensor-basiert, NetFlow/Syslog integriert, Windows-nativFlow-Daten + Syslog in einem Dashboard: Korrelation von Traffic und Logins

NetConf/YANG: Programmgesteuerte Netzwerk-Konfiguration

NetConf (RFC 6241) und YANG (RFC 6020/7950) sind die modernen Standards fur programmierbares Netzwerkmanagement. Wahrend SNMP Monitoring-dominierte Umgebungen bedient, erlaubt NetConf die automatisierte Konfiguration von Netzwerkgeraten — ein Paradigmenwechsel vom "Polling" zum "Intent-based Networking".

SchichtNetConf (RFC 6241)RESTCONF (RFC 8040)
TransportSSH (Port 830) — sicher per DefaultHTTPS — RESTful API
Operationenget-config, edit-config, lock, unlock, commit, rollbackGET, POST, PUT, PATCH, DELETE (HTTP-Methoden)
DatenmodellYANG (hierarchisch, typed)YANG + JSON/XML-Encoding
TransaktionenCandidate-Datastore + explizites Commit/RollbackKeine nativen Transaktionen; jede Anderung ist sofort aktiv

Forensisch relevant: NetConf-Logs (Audit-Trail) dokumentieren jede Konfigurationsanderung mit Benutzer, Zeitstempel und altem/neuem Wert — anders als SNMP-SETs, die haufig ohne Logging erfolgen. Eine NetConf-Infrastruktur ist daher ein Segen fur Post-Incident-Analysen: Sie zeigt punktgenau, wann eine ACL geandert oder ein Interface deaktiviert wurde.

Forensischer Nutzen

NetFlow-Daten als Timeline-Quelle

NetFlow-Daten erlauben die Rekonstruktion der Netzwerkkommunikation ohne vollstandige Packet Captures (PCAPs). Ein Flow-Export von einem Core-Router zeigt, welche internen Hosts mit externen IPs kommuniziert haben — inklusive Byte-Volumen, Port und Dauer. Das reicht haufig, um Command-and-Control-Traffic (periodische Verbindungen zu einer unbekannten IP auf Port 443) oder Data-Exfiltration (unubliche Byte-Mengen zu einer seltenen externen IP) zu identifizieren.

SNMP-Traps als Security-Events

SNMP-TrapOID (Basis)Security-Interpretation
linkDown1.3.6.1.6.3.1.1.5.3Physische Manipulation, Kabelziehen, Switch-Port deaktiviert
authenticationFailure1.3.6.1.6.3.1.1.5.5SNMP-Brute-Force-Angriff, falscher Community-String
coldStart1.3.6.1.6.3.1.1.5.1Gerat wurde neu gestartet — nach einem Angriff? Stromausfall?

Kombinierte Analyse: SNMP + NetFlow + Syslog

Die wahre forensische Starke entfaltet sich in der Korrelation: Ein Syslog-Eintrag "SSH login accepted for root from 203.0.113.42" + ein NetFlow-Record mit 1.2 GB Datenvolumen von 203.0.113.42 → Datenexfiltration. Ein SNMP-Trap "linkDown" an Port 17 + Syslog "Interface Gi0/17 changed state to down" + kein entsprechender Change-Ticket → physischer Angriff.

Forensik-Tipp: Viele Organisationen loschen NMS-Daten nach 30 oder 90 Tagen (Retention Policy). Bei einem Incident sofort einen Snapshot der NMS-Datenbank und der RRD-Dateien (Round-Robin Database fur Zeitreihen) ziehen. Diese Daten sind nach Ablauf der Retention Policy unwiederbringlich verloren und konnen die einzige Quelle fur historische Netzwerkzustande sein.
